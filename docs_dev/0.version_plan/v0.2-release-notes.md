> **状态**: 📋 规划中

## 1. 版本概述

v0.2 版本专注于**流程自动化**和**质量控制**，解决 v0.1 在实际使用中暴露的频繁阻塞问题。

### 核心目标

1. **流程自动化**：让用户能自动完成几百章的网文创作，无需频繁干预
2. **质量自动控制**：系统自动检查和修复生成质量，保持 AI 味可控
3. **异常自动处理**：API 失败、网络问题等自动恢复，用户无需手动处理
4. **中断恢复**：任何中断都能从断点继续，不丢进度

### 为什么做这个版本

v0.1 版本虽然完成了架构修复和易用性功能，但在实际使用中（特别是长篇小说创作）存在频繁阻塞问题：

| 问题 | 影响 | 频率 |
|------|------|------|
| API 失败需手动重试 | 用户必须盯着，频繁操作 | 每 10-20 章一次 |
| 质量不合格需人工修改 | 生成内容需反复调整 | 每 5-10 章一次 |
| 长时间任务阻塞界面 | 用户无法做其他事 | 始终 |
| 中断后进度丢失 | 必须从头开始 | 偶发 |
| 配置错误系统无法启动 | 用户查文档解决 | 偶发 |

**长篇小说场景**：网文通常 300-500 章，如果每 10 章需要一次人工干预，300 章需要 30 次干预，严重影响用户体验。

## 2. 核心功能详解

### 2.1 异常自动处理

**目标**：系统自动处理 80%+ 的异常，用户无需手动干预。

#### 2.1.1 API 失败自动重试

**场景**：调用 LLM API 时失败（网络超时、限流、服务不可用等）

**解决方案**：
```
API 调用
  ↓
失败？
  ↓ 是
  ↓
等待 N 秒（指数退避：1s → 2s → 4s）
  ↓
重试（最多 3 次）
  ↓
仍然失败？
  ↓ 是
  ↓
切换到备用模型
  ↓
备用模型也失败？
  ↓ 是
  ↓
标记为"需人工干预"，通知用户
```

**技术实现**：
- 指数退避算法：1秒、2秒、4秒、8秒...
- 最多重试 3 次
- 自动切换备用模型（如果配置了）
- 失败日志记录，便于排查

**验收标准**：
- [ ] 80% 的 API 失败自动恢复
- [ ] 用户无需手动点击重试
- [ ] 重试过程对用户透明（只看到最终结果）

#### 2.1.2 模型备用切换

**场景**：主模型不可用或响应质量差

**解决方案**：
```
配置主模型 + 备用模型
  ↓
主模型失败/质量差
  ↓
自动切换到备用模型
  ↓
继续生成，用户无感知
```

**技术实现**：
- 在设置中配置主模型和备用模型
- 质量评估：如果生成内容质量分 < 阈值，自动切换
- 切换过程透明，不中断生成

**验收标准**：
- [ ] 主模型失败时自动切换备用
- [ ] 切换过程不中断生成
- [ ] 用户可配置备用模型列表

#### 2.1.3 错误分类器

**场景**：不同类型的错误需要不同的处理方式

**错误分类**：
| 类别 | 示例 | 处理方式 |
|------|------|----------|
| 可恢复 | API 限流、网络超时 | 自动重试 |
| 需配置 | API Key 错误、模型不存在 | 通知用户修复配置 |
| 需人工 | 内容审核拒绝、安全问题 | 通知用户处理 |
| 系统错误 | 数据库错误、内存不足 | 记录日志，通知开发者 |

**技术实现**：
- 错误码映射表
- 自动分类逻辑
- 不同类别不同的处理流程
- 用户友好的错误消息

**验收标准**：
- [ ] 错误自动分类准确率 >90%
- [ ] 可恢复错误自动处理
- [ ] 需配置错误有清晰的修复指引

### 2.2 长篇生成优化

**目标**：支持 300+ 章的网文自动创作，不丢进度。

#### 2.2.1 检查点机制

**场景**：生成几百章需要很长时间，中间可能中断

**解决方案**：
```
每生成 N 章（默认 10 章）
  ↓
自动保存检查点
  ├─ 当前章节进度
  ├─ 所有生成的内容
  ├─ 人物状态快照
  ├─ 世界观状态
  └─ 配置和上下文
  ↓
保存到数据库/文件
  ↓
中断后可从检查点恢复
```

**技术实现**：
- 每 10 章自动保存（可配置）
- 检查点包含完整状态
- 支持多个检查点（按时间排序）
- 检查点管理界面（查看、删除、恢复）

**验收标准**：
- [ ] 每 10 章自动保存检查点
- [ ] 检查点包含完整状态
- [ ] 可查看和管理检查点

#### 2.2.2 断点续传

**场景**：网页刷新、浏览器关闭、网络断开等导致中断

**解决方案**：
```
检测到中断
  ↓
从最近的检查点恢复
  ↓
跳过已完成的章节
  ↓
从断点继续生成
  ↓
用户无感知（除了看到进度跳到断点）
```

**技术实现**：
- 页面加载时检查是否有未完成的任务
- 自动从最近检查点恢复
- 进度条显示"从第 X 章继续"
- 恢复过程自动化，无需用户操作

**验收标准**：
- [ ] 中断后自动恢复
- [ ] 不重复已完成的工作
- [ ] 恢复过程对用户透明

#### 2.2.3 批量任务队列

**场景**：300 章的网文需要排队生成

**解决方案**：
```
用户启动 300 章任务
  ↓
系统自动分解为 30 个批次（每批 10 章）
  ↓
按顺序执行每个批次
  ├─ 每个批次自动检查质量
  ├─ 不合格自动重新生成
  ├─ 每批次完成后保存检查点
  └─ 失败自动重试/切换模型
  ↓
全部完成后通知用户
```

**技术实现**：
- 任务队列管理
- 批次分解和调度
- 并发控制（避免 API 限流）
- 进度追踪和统计

**验收标准**：
- [ ] 支持 300+ 章排队执行
- [ ] 自动分解为批次
- [ ] 批次间自动衔接

#### 2.2.4 进度可视化

**场景**：长时间生成，用户不知道在做什么

**解决方案**：
```
实时显示：
├─ 当前章节：第 150 章 / 共 300 章
├─ 当前步骤：生成中 / 审核中 / 优化中
├─ 进度条：50% 完成
├─ 预计剩余时间：2 小时 30 分钟
├─ 当前质量分：85/100
└─ 最近状态：第 149 章生成完成，质量合格
```

**技术实现**：
- WebSocket 实时推送进度
- 前端进度组件
- 预计完成时间算法
- 关键节点通知（完成、失败、需人工）

**验收标准**：
- [ ] 实时显示生成进度
- [ ] 显示预计剩余时间
- [ ] 关键节点通知

### 2.3 质量自动控制

**目标**：生成质量自动检查，不合格自动修复，保持 AI 味可控。

#### 2.3.1 实时质量检查器

**场景**：每章生成后自动检查质量

**检查维度**：
| 维度 | 检查内容 | 阈值 |
|------|----------|------|
| 字数 | 是否达到目标字数 | ±10% |
| 结构 | 是否有开头、发展、结尾 | 必须有 |
| 人物 | 人物是否出场、对话是否合理 | 必须有 |
| 情节 | 情节是否连贯、有冲突 | 必须有 |
| AI 味 | 是否有明显 AI 痕迹 | <30% |
| 一致性 | 与前文是否矛盾 | 0 矛盾 |

**技术实现**：
- 每章生成后自动运行检查
- 多维度评分（0-100）
- 综合质量分 = 各维度加权平均
- 质量分 < 阈值（默认 70）触发重新生成

**验收标准**：
- [ ] 每章自动检查 6 个维度
- [ ] 质量分实时显示
- [ ] 不合格自动触发重新生成

#### 2.3.2 自动重新生成

**场景**：生成内容质量不合格

**解决方案**：
```
质量检查不合格（分 < 70）
  ↓
自动重新生成（最多 3 次）
  ├─ 第 1 次：直接重新生成
  ├─ 第 2 次：调整参数（温度、top_p）
  └─ 第 3 次：切换模型
  ↓
3 次都不合格？
  ↓ 是
  ↓
标记为"需人工干预"
  ↓
通知用户，显示问题和建议
```

**技术实现**：
- 重新生成策略（参数调整、模型切换）
- 最大重试次数限制
- 失败后用户通知
- 问题诊断和建议

**验收标准**：
- [ ] 不合格内容自动重新生成
- [ ] 最多重试 3 次
- [ ] 3 次失败后通知用户

#### 2.3.3 AI 味自动检测

**场景**：控制生成内容的 AI 味

**检测方法**：
- 基于规则：检查常见 AI 痕迹（重复句式、过度修饰、缺乏情感等）
- 基于模型：用分类器判断是否像人写的
- 基于对比：与用户提供的样本对比风格

**自动调整**：
- AI 味 > 阈值（默认 30%）
- 自动调整生成参数（降低温度、增加随机性）
- 注入反 AI 规则重新生成
- 如果还不合格，触发重新生成流程

**验收标准**：
- [ ] AI 味实时检测
- [ ] 超阈值自动调整
- [ ] 保持在可控范围内

#### 2.3.4 一致性实时监控

**场景**：跨章节矛盾检测

**检查内容**：
| 矛盾类型 | 示例 | 检测方法 |
|----------|------|----------|
| 时间矛盾 | 第 5 章是 2020 年，第 10 章变成 2018 年 | 时间线对比 |
| 人物矛盾 | 第 3 章人物 A 已死，第 10 章又出现 | 人物状态追踪 |
| 空间矛盾 | 第 2 章在北京，第 8 章在上海但没写移动 | 地点追踪 |
| 逻辑矛盾 | 第 4 章说不会开车，第 15 章突然会了 | 技能/知识追踪 |

**技术实现**：
- 每章生成后运行一致性检查
- 与前文自动对比
- 矛盾自动标记
- 严重矛盾触发重新生成

**验收标准**：
- [ ] 4 类矛盾自动检测
- [ ] 矛盾自动标记
- [ ] 严重矛盾自动处理

### 5.3 数据库设计

```sql
-- 任务表
CREATE TABLE tasks (
  id UUID PRIMARY KEY,
  novel_id UUID NOT NULL,
  status ENUM('pending', 'running', 'completed', 'failed', 'paused'),
  total_chapters INT,
  completed_chapters INT,
  current_chapter INT,
  quality_score DECIMAL,
  error_message TEXT,
  created_at TIMESTAMP,
  updated_at TIMESTAMP
);

-- 检查点表
CREATE TABLE checkpoints (
  id UUID PRIMARY KEY,
  task_id UUID NOT NULL,
  chapter_number INT,
  state JSONB,  -- 完整状态快照
  created_at TIMESTAMP
);

-- 质量检查记录表
CREATE TABLE quality_checks (
  id UUID PRIMARY KEY,
  chapter_id UUID NOT NULL,
  dimension VARCHAR(50),  -- word_count, structure, character, plot, ai_smell, consistency
  score DECIMAL,
  details JSONB,
  passed BOOLEAN,
  created_at TIMESTAMP
);

-- 错误日志表
CREATE TABLE error_logs (
  id UUID PRIMARY KEY,
  task_id UUID,
  chapter_id UUID,
  error_type VARCHAR(50),  -- api, config, quality, system
  error_code VARCHAR(50),
  message TEXT,
  auto_recovered BOOLEAN,
  recovery_action VARCHAR(50),
  created_at TIMESTAMP
);
```

## 6. 风险与缓解

### 6.1 技术风险

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|----------|
| 自动重新生成循环 | 任务卡住 | 中 | 最大重试次数限制 + 人工干预 |
| 检查点数据量大 | 存储成本高 | 中 | 压缩 + 清理策略 |
| 质量检查误判 | 合格内容被重新生成 | 高 | 可调阈值 + 用户反馈 |
| 并发控制复杂 | 死锁或资源争用 | 低 | 成熟的队列框架 |

### 6.2 产品风险

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|----------|
| 自动化过度 | 用户失去控制感 | 中 | 保留手动干预选项 |
| 通知过多 | 用户疲劳 | 中 | 通知聚合 + 优先级 |
| 质量标准不一 | 用户不满意 | 高 | 可配置的质量标准 |
| 性能问题 | 生成速度慢 | 中 | 异步处理 + 缓存 |
