## v0.3 — 人物时间线管理 + 深度优化

**状态**: 💡 规划中  
**预估时间**: 2026-08-09 → 2026-09-05（4 周）  
**目标**: 在v0.2人物状态数据基础上，构建完整的人物时间线管理系统，支持长篇小说人物按时间顺序发展

### 核心价值

v0.2已建立人物状态数据基础（CharacterState表+一致性检查），v0.3在此基础上提供：
- **可视化**：让用户看到人物状态如何随章节变化
- **可控性**：让用户能编辑和管理人物发展轨迹
- **可追溯**：支持回溯编辑，自动维护一致性

### 人物状态更新机制：快照方案

**设计方案**：每章一个人物状态快照

```
时间线：
第1章开始前 → 第1章结束后 → 第2章结束后 → ... → 第N章结束后
    ↓              ↓              ↓                    ↓
  快照1          快照2          快照3                快照N

特点：
- 每章结束后保存一个快照
- 快照包含该章所有人物的完整状态
- 第N章开始前的状态 = 第N-1章结束后的状态（无需重复存储）
- 存储简单，查询高效
```

**为什么选择快照而非链表**：
1. **查询简单**：获取第X章的人物状态，直接查快照表，O(1)复杂度
2. **实现直观**：每章生成后提取状态，保存即可
3. **便于UI展示**：时间线就是快照的序列
4. **支持任意跳转**：可以快速跳到任意章节查看状态

**存储开销**：
- 160章小说 × 10个主要人物 × 平均1KB/人物状态快照 = 1.6MB
- 完全可以接受（相比小说文本本身）

#### M3: 状态同步机制（4天）

| 任务 | 内容 | 目标 | 优先级 |
|------|------|------|--------|
| REQ-7074 | 上下文注入 | 基于前一章人物状态生成当前章 | P1 |
| REQ-7075 | 状态一致性验证 | 确保生成内容与人物状态一致 | P1 |

**技术要点**：
```
生成第N章时：
1. 读取第N-1章的人物状态快照
2. 提取人物特征作为上下文
3. LLM生成第N章内容（参考人物状态）
4. 提取第N章的人物状态
5. 保存第N章快照
6. 对比前后章节，检测变化是否合理
```

**验收标准**：
- [ ] 生成内容参考人物状态
- [ ] 人物特征前后一致
- [ ] 变化有合理解释

#### M4: 回溯编辑支持（5天）

| 任务 | 内容 | 目标 | 优先级 |
|------|------|------|--------|

| REQ-7078 | 冲突检测与解决 | 标记与手动编辑的冲突，让用户选择 | P1 |

**回溯编辑流程**：
```
用户修改第10章的人物状态（江夜从"愤怒"改为"平静"）
  ↓
系统检测到：第11-50章的人物状态可能受影响
  ↓
自动重新计算：第11-50章的快照
  ↓
检测冲突：
  - 第11章写"江夜仍然愤怒" → 冲突
  - 第12章写"江夜渐渐平静" → 无冲突
  ↓
标记冲突：第11章需要修改
  ↓
用户选择：
  - 自动更新：将第11章的"愤怒"改为"平静"
  - 手动调整：用户自己修改第11章
```

**验收标准**：
- [ ] 支持编辑任意章节的人物状态快照
- [ ] 编辑后自动重算后续快照
- [ ] 冲突正确检测和标记
- [ ] 用户可以选择自动更新或手动调整

## 版本迭代原则

1. **功能冻结**: 每个版本中期进入功能冻结期（1 周），专注质量提升
2. **向后兼容**: 重大变更需提供迁移指南
3. **向后兼容**: API 变更需版本化管理（/v1/, /v2/）
4. **发布节奏**: 每 2-3 周发布一个版本（alpha/beta/rc）
5. **质量优先**: 宁可延期发布，不可带已知 P0/P1 问题发布

## 附录: 版本编号规则

采用语义化版本（Semantic Versioning）：

```
vMAJOR.MINOR.PATCH
  │       │     └── 修复（不影响功能）
  │       └──────── 新功能（向后兼容）
  └──────────────── 重大变更（可能破坏向后兼容）
```

**示例**:
- v0.1.0 → v0.1.1（修复 bug）
- v0.1.1 → v0.2.0（新增功能）
- v0.2.0 → v1.0.0（正式发布）
