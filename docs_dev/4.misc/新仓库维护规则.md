
# AI-Novel-Writing-Assistant 分叉项目标准化维护手册

## 文档说明

本项目基于上游开源项目深度二次魔改，自有代码与上游技术路线已严重分化，**不做全量分支合并**。采用「自有主线分支 + 纯净上游快照分支 + Git Worktree 独立原版目录」架构，借助AI分析上游更新，仅选择性摘取有效改动，避免大规模冲突。

## 1 仓库基础信息

### 1.1 远程仓库地址


| 远程别名   | 仓库地址                                                          | 读写权限       | 用途                                   |
| ------------ | ------------------------------------------------------------------- | ---------------- | ---------------------------------------- |
| `origin`   | https://github.com/Linearl/AI-Novel-Writing-Assistant             | 可读可写       | 个人Fork私有托管仓库，存放自有魔改主线 |
| `upstream` | https://github.com/ExplosiveCoderflome/AI-Novel-Writing-Assistant | 仅拉取禁止推送 | 上游原作者官方仓库                     |

### 1.2 分支规划（固定架构，不可随意改动）


| 分支名称        | 定位                           | 修改权限                  | 强制约束                                                                       |
| ----------------- | -------------------------------- | --------------------------- | -------------------------------------------------------------------------------- |
| `main`          | 自有魔改唯一主线，日常开发分支 | ✅ 允许提交自定义代码     | 所有二次开发、功能迭代、BUG修复仅在此分支；远端默认分支                        |
| `upstream-sync` | 上游纯净代码快照分支           | ❌ 禁止手动提交自定义代码 | 每次同步上游会强制重置对齐原版，本地临时修改会被清空；仅用于对比、提取上游更新 |

### 1.3 本地目录结构

1. **主工作目录**：本地现有带完整Git历史的项目文件夹，绑定`main`分支，日常编码调试、提交推送均在此操作。
2. **Worktree 独立上游目录**：同级目录 `../upstream-tmp`，永久绑定纯净`upstream-sync`分支，独立隔离原版代码，无需切换分支即可双版本共存对比。

## 2 首次初始化完整操作

适用场景：本地已有完整魔改Git历史，`git remote -v` 输出为空，全新配置双远程、分支、Worktree。

### 2.1 绑定双远程仓库

```bash
# 绑定个人Fork仓库 origin
git remote add origin https://github.com/Linearl/AI-Novel-Writing-Assistant
# 绑定上游原仓库 upstream
git remote add upstream https://github.com/ExplosiveCoderflome/AI-Novel-Writing-Assistant

# 校验远程配置
git remote -v
```

预期输出：

```
origin  https://github.com/Linearl/AI-Novel-Writing-Assistant (fetch)
origin  https://github.com/Linearl/AI-Novel-Writing-Assistant (push)
upstream        https://github.com/ExplosiveCoderflome/AI-Novel-Writing-Assistant (fetch)
upstream        https://github.com/ExplosiveCoderflome/AI-Novel-Writing-Assistant (push)
```

### 2.2 推送本地魔改历史至远端 main 主线

```bash
# 首次关联推送本地main分支到远端origin
git push -u origin main
```

### 2.3 创建纯净上游快照分支 upstream-sync

```bash
# 拉取上游全部提交、标签
git fetch upstream
# 基于上游main分支新建纯净同步分支
git checkout -b upstream-sync upstream/main
# 将纯净快照同步至个人Fork远端（网页端支持在线对比）
git push -u origin upstream-sync
# 切回自有开发主线main，回到日常开发环境
git checkout main
```

### 2.4 创建独立Worktree上游临时目录

```bash
# 在项目同级生成独立原版代码文件夹 upstream-tmp
git worktree add ../upstream-tmp upstream-sync

# 查看所有工作树，确认创建成功
git worktree list
```

执行后效果：

- 主目录保持`main`分支，自有魔改代码无变动；
- `../upstream-tmp` 独立文件夹存放纯上游原版代码，不受主目录分支切换影响。

## 3 定期同步上游标准流程

### 3.1 主目录刷新 upstream-sync 快照分支

```bash
# 拉取上游原仓库最新提交
git fetch upstream
# 切换至同步快照分支
git checkout upstream-sync
# 强制对齐上游最新代码，清空该分支所有临时改动
git reset --hard upstream/main
# 同步纯净快照至远端Fork（可选，用于网页在线查看）
git push origin upstream-sync --force-with-lease
# 切回自有开发主线main
git checkout main
```

### 3.2 刷新Worktree临时目录原版代码

```bash
# 进入独立上游目录
cd ../upstream-tmp
# 拉取最新同步后的纯净代码
git pull
# 返回主开发目录
cd 你的项目主目录文件夹名
```

## 4 AI 差异分析标准流程

### 4.1 导出结构化完整Diff文件（推荐交付AI解析）

在**主开发目录**执行，对比自有主线与纯净上游快照，生成差异文本文件：

```bash
git diff main upstream-sync > upstream_full_diff.txt
```

将 `upstream_full_diff.txt` 完整交付AI，要求AI输出分析报告，必须包含以下维度：

1. 上游本次新增功能模块、业务逻辑；
2. 上游修复的BUG、兼容性补丁、安全修复；
3. 上游架构、配置、依赖、脚本文件改动；
4. 与当前自有魔改代码存在冲突的文件、函数、变量；
5. 逐条评估改动是否适合引入自有版本，标注改造工作量、风险点；
6. 分类标注：可直接复用补丁、需要二次改造、完全不兼容废弃改动。

### 4.2 可视化文件夹人工辅助校验

使用VSCode「比较文件夹」功能同时打开两个目录：

1. 主目录：自有魔改版本
2. `../upstream-tmp`：上游最新原版
   直观查看新增/删除/修改文件，核对AI分析报告。

## 5 选择性引入上游改动规范

### 5.1 方式1：精准摘取上游提交（cherry-pick，优先推荐）

1. 查看上游新增提交列表，获取目标commit哈希
   ```bash
   git log upstream-sync --oneline
   ```
2. 在自有`main`分支摘取指定提交
   ```bash
   git cherry-pick 目标commit哈希值
   ```
3. 冲突处理规则：
   - 优先保留自有自定义逻辑；
   - 仅保留AI评估通过的上游修复/功能；
   - 完全冲突、不兼容的上游代码直接丢弃；
4. 冲突处理完成后正常提交推送。

### 5.2 方式2：手动复制代码片段（小规模改动）

打开`../upstream-tmp`原版文件，复制需要的函数、修复代码、配置片段，粘贴至自有项目对应文件，手动提交。

### 禁止操作

严禁执行 `git merge upstream-sync`，会一次性合并全部上游改动，造成大规模冲突、破坏分叉架构。

## 6 日常开发提交规范

1. 所有新增功能、优化、BUG修复仅在主目录`main`分支完成；
2. 提交注释统一规范：
   - `feat: 自定义新增功能`
   - `fix: 自有版本BUG修复`
   - `import: cherry-pick 引入上游XX补丁`
   - `refactor: 自有代码重构`
3. 提交后推送远端自有仓库：
   ```bash
   git push origin main
   ```

## 7 Worktree 常用管理命令

```bash
# 查看全部工作树
git worktree list

# 不需要临时上游目录时删除（会删除文件夹内原版代码）
git worktree remove ../upstream-tmp

# 新增多版本Worktree用于对比上游历史Tag版本
git worktree add ../upstream-v1.0 v1.0
```

## 8 长期维护硬性红线（交付AI执行必须遵守）

1. **禁止在 upstream-sync 分支、upstream-tmp Worktree 目录编写任何自定义代码**，同步上游时会强制重置清空；
2. 不向上游原仓库提交PR，本项目技术路线已完全分叉，无合并价值；
3. 禁止执行全分支合并命令 `git merge upstream-sync`；
4. 同步上游操作仅用于分析评估，所有上游改动必须人工+AI审核后选择性引入；
5. `main` 分支永久作为自有唯一主线，不切换其他分支用于日常开发。

## 9 故障恢复方案

### 9.1 误操作在 upstream-sync 写入自定义代码

直接执行3.1章节同步重置流程，临时修改会被覆盖，无需额外处理，该分支仅作快照。

### 9.2 自有main分支代码被污染，需要回退

```bash
# 查看提交历史，找到正常版本哈希
git log --oneline
# 软回退至指定版本，保留本地文件
git reset 正常commit哈希
# 硬回退（彻底丢弃后续所有改动，谨慎使用）
git reset --hard 正常commit哈希
git push origin main --force-with-lease
```
