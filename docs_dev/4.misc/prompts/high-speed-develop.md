## 模型配置
- 模型A：`fable`（mimo-v2.5-pro-ultraspeed，高速-能力强-昂贵，用于高风险/关键路径）
- 模型B：`opus`（mimo-v2.5-pro，中速-能力强-价格中等，用于复杂任务）
- 模型C：`sonnet`（mimo-v2.5，快速-能力够用-价格便宜，用于简单任务/并行批量）

## 执行策略

### 第一阶段：规划与分解（主对话，不派发开发任务）
1. 扫描所有任务包，读取每个包的 REQ 和 tasks.md
2. 评估每个任务包的复杂度（简单/复杂）、开发量（小/大）、风险等级（低/高/极高）
3. 尝试拆解复杂任务为多个简单子任务
4. 制定执行计划，写入 docs_dev/1.task/todolist/，包含：
   - 任务包清单与评估结果
   - 模型分配方案（A/B/C）
   - 执行顺序（优先关键路径）
   - 并行批次规划

### 第二阶段：分配执行（子代理）
使用 ll-workflow-core dev 组织开发，按以下矩阵分配：

| 场景 | 模型 |
|------|------|
| 简单/边界清晰/小开发量 | C (sonnet) |
| 简单/边界清晰/大开发量 | 多个 C 并行 |
| 复杂/需理解上下文 | B (opus) |
| 高风险/涉及核心架构/安全 | A (fable) |
| 复杂大任务 | 主代理先拆解 → 多个 C |

### 第三阶段：监督与升级
- C 模型任务：每 10-15 分钟检查进度，卡住升级到 B
- B 模型任务：每 20-30 分钟检查进度
- A 模型任务：关键节点检查
- 升级条件：连续 2 轮无进展 或 遇到非预期跨模块依赖

### 第四阶段：验证与收尾
- 每个任务包完成后：typecheck + 相关测试
- 全部完成后：pnpm typecheck && pnpm test:all
- 汇总执行报告到 docs_dev/1.task/todolist/

## 约束
- 主对话不承担具体开发，只负责协调、检查、汇报
- 串行关键路径用 A/B，非关键路径用 C
- 记录每个任务的实际难度 vs 预估难度，用于后续优化
