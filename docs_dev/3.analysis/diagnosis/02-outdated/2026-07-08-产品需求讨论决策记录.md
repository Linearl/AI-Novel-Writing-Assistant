# AI Novel 产品需求讨论 — 决策记录

> 日期：2026-07-08
> 方法论：王慧文清华产品课框架 + grill-with-docs

---

## 决策 1：目标用户

**选择**：A（网文新人）+ E（效率型作者）

**理由**：
- 核心诉求是"快速产出成体量的文字"
- 起点读者对质量有要求，番茄批量生成已是红海 → 质量不能降
- 用户角色定位：作者负责审核和精修，AI 负责产出初稿

**排除的用户群**：
- B（人生记录者）— 不是核心目标
- C（文学爱好者）— 需求差异大
- D（好奇尝鲜者）— 不匹配

---

## 决策 2：文学质量体系保留

**选择**：保留当前质量体系（伏笔、连贯性、角色一致性等）

**理由**：
- 起点读者对质量有底线要求，太烂没人看
- 番茄批量机器生成已是红海，差异化靠质量
- 用户定位是"审核+精修"而非"全部重写"，初稿质量直接影响用户时间成本

---

## 决策 3：新增核心需求 — 章节质量差异化

**需求描述**：不同章节对质量的要求不同，读者不能接受整本书都是"白开水"，中间需要"酸辣粉"和"鸡汤"。

**含义**：
- 当前质量体系是均匀施加的（每章同样的质量标准）
- 需要引入"章节节奏张力"概念：高潮章节需要更高质量标准，过渡章节可以适度放松
- 这是**节奏控制**层面的质量差异化，不是降低质量

---

## 决策 4：新增核心需求 — 写作手法（文笔）一致性

**问题**：当前 AI 输出的文笔一致性过高（AI 味重），用户每次手动修改意味着时间成本大幅提升。

**改进方向**：

### 方向 A：外部资料引导
- 从外部资料（范文、写作指南）收集写法
- 建立结构化的"写作手法库"
- AI 根据剧情语境自动匹配合适的写法

### 方向 B：从人类修改中学习
- AI 审视用户对生成内容的修改
- 总结用户的修改意图
- 提炼出结构化的写法规则
- 保证后续生成的文笔风格与用户偏好一致

**两个方向的关系**：A 是冷启动（新用户没有修改历史时），B 是持续优化（有修改历史后逐步替代 A）

---

## 决策 5：功能取舍

### 漫画/短剧 → 移除

**选择**：从当前产品中移除漫画和短剧模块。

**理由**：
- 技术成本极高（图像生成、视频生成、配音合成）
- A+E 用户（网文作者）核心需求是文字产出，不是跨媒体转化
- 当前阶段属于 PMF 未验证时的功能蔓延

### 拆书 → 保留

**选择**：保留拆书功能，但不作为当前阶段重点。

**理由**：
- 拆书能提炼写法、世界观、套路结构 → 直接服务于"文笔一致性"需求（见决策4-方向A）
- 拆书结果可作为 AI 生成时的风格参考和结构参考
- 与"外部资料引导"方向天然互补：拆书 = 从已有作品中提取结构化写法

**与决策4的关系**：拆书是"外部资料引导（方向A）"的核心数据来源之一。

---

## 决策 6：开书体验 — 统一入口 + 灵活导入

**选择**：保留一个入口，但增强创建页的素材导入灵活性。

**现状**：
- 步骤间链式传递已实现（步骤1数据注入步骤2/3/4的LLM生成）
- Tab可自由切换，不是强制向导
- 限制：世界观只能在步骤1管理，角色只能在步骤3导入

**改造方向**：
- 创建页允许用户一次性粘贴大段素材（世界观、角色、大纲等）
- 系统自动拆分到对应字段，而非只能在固定表单里逐项填写
- 不需要做"任意步骤都能导入任何类型素材"的大改造

**理由**：链式传递已满足主流程，瓶颈在入口的灵活性，不在步骤间的传递。

---

## 决策 7：文笔体系闭环 — 两个补丁点

**现状**：styleEngine 基础设施完善，消费侧（生成注入）已闭环，生产侧（提取触发）需手动。

**需要补齐的两个自动闭环**：

### 补丁 A：章节编辑后自动提取风格

- 在 `updateChapter` 末尾或通过 `chapter:updated` 事件 handler 调用 `ChapterEditDiffService`
- 用户每次修改章节后，自动提取反AI规则和写法偏好
- 逐步积累用户风格档案，减少后续生成的修改量

### 补丁 B：拆书完成后自动转化为 StyleProfile

- 在 bookAnalysis pipeline 完成事件中调用 `StyleProfileService.createFromBookAnalysis`
- 拆书结果自动变成可用的风格档案，供后续写作引用

**与决策4的关系**：补丁A = 方向B（从人类修改中学习）的自动化；补丁B = 方向A（外部资料引导）的自动化。

## 决策 8：商业化路径

**选择**：当前阶段自用为主，后续通过独立闭源增强模块收费。

**许可证约束**：
- 上游项目 AGPL-3.0-only，fork 继承
- AGPL 主体的所有修改必须开源
- 独立进程通过 API 通信的增强模块可以闭源（"聚合"而非"衍生作品"）

**架构规划**：
- AGPL 开源部分：核心写作引擎、自动导演、章节生成/审校、基础风格引擎、桌面端
- 闭源增强模块（独立进程 + API 通信）：高级文笔分析、拆书深度分析、平台套路模板库、批量生产调度、云端协作、模型托管
- 主程序需要预留开放的插件接口

**当前优先级**：商业化不是当前重点，先做好产品本身。但架构上需要提前规划插件接口，为后续闭源增强模块留空间。

## 决策 9：写法模板市场（规模效应路径）

**选择**：建立写法模板共享机制，后续发展为 Marketplace。

**核心逻辑**：
- 用户 A 用拆书功能提炼了一套"都市爽文"写法 → 存入写法库
- 用户 B 想写都市爽文 → 直接引用 A 的写法模板，省去冷启动
- 用户越多 → 模板越丰富 → 新用户体验越好 → 吸引更多用户（正循环）

**参考模式**：
- 酒馆（SillyTavern）角色卡市场
- Skill 市场
- Obsidian 社区插件库

**实现路径**：
1. **第一步**：StyleProfile 支持导出/导入 JSON（当前已有基础）
2. **第二步**：建立社区共享平台（GitHub Discussions / 独立网站）
3. **第三步**：产品内集成市场入口（浏览、一键导入、评分、分类）

**护城河分析**：
- 全书状态管理 → ChatGPT 做不到（技术护城河）
- 写法模板市场 → 用户越多越好用（网络效应护城河）
- 两者叠加 = 既有技术壁垒又有规模效应

## 决策 10：灵感页快速预览

**选择**：在灵感页提供 500 字预览 / 前三章快速生成功能。

**来源**：诊断报告中的"零认知开书体验"方案，经讨论后确认纳入。

**核心逻辑**：
- 当前从灵感到看到第一段文字需要 10+ 分钟 → 用户流失率极高
- 快速预览让用户在 30 秒内"看到"自己的故事，降低心理门槛
- 选定方向后再进入正式的完整流程

**功能设计**：
- 灵感页输入框旁提供"快速预览"按钮
- 用户输入灵感 → AI 生成 3 个方向候选（标题 + 梗概 + 前 500 字预览）
- 用户选定方向后可选择"继续写前 3 章"（快速生成模式）
- 满意后一键转入正式的自动导演流程

**验收标准**：新用户从打开产品到看到自己故事的第一段文字 < 2 分钟。

**与决策6的关系**：决策6解决"素材导入灵活性"，本决策解决"即时体验"。两者互补 — 快速预览降低门槛，素材导入提升精度。

## 附录：讨论细节确认

### A. 快速预览与统一入口的关系

**确认**：快速预览放在创建页（NovelCreate）内部，作为"输入灵感后"的默认体验。

- 创建页 = 统一入口
- 入口内有两种模式：快速预览（默认）/ 正式开书
- 用户看到预览满意后，点击"正式开书"进入编辑工作台
- 不是独立的新页面，和决策 6 不矛盾

### B. 节奏张力的实现粒度

**确认**：AI 自动标记 + 用户可手动调整。

- 步骤 4（卷战略/卷骨架）生成 beatSheet 时，AI 自动为每个章节标记张力等级
- 不需要用户手动确认（减少操作负担）
- 用户可以在章节列表中手动调整张力等级（保留控制权）
- 质量系统根据张力等级调整审校标准

### C. 任务包优先级

| 顺序 | 任务包 | 理由 |
|------|--------|------|
| 1 | 移除漫画/短剧 | 最快完成，释放代码库空间 |
| 2 | 文笔闭环 | A+E 用户核心痛点 |
| 3 | 灵感页快速预览 | PMF 验证前提 |
| 4 | 创建页素材导入 | 提升开书效率 |
| 5 | 章节节奏张力 | 质量提升 |
| 6 | 写法模板导出/导入 | Marketplace 铺路 |
