# 代码审计-full 独占发现 — 对比全量审计

## 概要

本文档对比了两个审计报告：

- **全量代码审计-full**（`audit-findings.json`）：150 条发现，9 维度
- **代码审计-full**（`scan-findings.json`）：164 条原始发现（去重后 134 条唯一），9 维度

两个审计使用**独立的 ID 编号体系**且粒度不同——代码审计逐文件枚举具体位置，全量审计更模式化。经 ID + 标题语义 + 文件+维度交叉匹配后，全量审计**未能覆盖代码审计中的约 79 条发现**。

## 对比方法

1. 提取全量审计发现特征库（ID、标题核心词、文件路径、维度）
2. 逐条交叉验证：
   - 相同 ID → 匹配
   - 标题核心词重叠 ≥60% 且同维度 → 语义匹配
   - 同文件 + 同维度 + ≥3 标题词重叠 → 上下文匹配
   - 剩余项 → 独占发现

匹配成功约 55 条（跨编号体系），独占约 79 条。

## 差异根因分析

1. **粒度差异**：全量审计以"模式"为单位聚合（"14 处 .catch()"、"30 个超大文件"），代码审计以"具体文件/行号"为单位标注。后者对修复执行更有指导价值。

2. **兼容性覆盖盲区**：全量审计偏重 Node 运行时和模型列表兼容性，对 Desktop 跨平台（XDG、.ico、updater）、浏览器兼容（Safari）、数据库方言差异（SQLite vs PostgreSQL boolean）、API 版本化（/api/v1/）等子维度几乎没覆盖。

3. **遗漏循环依赖**：全量审计仅标注 planner ↔ novel 一条循环引用，遗漏了 llm/factory↔settings、DirectorCommandService↔workers、structuredInvoke↔novelP0Utils 三条同等严重的循环链路。

4. **客户端性能被忽略**：全量审计的性能发现全是服务端（无分页查询、串行 DB），客户端 N+1 查询、useQuery staleTime 缺失、7 个并发 useQuery 等未覆盖。

5. **Desktop 包未被充分审计**：COMPAT-007、011~013 和 QUA-040 均涉及 `desktop/` 包。全量审计虽声称覆盖 desktop，但实际发现中无 Desktop 跨平台兼容问题。
