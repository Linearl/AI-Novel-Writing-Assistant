# AI Novel 项目主工作流诊断报告

> 日期：2026-06-26
> 范围：项目端到端工作流全景分析

---

## 1. 项目概览

AI Novel 是一个 AI 辅助长篇小说创作平台，支持从故事构思到章节正文生成的全流程自动化。项目采用 monorepo 结构（shared / server / client / desktop），后端基于 Node.js + Prisma + PostgreSQL，前端为 React + Vite，可选 Electron 桌面外壳。

核心设计理念：**两条创作通道并行**

| 通道 | 说明 |
|------|------|
| `manual_create` | 用户手动驱动每一步，AI 辅助 |
| `auto_director` | AI Director 全自动编排整条流水线，关键节点挂检查点等待人工确认 |

## 3. 各阶段详解

### 3.1 阶段 1：故事宏观规划 (Story Macro Plan)

| 项 | 值 |
|----|-----|
| 入口 | `StoryMacroPlanService.decompose(novelId, storyInput)` |
| 输入 | 用户自由文本故事创意 (`storyInput`) |
| 处理 | 加载小说上下文 + 世界上下文 → LLM 分解 → 产出 expansion + decomposition |
| 输出 | `StoryMacroPlan` 记录（storyInput / expansionJson / decompositionJson / constraintEngineJson） |
| 子操作 | `regenerateField()` 单字段重新生成；`updatePlan()` 手动编辑；`buildConstraintEngine()` 重建约束引擎 |
| 关键产物 | 约束引擎 (constraintEngine)：阶段模型 + 故事约束，供下游章节生成使用 |

**Auto Director 模式**：自动生成候选方向 → 用户选择 → 运行 decompose → 生成书级阅读契约 (Book Contract)

---

### 3.2 阶段 2：角色准备 (Character Setup)

| 项 | 值 |
|----|-----|
| 入口 | `NovelCoreCharacterService` |
| 手动模式 | 用户逐步创建角色，可关联基础角色库 (`BaseCharacter`) |
| Auto Director | 生成 `CharacterCastOption`（多候选角色方案）→ 质量评估 → 通过则批量创建，不通过则挂检查点 `character_setup_required` |
| 时间线同步 | `syncCharacterTimeline()` 扫描所有章节提取角色事件 |
| 角色进化 | `evolveCharacter()` 基于时间线 + RAG 上下文更新角色状态 |
| 世界校验 | `checkCharacterAgainstWorld()` 校验角色是否符合世界规则 |

**数据模型**：Character / CharacterTimeline / CharacterCastOption / CharacterRelation / CharacterRelationStage / CharacterVolumeAssignment 等 10+ 表

---

### 3.3 阶段 3：卷战略 / 卷骨架 (Volume Strategy / Skeleton)

| 项 | 值 |
|----|-----|
| 入口 | `NovelVolumeService.generateVolumes()` |
| 编排器 | `volumeGenerationOrchestrator.generateVolumePlanDocument()` |
| 生成粒度 | `"strategy"`（卷级弧线）→ `"skeleton"`（每卷章节列表）→ `"chapter_list"` → `"chapter_detail"` |
| 核心数据结构 | `VolumePlanDocument`：volumes[] → chapters[] + beatSheets[] |
| 版本管理 | `VolumePlanVersion`：draft → active → frozen，支持激活/冻结切换 |
| 章节回写 | `syncVolumeChapters()` 将卷规划章节数据同步到 `Chapter` 记录 |

**Auto Director**：两遍生成（先 strategy，再 skeleton），通过后挂检查点 `volume_strategy_ready`

---

### 3.4 阶段 4：节奏 / 拆章 (Structured Outline)

| 项 | 值 |
|----|-----|
| 入口 | Auto Director: `runDirectorStructuredOutlinePhase()` |
| 处理 | 逐卷生成 beat sheet → 章节列表 → 章节详情（标题、摘要、冲突级别、揭示级别、任务卡、场景卡） |
| 产物 | `BeatSheet` + 章节级 `VolumeChapterPlan` |
| 执行契约 | `ChapterExecutionContractService.ensureChapterExecutionContract()` 为每章生成 taskSheet + sceneCards |

---

### 3.5 阶段 5：章节执行 (Chapter Execution)

这是最复杂的阶段，包含三层子系统：

#### 5a. 章节规划 (Planner)

| 项 | 值 |
|----|-----|
| 入口 | `PlannerService` |
| 规划层级 | `book`（书级）→ `arc`（弧线级）→ `chapter`（章节级） |
| 上下文来源 | 卷规划、伏笔账本、角色动态、故事宏观、风格引擎、状态快照 |
| 输出 | `StoryPlan` 记录 |
| 重规划 | `plannerService.replan()`：审计问题超阈值时触发，确定受影响章节窗口，重新生成计划 |

#### 5b. 章节内容生成

| 模式 | 流程 |
|------|------|
| **单章流式** | `ChapterRuntimeCoordinator.createChapterStream()` → 组装上下文 → 校验就绪 → LLM 流式生成 → 保存内容 + 同步产物 + 记录质量 |
| **批量管线** | `NovelCorePipelineService.startPipelineJob()` → 创建 `GenerationJob` → 循环执行每章 → 每章完成后质量审查 + 质量循环评估 + JIT 预取下一章任务卡 |

**上下文组装** (`GenerationContextAssembler.assemble()`)：小说信息、章节数据、RAG 检索结果、状态快照、风格绑定

#### 5c. 章节状态机（双轨制）

| 字段 | 状态链 |
|------|--------|
| `generationState` | `planned` → `drafted` → `reviewed` → `repaired` → `approved` → `published` |
| `chapterStatus` | `unplanned` → `pending_generation` → `generating` → `pending_review` → `needs_repair` → `completed` |

状态对由 `chapterLifecycleState.ts` 统一管理，必须成对更新。

---

### 3.6 阶段 6：质量修复 (Quality Repair)

#### 审查 (Review)

| 项 | 值 |
|----|-----|
| 入口 | `NovelCoreReviewService.reviewChapter()` |
| 处理 | 组装审计上下文 → `auditService.auditChapter()` → 产出 `QualityScore`（连贯性/重复/节奏/语调/吸引力/综合）+ `ReviewIssue[]` |
| 存储 | `QualityReport` 记录 |
| 严重问题 | 触发 `plannerService.replan()` |

#### 质量循环 (Quality Loop)

| 项 | 值 |
|----|-----|
| 入口 | `ChapterQualityLoopService.recordAssessment()` |
| 评估 | 连续性信号 + 角色一致性 + 预算（尝试次数 vs 最大次数） |
| 决策 | `recommendedAction`: `continue` / `repair` / `defer_and_continue` |

#### 修复 (Repair)

| 项 | 值 |
|----|-----|
| 入口 | `NovelCoreReviewService.createRepairStream()` |
| 修复模式 | `detect_only` / `light_repair` / `heavy_repair` / `continuity_only` / `character_only` / `ending_only` |
| 流程 | 组装上下文 → LLM 流式修复 → 重新审查 → 仍不通过可触发重规划 |

#### 状态管理 (State Service)

| 操作 | 说明 |
|------|------|
| `getNovelState()` | 读取权威状态 |
| `getLatestSnapshot()` | 最新 `StoryStateSnapshot` |
| `rebuildState()` | AI 重新从章节中提取状态 |
| `detectStateDiffConflicts()` | 检测章节间状态冲突 |

## 4. 状态机全景

项目中有 **7+ 个独立状态机**，通过 `NovelWorkflowTask` 串联：

| 状态机 | 模型 | 状态流 |
|--------|------|--------|
| **工作流任务** | `NovelWorkflowTask` | `queued` → `running` → `waiting_approval` → `succeeded`/`failed`/`cancelled` |
| **章节生成** | `Chapter` | 双轨：`generationState` + `chapterStatus` |
| **生成任务** | `GenerationJob` | `queued` → `running` → `succeeded`/`failed`/`cancelled` |
| **Director 运行时** | `DirectorRuntimeInstance` | 三层架构：Task Dispatch → Worker Consumer → Persistence |
| **RAG 任务** | `RagJob` | `queued` → `running` → `succeeded`/`failed`/`cancelled` |
| **书籍分析** | `BookAnalysis` | `draft` → `queued` → `running` → `succeeded`/`failed`/`cancelled`/`archived` |
| **知识索引** | `KnowledgeIndex` | `idle` → `queued` → `running` → `succeeded`/`failed` |
| **Agent 运行** | `AgentRun` | `queued` → `running` → `waiting_approval` → `succeeded`/`failed`/`cancelled` |

**工作流阶段 (NovelWorkflowStage) 进度映射**：

```

auto_director      0.15
story_macro        0.26
character_setup    0.34
volume_strategy    0.50
structured_outline 0.68
chapter_execution  0.84

```

**检查点 (NovelWorkflowCheckpoint)**：

```
candidate_selection_required → book_contract_ready → character_setup_required
```

## 5. Auto Director 流水线（端到端编排）

Auto Director 是项目的旗舰功能，将所有阶段串联为全自动流水线：

```
1. 候选方向生成 (novelDirectorCandidateStage)
   │  生成多个候选方向 → 等待用户选择 / 自动选择
   │  检查点: candidate_selection_required
   ▼
2. 书级契约 + 故事宏观 (novelDirectorStoryMacroPhase)
   │  生成 story macro plan + book contract
   │  检查点: book_contract_ready
   ▼
3. 角色阵容 (runDirectorCharacterSetupPhase)
   │  生成 cast options → 质量评估 → 应用
   │  检查点: character_setup_required (评估不通过时)
   ▼
4. 卷战略 (runDirectorVolumeStrategyPhase)
   │  两遍生成: strategy → skeleton
   │  检查点: volume_strategy_ready
   ▼
5. 结构化大纲 (runDirectorStructuredOutlinePhase)
   │  逐卷: beat sheet → 章节列表 → 章节详情
   ▼
6. 章节执行
   │  触发批量章节生成管线 (NovelCorePipelineService)
   │  检查点: chapter_batch_ready
   ▼
7. 质量修复
   │  监控质量 → 触发修复循环
   │  检查点: replan_required (严重问题时)
   ▼
完成
   检查点: workflow_completed
```

**Director 运行时架构**（三层）：

| 层 | 组件 | 职责 |
|----|------|------|
| 任务调度 | `TaskDispatcher` + `DirectorTaskQueue` | 事件总线唤醒 worker |
| Worker 消费 | `DirectorWorker` | `waitForWork → leaseNext → acquireResourceGate → markRunning → executeCommand → completeTask` |
| 持久化 | `DirectorCommandService` / `DirectorRuntimeStore` | 命令队列、执行记录、检查点、事件日志 |

## 8. 工作流设计评价

### 优势

1. **双轨并行**：manual_create 与 auto_director 共享同一套阶段定义和数据模型，用户可以随时切换
2. **检查点机制**：Auto Director 在关键节点挂 `waiting_approval`，允许人工介入而不丢失上下文
3. **版本化规划**：VolumePlanVersion 支持 draft → active → frozen，可安全回滚
4. **双轨章节状态**：generationState（管线语义）+ chapterStatus（编辑语义）分离关注点
5. **质量循环**：审查 → 评估 → 修复/重规划的闭环，支持预算控制和延迟继续
6. **JIT 预取**：批量管线中提前生成下一章的任务卡，减少等待

### 风险点

1. **状态机数量多（7+）**：各状态机独立演进，缺乏统一的状态图文档，维护者需要逐个理解
2. **服务门面层厚**：`NovelCoreService` 聚合了过多子服务，新开发者容易迷路
3. **Legacy 共存**：旧版 outline 系统 (`createOutlineStream`) 与新版卷规划系统并存，存在功能重叠
4. **Prisma 模型膨胀**：Chapter 模型有 30+ 字段，混合了内容、生成参数、质量分数、状态等不同关注点
