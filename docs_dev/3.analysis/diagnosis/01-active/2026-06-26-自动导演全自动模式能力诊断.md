# 自动导演全自动模式能力诊断

> **诊断范围**：卷战略和节奏板（volume_strategy + structured_outline）之后，系统是否支持全自动执行链。
>
> **诊断目标**：回答三个问题——
> 1. 自动审校和修复是否已实现？
> 2. 能否按风险等级设定哪些环节需上报人工决策？
> 3. 自动导演任务运行中能否调整参数？

| # | 能力 | 后端 | 前端 | 评级 |

## 二、能力逐项诊断

### 2.1 自动审校和修复

#### 2.1.1 后端实现

**状态：✅ 完整实现**

自动审校修复是 `full_book_autopilot` 模式的核心能力，由三层防御体系驱动：

**第一层：章节生成后自动审校**

| 组件 | 职责 | 关键文件 |
|------|------|----------|
| Pipeline Job | 章节批量生成，完成后自动触发审校 | `novelDirectorAutoExecutionRuntime.ts` |
| Review API | 单章审校（连贯性/角色/情节/全维度） | `novelReviewRoutes.ts` |
| Repair API | 单章修复（detect_only / light_repair / heavy_repair 等 6 种模式） | `novelReviewRoutes.ts` |

执行计划中的开关：
```typescript
// shared/types/novelDirector.ts — AutoExecutionPlan
autoReview: boolean    // 正文后自动审核
autoRepair: boolean    // 审核不通过时自动修复（依赖 autoReview=true）
```

**第二层：质量修复风险评估**

文件：`server/src/services/novel/director/phases/novelDirectorQualityRepairRisk.ts`

| 风险等级 | 含义 | 自动继续 | 触发条件 |
|----------|------|----------|----------|
| `low` | 轻微质量问题 | ✅ 是 | 轻度修复模式 + 受影响章节 <25% |
| `large_scope` | 大范围修复 | ❌ 暂停 | heavy_repair 模式或受影响章节 ≥25% |
| `replan` | 需要重新规划 | ❌ 暂停 | 质量问题影响执行路径 |

**第三层：质量循环预算 + 断路器**

| 组件 | 阈值 | 超限行为 |
|------|------|----------|
| `patchRepair` 预算 | 2 次/章节 | 升级到 chapterRewrite |
| `chapterRewrite` 预算 | 1 次/章节 | 升级到 windowReplan |
| `windowReplan` 预算 | 1 次/窗口 | 升级到断路器 |
| 断路器 `patchFailureOpenAt` | 3 次同章 | 打开断路器 |
| 断路器 `replanLoopOpenAt` | 3 次循环 | 打开断路器 |
| 断路器 `modelFailureOpenAt` | 3 次 | 打开断路器 |
| 章节 Token 上限 | 80,000 | 打开断路器 |

**全自动驾驶（full_book_autopilot）特有行为：**
- 断路器因 `auto_repair_exhausted` 或 `replan_loop` 打开时 → **不暂停**，记录质量债务（deferred quality debt）并继续下一章
- 自动解析 state proposals（replan / skip 决策）→ 无人值守
- 允许 lazy chapter planning（按需生成章节任务书）

#### 2.1.2 前端实现

**状态：✅ 有 UI 入口**

| UI 组件 | 功能 | 位置 |
|---------|------|------|
| `directorAutoExecutionPlan.shared.tsx` | autoReview / autoRepair 开关 | 创建任务时配置 |
| `ChapterExecutionResultPanel.tsx` | "运行完整审校" + "一键修复" 按钮 | 单章级别手动触发 |
| `AutoDirectorFollowUpDetail.tsx` | "一键安全修复" + "一键重新校验" | 跟进中心 |
| `DirectorRuntimeProjectionCard.tsx` | 质量债务摘要 + 质量预算摘要 | 进度面板实时展示 |

#### 2.1.3 缺口

| 缺口 | 严重度 | 说明 |
|------|--------|------|
| 无流式进度展示 | 低 | 主进度面板仅展示步骤状态和历史事件，不展示 AI 当前生成内容的实时流 |
| 无质量阈值可调 UI | 中 | `qualityThreshold` 在 pipeline API 中已定义（0-100），但前端无配置入口 |
| 无回滚机制 | 中 | 系统仅支持前向修复，章节一旦生成提交无法撤销 |

### 2.3 运行中参数调整

#### 2.3.1 后端实现

**状态：⚠️ 部分实现**

**可运行中调整的参数（通过 `policy_update` 命令）：**

```typescript
// shared/types/directorRuntime.ts
interface DirectorRuntimePolicyUpdateRequest {
  mode: "suggest_only" | "run_next_step" | "run_until_gate" | "auto_safe_scope"
  mayOverwriteUserContent?: boolean
  allowExpensiveReview?: boolean
  modelTier?: "cheap_fast" | "balanced" | "high_quality"
}
```

| 参数 | 可调 | 说明 |
|------|------|------|
| `mode`（推进模式） | ✅ | 4 种模式可随时切换 |
| `mayOverwriteUserContent` | ✅ | 是否允许覆盖用户手动编辑内容 |
| `allowExpensiveReview` | ✅ | 是否允许深度审校 |
| `modelTier` | ✅ | 模型档位切换 |
| `provider` / `model` / `temperature` | ⚠️ | 仅在 resume 时可改，非实时热切换 |
| `autoExecutionPlan`（范围/开关） | ⚠️ | 仅在 resume 时可重新指定，当前批次执行中不可改 |
| 审批点配置 | ❌ | 任务创建后不可修改 |

API 端点：
```
POST /api/novels/director/tasks/:taskId/commands
Body: { commandType: "policy_update", ...policyFields }
```

## 三、架构全景

### 3.1 三层防御模型

```
┌─────────────────────────────────────────────────────────┐
│                    策略引擎 (Policy Engine)               │
│  逐节点决策：模式 + 风险标签 + 质量状态 → 放行/阻断       │
│  文件：DirectorPolicyEngine.ts                           │
├─────────────────────────────────────────────────────────┤
│                 质量门 (Quality Gate)                     │
│  逐批次决策：审校结果 → 风险评估 → 自动继续/暂停          │
│  文件：novelDirectorQualityRepairRisk.ts                 │
├─────────────────────────────────────────────────────────┤
│                 断路器 (Circuit Breaker)                  │
│  跨章节累积：失败信号计数 → 阈值超限 → 停止 + 恢复建议    │
│  文件：DirectorCircuitBreakerService.ts                  │
└─────────────────────────────────────────────────────────┘
```

### 3.2 全自动驾驶状态机

```
[范围解析] → [启动 Pipeline Job] → [轮询 Job 状态]
  ├─ 成功（无问题）→ 继续循环 或 记录完成
  ├─ 成功（有质量通知）→ 质量修复风险评估
  │    ├─ low + 自动继续策略 → 继续循环
  │    └─ large_scope / replan → 暂停（记录 checkpoint）
  └─ 失败/取消 → 断路器评估
       ├─ 预算允许修复 → 带修复模式重试
       ├─ 预算耗尽 + full_book_autopilot → 延迟质量债务，继续
       ├─ 预算耗尽 + 其他模式 → 暂停（记录 checkpoint）
       └─ 断路器打开 → 停止 + 恢复动作
```

### 3.3 运行模式对比

| 特性 | stage_review | auto_to_ready | auto_to_execution | full_book_autopilot |
|------|-------------|---------------|-------------------|---------------------|
| 规划阶段 | 逐步暂停 | 自动 | 自动 | 自动 |
| 章节执行 | 手动 | 手动启动 | 按范围自动 | 全书自动 |
| 质量审校 | 手动 | 手动 | 批次间 | 自动 |
| 质量修复 | 手动 | 手动 | 手动/自动 | 自动 |
| 质量债务延迟 | ❌ | ❌ | ❌ | ✅ |
| 断路器行为 | 暂停 | 暂停 | 暂停 | 延迟债务后继续 |
| 初始策略模式 | run_next_step | run_until_gate | run_until_gate | auto_safe_scope |

## 六、回答原始问题

### Q1: 步骤 5（卷战略和节奏板）之后是否支持自动审校和修复？

**答：支持。** `full_book_autopilot` 模式下，章节生成完成后自动触发审校，审校不通过自动触发修复。修复有预算限制（patch ×2 → rewrite ×1 → replan ×1），超限后记录质量债务继续下一章。三层防御（策略引擎 → 质量门 → 断路器）确保不会陷入无限修复循环。

### Q2: 能否设置需要上报人工决策的风险等级？

**答：能。** 系统提供 8 个审批点，按 low/medium/high 三级风险分类。用户可在任务创建时通过多选组件选择哪些审批点自动批准（默认仅 4 个 low 风险点）。medium 和 high 风险点默认需人工确认。此外，质量修复风险 `large_scope` 和 `replan` 级别会自动暂停等待人工决策。支持钉钉/企业微信通知渠道远程审批。

### Q3: 已开启的自动导演任务能否调整参数？

**答：部分能。** 运行中可通过 `policy_update` 命令实时调整：推进模式（4 种）、是否允许覆盖用户内容、是否允许深度审校、模型档位。但执行范围、autoReview/autoRepair 开关、审批点配置、LLM 具体模型在任务创建后不可运行中修改，需等 resume 时重新指定。
