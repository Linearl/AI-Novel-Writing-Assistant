# 图数据库能力诊断报告

## 1. 诊断范围

本次诊断回答一个核心问题：**项目是否需要引入专用图数据库来管理小说创作过程中产生的结构化关系数据？**

诊断覆盖：
- Prisma schema 中所有图结构模型（131 个模型中的图边/节点模型）
- `shared/types/` 中的世界图类型定义
- 服务层图数据消费方式（遍历、查询、Prompt 注入）
- 7 个候选图数据库的适配度评估

## 2. 总体结论

| 维度 | 现状 | 评级 |
|------|------|------|
| 图结构覆盖 | 9 个显式图模型 + 3 个 JSON 嵌入式图 | ✅ 充分 |
| 存储引擎 | 关系表模拟图（行存边）+ JSON 数组软引用 | ⚠️ 有瓶颈 |
| 图遍历能力 | **零**——无任何遍历算法实现 | ❌ 缺失 |
| 查询方式 | Prisma findMany + 应用层 JSON 解析 | ⚠️ 不支持多跳 |
| 可视化 | worldVisualization.ts 构建内存邻接表 | ✅ 有基础 |
| 专用图数据库 | 未引入，无任何图 DB 代码 | — |

**核心判断**：当前架构在**单跳查询**（"A 和 B 的关系是什么？"）场景下够用，但在以下场景存在结构性瓶颈：

1. **多跳遍历**（"A 的盟友的敌人是谁？"）——需要应用层多次查询拼接
2. **社区发现**（"哪些角色形成紧密阵营？"）——无算法支持
3. **时间线因果链**（"这个事件的前因后果全链"）——JSON 软引用无 FK 约束、无遍历
4. **信息流追踪**（"这个秘密经过几手传递到达反派？"）——完全不可查询

**建议**：当前阶段**不需要立即引入**图数据库，但应将 KuzuDB 作为中期技术储备纳入规划。短期优先修复 JSON 软引用的 FK 缺失问题。

## 3. 逐项诊断

### 3.1 CharacterRelation — 人物关系图

**结论**：设计最完善的图结构，但缺乏遍历能力。

**证据**：

| 属性 | 值 |
|------|------|
| 位置 | `server/src/prisma/schema.prisma:642` |
| 边模型 | `CharacterRelation`（有向边，带权重属性） |
| 时序模型 | `CharacterRelationStage`（卷/章级快照，`isCurrent` 标记） |
| 快照嵌入 | `RelationState`（`StoryStateSnapshot` 子记录） |
| 唯一约束 | `@@unique([novelId, sourceCharacterId, targetCharacterId])` |
| 索引覆盖 | source/target/updatedAt 三路索引 |
| 消费方 | 16 个服务文件引用 |

**属性丰富度**（每条边）：

```
表面关系(surfaceRelation) → 隐藏张力(hiddenTension) → 冲突源(conflictSource)
信任(trustScore 0-100) · 冲突(conflictScore) · 亲密(intimacyScore) · 依赖(dependencyScore)
秘密不对称(secretAsymmetry) · 动态标签(dynamicLabel) · 下一转折(nextTurnPoint)
证据链(evidence)
```

**问题**：

- **P1 — 无图遍历**：无法执行 "A 的二度关系网" 查询，需 N+1 次 Prisma 调用
- **P2 — 无社区检测**：无法发现角色阵营聚类
- **P3 — Stage 查询低效**：`CharacterRelationStage` 通过 `findMany` + `isCurrent` 过滤，无图索引加速

**建议**：短期保持现状；中期将 CharacterRelation 同步到 KuzuDB，获得 Cypher 遍历能力。

---

### 3.2 StoryTimelineEvent — 时间线事件图

**结论**：图结构存在但以 JSON 软引用实现，**无引用完整性、无遍历能力**。

**证据**：

| 属性 | 值 |
|------|------|
| 位置 | `server/src/prisma/schema.prisma:2836` |
| 边存储 | `prerequisiteIdsJson` / `consequenceIdsJson`（JSON 数组，`@default("[]")`） |
| 参与者 | `participantIdsJson`（JSON 数组） |
| 势力关联 | `factionIdsJson`（JSON 数组） |
| 索引 | 仅 `novelId + chapterIndex / eventOrder / status / eventKey`，**无边索引** |

**问题**：

- **P0 — 无 FK 约束**：`prerequisiteIdsJson` 中的 ID 可能指向已删除事件，无级联更新/删除
- **P1 — 无法遍历**：查询 "事件 A 的所有下游后果" 需加载全部事件后应用层解析
- **P1 — JSON 解析开销**：每次查询需反序列化全部事件的 JSON 字段
- **P2 — 双向一致性无保证**：事件 A 的 `consequenceIdsJson` 包含 B，但 B 的 `prerequisiteIdsJson` 不一定包含 A

**建议**：P0 问题应在不引入图数据库的前提下修复——将 JSON 软引用迁移为独立的 `TimelineEventEdge` Prisma 模型（FK 约束 + 级联删除）。

---

### 3.3 ChapterTimeAnchor — 章节时间锚点图

**结论**：密集的 JSON 软引用网络，是时间线系统中最复杂的图结构。

**证据**：

| 属性 | 值 |
|------|------|
| 位置 | `server/src/prisma/schema.prisma:2867` |
| 边字段 | `startsAfterIdsJson` · `plannedEventIdsJson` · `endedWithIdsJson` · `previousHookIdsJson` · `nextHookIdsJson` · `forbiddenEventIdsJson` |

**6 个 JSON 数组字段**构成一个隐式有向图，但全部无 FK 约束。

**问题**：与 3.2 相同，且复杂度更高（6 种边类型 × 每种可包含多个目标 ID）。

---

### 3.4 World Relations — 世界势力/地点图

**结论**：图类型定义完善，但存储在非结构化 JSON Blob 中，查询能力为零。

**证据**：

| 属性 | 值 |
|------|------|
| 类型定义 | `shared/types/world.ts:166-190` |
| 存储 | `World.structureJson`（未分表的 `String` 字段） |
| 图类型 | `WorldForceRelation` · `WorldLocationControlRelation` · `WorldLocationConnectionRelation` |
| 可视化 | `worldVisualization.ts` 构建 `factionGraph` / `geographyMap` 邻接表 |

**问题**：

- **P1 — 全量反序列化**：查询任何一条势力关系需加载并解析整个 `structureJson`
- **P1 — 无独立索引**：无法按 `sourceForceId` 或 `tension` 筛选
- **P2 — 无 FK 约束**：forceId / locationId 无外键级联
- **P2 — fallback 文本解析脆弱**：`buildFallbackWorldVisualizationPayload` 使用正则+共现推断边，准确率不可控

**建议**：将 `WorldRelations` 从 JSON Blob 提取为独立 Prisma 模型（`WorldForceRelation`、`WorldLocationConnection` 表），保留结构化查询能力。是否上图数据库可视后续遍历需求决定。

---

### 3.5 DirectorArtifactDependency — 导演制品依赖图

**结论**：实现最规范的图边模型。

**证据**：

| 属性 | 值 |
|------|------|
| 位置 | `server/src/prisma/schema.prisma:2161` |
| 唯一约束 | `@@unique([artifactId, dependsOnArtifactId])` |
| 双向关系 | `DirectorArtifact` 上的 `dependencies`(出边) / `dependents`(入边) |
| 版本固定 | `dependsOnVersion` |

**问题**：无——这是目前唯一同时具备 FK 约束、唯一约束和双向关系的图边模型。可作为其他图结构重构的参考模板。

---

### 3.6 StoryPlan — 故事计划层级树

**结论**：标准的自引用树，实现无问题。

**证据**：`parentId` FK + 自引用关系 `StoryPlanHierarchy`，支持 book → arc → chapter 三级树。

**问题**：无。

---

### 3.7 InformationState — 信息不对称图

**结论**：概念正确但查询能力极弱。

**证据**：

| 属性 | 值 |
|------|------|
| 位置 | `server/src/prisma/schema.prisma:2295` |
| 存储 | `StoryStateSnapshot` 子记录 |
| 字段 | `holderType` · `holderRefId` · `fact` · `status` |

**问题**：

- **P1 — 无图语义**：无法查询 "谁知道这个事实" 的反向边（需要从 fact 内容匹配）
- **P2 — 无扩散追踪**：无法回答 "这条信息经过哪些角色传递"
- **P2 — 快照粒度**：信息状态绑定在 Snapshot 上，无法做跨快照的信息流分析

---

### 3.8 ForeshadowState — 伏笔-回收图

**结论**：隐式的 setup → payoff 有向边，存储在快照子表中。

**证据**：`StoryStateSnapshot` 子记录，关联伏笔的设置章节与回收章节。

**问题**：无独立的边模型，依赖快照查询。

---

### 3.9 TimelineHook / TimelineConstraint — 时间线钩子与约束图

**结论**：通过 `relatedEventIdsJson` / `relatedHookIdsJson` / `relatedCharacterIdsJson` 构成交叉引用网络。

**问题**：与 3.2 一致——JSON 软引用，无 FK，无遍历。

## 4. 图数据库候选评估

| 排名 | 数据库 | 许可证 | 部署模式 | 查询语言 | 评分 | 关键优势 | 关键风险 |
|------|--------|--------|----------|----------|------|----------|----------|
| **1** | **KuzuDB** | MIT | 嵌入式（SQLite 式文件） | Cypher | **9/10** | 零 Docker、TS 驱动、内置 GDS | 较年轻项目 |
| 2 | Apache AGE | Apache 2.0 | PG 扩展 | Cypher（不完整） | 7/10 | 零新基础设施 | Cypher 子集、无 GDS |
| 3 | Neo4j CE | AGPL v3 | 独立服务 | Cypher（完整） | 6/10 | 生态最成熟 | AGPL 许可证风险、需独立进程 |
| 4 | FalkorDB | SSPL | Docker | Cypher | 5/10 | Redis 速度 | SSPL 许可证 |
| 5 | Memgraph | BSL | Docker | Cypher | 4/10 | 流式分析 | 流式定位不匹配 |
| 6 | ArangoDB | BSL | Docker | AQL | 4/10 | 多模型 | 过度复杂、BSL |
| 7 | DGraph | BSL/Proprietary | 多容器 | DQL | 3/10 | — | 非 Cypher、多容器 |

**推荐选型理由**：

KuzuDB 与项目当前架构最契合——项目用 SQLite 嵌入式数据库（零运维），KuzuDB 提供相同部署模型的图存储。小说领域的图规模（百级节点、千级边）完全在其能力范围内。

## 5. 跨切面观察

### 5.1 三种图存储模式共存

项目当前使用三种不同的方式存储图数据，增加了认知和维护成本：

| 模式 | 示例 | 优劣 |
|------|------|------|
| **行存边（Prisma 模型）** | CharacterRelation、DirectorArtifactDependency | ✅ FK 约束、索引支持 ❌ 无遍历 |
| **JSON 数组软引用** | StoryTimelineEvent.prerequisiteIdsJson、ChapterTimeAnchor.*IdsJson | ❌ 无 FK、无索引、无遍历 ❌ 双向一致性无保证 |
| **JSON Blob 嵌入** | World.structureJson | ❌ 全量反序列化、无独立查询 ❌ 无 FK |

### 5.2 图数据消费模式单一

所有图数据最终只有两种消费路径：

1. **Prisma findMany** → 应用层过滤 → 返回扁平列表
2. **序列化为 LLM Prompt 文本** → 注入 AI 上下文

**没有一次真正的图查询**（多跳遍历、路径查找、子图提取、社区检测）。

### 5.3 世界可视化是唯一的图构建代码

`worldVisualization.ts`（1082 行）是项目中唯一将关系数据构建为 `{nodes, edges}` 邻接表的代码，但仅用于前端可视化渲染，不参与业务逻辑或 AI 决策。

## 6. 建议处理顺序

| 优先级 | 行动项 | 工作量 | 收益 |
|--------|--------|--------|------|
| **P0** | 将时间线 JSON 软引用迁移为独立 Prisma 边模型（`TimelineEventEdge` 等） | 中 | 消除 FK 缺失、双向不一致 |
| **P1** | 将 `World.structureJson` 中的关系提取为独立 Prisma 表 | 中 | 结构化查询、独立索引 |
| **P2** | 引入 KuzuDB 作为图遍历层，CharacterRelation 首批同步 | 中 | 多跳查询、社区检测 |
| **P3** | 将时间线、信息流图数据同步到 KuzuDB | 小 | 因果链遍历、信息扩散追踪 |

## 7. 结论

项目已建立**丰富的图结构语义**（9 个图模型 + 3 种 JSON 嵌入式图），覆盖了人物关系、势力网络、时间线因果、信息不对称、伏笔回收等核心叙事维度。但全部依赖关系表模拟或 JSON 软引用存储，**缺乏图遍历和算法能力**。

**短期**（当前版本）：不需要图数据库。优先修复 JSON 软引用的 FK 缺失问题，将关键图结构从 JSON Blob 提取为独立表。

**中期**（下一版本）：引入 KuzuDB 作为增量图遍历层，与 Prisma CRUD 并行使用。首批同步 CharacterRelation 数据，获得多跳查询和社区检测能力。
