# REQ-3008 决策日志

## D1: 日志库选择 winston

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-06-30 |
| 决策人 | AI |
| 决策 | 使用 winston + winston-daily-rotate-file 作为后端日志库 |
| 备选方案 | pino（性能更高但生态 transport 少）；自研简单 logger（功能不足） |
| 原因 | winston 是 Node.js 生态最成熟的日志库，内置多 transport、日志轮转、结构化 JSON 输出，社区文档完善。项目不需要极致性能（日志量有限），winston 的功能完备性更重要。 |
| 影响 | 新增 2 个依赖（winston + winston-daily-rotate-file） |

## D2: 日志存储为 JSON Lines 文件

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-06-30 |
| 决策人 | AI |
| 决策 | 日志以 JSON Lines 格式存储在 `server/logs/` 目录，按日期分割文件 |
| 备选方案 | SQLite 数据库；写入 Prisma 管理的 Log 表 |
| 原因 | JSON Lines 文件轻量、无额外依赖、便于 grep 和程序化读取。当前日志量不需要数据库级查询性能。避免给 Prisma schema 增加复杂度。 |
| 影响 | 查询 API 需要自行实现文件读取和过滤逻辑，但复杂度可控 |

## D3: 查询 API 读文件而非数据库

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-06-30 |
| 决策人 | AI |
| 决策 | `GET /api/logs` 直接读取日志文件并解析，不做预索引 |
| 备选方案 | 日志写入时同步存数据库；使用全文搜索引擎 |
| 原因 | 日志查询是低频操作（开发者排查问题时使用），不需要实时高性能。限制最近 30 天 + 分页，单次查询的数据量可控。 |
| 影响 | 大量日志文件时首次查询可能较慢，可通过缓存最近文件索引优化 |

## D4: toast 通知增强在 axios 拦截器层

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-06-30 |
| 决策人 | AI |
| 决策 | 在现有 axios 响应拦截器中增强错误处理，不引入全局错误边界 |
| 备选方案 | React ErrorBoundary + 全局错误状态管理 |
| 原因 | 现有拦截器已经是统一的错误处理入口，增强它即可覆盖所有 API 错误。ErrorBoundary 主要处理渲染错误，与 API 错误场景不匹配。 |
| 影响 | 仅修改 `client/src/api/client.ts` 一个文件 |

## D5: 日志中心页面独立路由

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-06-30 |
| 决策人 | AI |
| 决策 | 日志中心作为独立页面 `/logs`，不嵌套在 Settings 页面下 |
