# REQ-3001 任务中心多选与批量归档

> 状态：⏳ 进行中（完成后改为 ✅ 完成）

## 1. 背景与问题

当前任务中心（`/tasks`）的任务列表采用**单选+详情**的交互模式：用户点击某个任务卡片，右侧详情面板展示该任务的完整信息和操作按钮（重试 / 取消 / 归档 / 打开来源页面）。

当用户积累了大量历史任务（尤其是 `succeeded`、`failed`、`cancelled` 状态），需要清理时，只能逐个执行"点击 → 归档"循环，操作路径为：

1. 点击列表中的某个任务
2. 等待详情加载
3. 在详情面板找到"归档"按钮并点击
4. 等待归档完成
5. 回到列表重复 1-4

每归档一个任务需要 3-4 次交互，批量清理 20 个任务需要 60-80 次交互，效率极低。

---

## 2. 目标与范围

### 2.1 目标

1. 任务列表支持多选模式，用户可通过 Checkbox 批量勾选任务
2. 提供"全选/取消全选"快捷操作，快速选择当前筛选结果中的所有可归档任务
3. 提供"批量归档"按钮，一次性归档所有已选任务
4. 后端提供批量归档 API，单次请求处理多个任务

### 2.2 In Scope

**前端**：
- `TaskCenterListPanel` 中每个任务项增加 Checkbox 多选控件
- 列表顶部增加选择工具栏（已选计数 + 全选/取消全选 + 批量归档按钮）
- 多选状态管理（`Set<string>` 存储已选任务的 `kind:id` 复合键）
- 批量归档时的确认对话框和进度反馈
- 归档完成后自动刷新列表并清空选中状态

**后端**：
- 新增 `POST /api/tasks/batch-archive` 端点
- 接收任务列表 `Array<{kind, id}>`，逐个调用归档逻辑
- 返回每个任务的归档结果（成功 / 失败 + 原因）

**共享**：
- 新增 `BatchArchiveRequest` / `BatchArchiveResponse` 类型定义

### 2.3 Out of Scope

- 批量重试 / 批量取消（本版本仅做批量归档，后续可按需扩展）
- 跨页多选（当前列表无分页，limit=80 以内全量展示，不需要）
- 键盘快捷键多选（Shift+Click 范围选择等高级交互，后续迭代）
- 拖拽归档等手势操作

## 3. 需求详情

### 3.1 多选 Checkbox

WHEN 用户在任务列表中查看任务项
THE SYSTEM SHALL 在每个任务项的左侧显示一个 Checkbox 控件
THE SYSTEM SHALL 允许用户点击 Checkbox 切换该任务的选中/未选状态
THE SYSTEM SHALL Checkbox 选中时高亮该任务项（与单选高亮视觉区分）
THE SYSTEM SHALL Checkbox 点击不影响当前单选的详情查看（即点击 Checkbox 不触发 `onSelectTask`）

### 3.2 选择工具栏

WHEN 任务列表中至少有一个任务存在
THE SYSTEM SHALL 在列表顶部（CardHeader 区域）展示选择工具栏：
- 左侧显示"已选 N / 共 M 个"计数文案
- 右侧显示"全选"和"取消全选"按钮

WHEN 用户点击"全选"
THE SYSTEM SHALL 将当前筛选结果中所有**可归档状态**（`ARCHIVABLE_STATUSES`）的任务标记为选中
THE SYSTEM SHALL 不选中非可归档状态的任务（queued / running / waiting_approval）

WHEN 用户点击"取消全选"
THE SYSTEM SHALL 清空所有选中状态

### 3.3 批量归档按钮

WHEN 有至少一个任务被选中
THE SYSTEM SHALL 在选择工具栏显示"批量归档"按钮
THE SYSTEM SHALL 按钮旁显示已选数量（如"批量归档 (3)"）

WHEN 用户点击"批量归档"
THE SYSTEM SHALL 弹出确认对话框，提示"确定归档选中的 N 个任务？归档后任务将从列表中隐藏。"
THE SYSTEM SHALL 用户确认后，调用批量归档 API
THE SYSTEM SHALL 在归档过程中显示加载状态（按钮 disabled + loading 指示）
THE SYSTEM SHALL 归档完成后：
- 若全部成功：Toast 提示"已成功归档 N 个任务"，清空选中状态，刷新列表
- 若部分失败：Toast 提示"已归档 X 个任务，Y 个归档失败"，仅清空成功项的选中状态
- 若全部失败：Toast 错误提示"批量归档失败"，保留选中状态

### 3.4 批量归档 API

WHEN 前端发起批量归档请求
THE SYSTEM SHALL 接受 `POST /api/tasks/batch-archive` 请求，请求体为：
```typescript
{
  tasks: Array<{ kind: TaskKind; id: string }>
}
```

THE SYSTEM SHALL 逐个执行归档操作（复用现有 `TaskCenterService.archiveTask` 逻辑）
THE SYSTEM SHALL 单个任务归档失败不阻断其他任务
THE SYSTEM SHALL 返回：
```typescript
{
  results: Array<{
    kind: TaskKind;
    id: string;
    status: "success" | "failed";
    error?: string;
  }>;
  summary: { total: number; succeeded: number; failed: number }
}
```

### 3.5 筛选联动

WHEN 用户切换筛选条件（kind / status / keyword / onlyAnomaly / sortMode）
THE SYSTEM SHALL 清空当前选中状态（避免筛选后选中状态指向不可见的任务）

WHEN 列表自动刷新（refetchInterval 触发）
THE SYSTEM SHALL 保留当前选中状态（仅移除已不存在于列表中的选中项）

## 4. 验收标准

- [ ] 任务列表每个任务项左侧显示 Checkbox
- [ ] 点击 Checkbox 可选中/取消选中，不影响右侧详情面板的当前查看任务
- [ ] 列表顶部显示"已选 N / 共 M 个"计数和全选/取消全选按钮
- [ ] "全选"仅选中可归档状态（succeeded / failed / cancelled）的任务
- [ ] 选中至少一个任务后显示"批量归档 (N)"按钮
- [ ] 点击"批量归档"弹出确认对话框
- [ ] 确认后调用批量归档 API，显示加载状态
- [ ] 全部成功时 Toast 提示 + 刷新列表 + 清空选中
- [ ] 部分失败时 Toast 提示成功和失败数量，仅清空成功项
- [ ] 切换筛选条件时自动清空选中状态
- [ ] 后端 `POST /api/tasks/batch-archive` 端点正常工作
- [ ] 单个任务归档失败不阻断其他任务

## 5. 风险与约束

| 风险 | 缓解 |
| ---- | ---- |
| 批量归档大量任务可能导致响应时间过长 | 后端逐个处理并返回结果，前端合理 loading；单次上限 80 个（与列表 limit 对齐） |
| 归档操作不可逆（归档后任务从列表隐藏） | 确认对话框 + 仅允许归档 `ARCHIVABLE_STATUSES` 状态的任务 |
| 列表自动刷新时选中状态可能与实际列表不一致 | refetch 时过滤掉不存在于最新列表中的选中项 |
| 批量操作期间任务状态可能变化（如正在运行的任务突然失败变为可归档） | 仅允许选中当前已处于可归档状态的任务；批量执行时若任务状态变化导致归档失败，在结果中标记 |

---

## 6. 关联与边界

- 与单任务归档（`archiveTask`）的关系：批量归档是单任务归档的批量包装，复用相同的归档逻辑
- 与任务筛选面板（`TaskCenterFilterPanel`）的关系：全选操作基于当前筛选结果
- 与任务详情面板的关系：多选操作不影响单选查看详情的交互
- 与共享类型（`shared/types/task.ts`）的关系：新增批量归档相关类型
