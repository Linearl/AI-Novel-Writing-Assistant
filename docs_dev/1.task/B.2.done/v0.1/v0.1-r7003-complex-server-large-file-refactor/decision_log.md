# REQ-7003 决策日志

## D1: 按功能域分批而非一次性全量拆分

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-06-28 |
| 决策人 | AI |
| 决策 | 将 18 个超大文件按功能域分为 5 个批次，每批独立可验证 |
| 备选方案 | A: 一次性全量拆分所有文件；B: 按文件大小排序逐个拆分 |
| 原因 | 18 个文件涉及 9 个功能域，一次性拆分风险过高，出问题难以定位。按功能域分批可以利用域内文件的关联性，同批拆分时共享分析结果。每批独立 typecheck + test 验证，控制爆炸半径。 |
| 影响 | 需要维护批次进度跟踪，但每批可独立交付 |

## D2: Facade 重导出保持公共 API 不变

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-06-28 |
| 决策人 | AI |
| 决策 | 拆分后的原始文件通过 `export { ... } from './internal/...'` 重导出，保持所有现有 import 路径 |
| 备选方案 | A: 修改所有引用方的 import 路径指向新文件 |
| 原因 | 项目有大量跨模块 import，修改路径牵连面广且容易遗漏。Facade 模式让拆分对调用方完全透明，typecheck 可以立即发现遗漏。 |
| 影响 | 原文件变为轻量 facade，实际逻辑在子目录中 |

## D3: 批次优先级排序 — 最大文件优先

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-06-28 |
| 决策人 | AI |
| 决策 | Batch 1（world，最大 1311 行）优先，其次 prompting、director |
| 备选方案 | A: 按变更频率排序（高频文件优先）；B: 按依赖关系排序（被依赖最多的优先） |
| 原因 | 最大文件收益最高（1311 → ≤600 行，改善率 >50%）。world 域文件间依赖紧密，一起拆分效率高。按大小排序简单直观，不易产生优先级争议。 |
| 影响 | 前期收益高，后期批次（803 行文件）边际收益递减 |

## D4: 拆分粒度 — 按职责而非按行数

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-06-28 |
| 决策人 | AI |
| 决策 | 每个子模块有明确的单一职责，不按行数机械切割 |
| 备选方案 | A: 每 400 行切一刀 |
| 原因 | 按职责拆分的模块内聚度高、可维护性好。按行数切割会导致同一函数/逻辑被拆到两个文件，引入不必要的跨文件引用。600 行是上限约束，不是目标行数。 |
| 影响 | 需要先分析每个文件的职责区块，再确定拆分边界 |
