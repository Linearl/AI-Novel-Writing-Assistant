# REQ-7034 决策留痕

## 决策记录

### D-01：应用"删除测试"判定方法去留

| 项目 | 内容 |
| ---- | ---- |
| 决策日期 | 2026-07-10 |
| 决策者 | AI 建议 |
| 决策内容 | 对每个方法应用删除测试：删除该方法后，调用方是否能以相同或更少的代码量直接调用子服务 |
| 决策理由 | 删除测试是 Karpathy 编码准则中的客观判定标准，避免主观判断。纯委托（1 行转发）删除后净复杂度不变，无杠杆；协调方法（多服务调用 + 事务）删除后每个调用方需重复协调逻辑，有杠杆 |
| 备选方案 | 凭直觉判断 — 可能保留不必要的委托或误删协调逻辑 |

### D-02：不创建中间层替代门面

| 项目 | 内容 |
| ---- | ---- |
| 决策日期 | 2026-07-10 |
| 决策者 | AI 建议 |
| 决策内容 | 移除纯委托后，调用方直接注入子服务，不创建新的 "Facade Lite" / "Service Locator" / "NovelServiceProvider" 等中间层 |
| 决策理由 | 创建新的中间层会重复门面问题——又一个纯委托层。问题不是"哪个类做委托"，而是"委托本身无价值"。直接注入子服务是最简方案 |
| 备选方案 | 创建 ServiceLocator 替代门面 — 换了个名字，本质还是委托层 |

### D-03：路由文件可注入多个子服务

| 项目 | 内容 |
| ---- | ---- |
| 决策日期 | 2026-07-10 |
| 决策者 | AI 建议 |
| 决策内容 | 路由注册文件（novelRouteRegistration.ts）可注入多个具体子服务，不要求只注入一个门面 |
| 决策理由 | 路由文件是 HTTP 层入口，天然需要协调多个子服务（CRUD + Chapter + World 等）。注入多个子服务是合理的，不违反单一职责 |
| 备选方案 | 限制每个路由只注入一个服务 — 不现实，HTTP handler 需要多个后端服务 |

### D-04：协调方法的判定门槛

| 项目 | 内容 |
| ---- | ---- |
| 决策日期 | 2026-07-10 |
| 决策者 | AI 建议 |
| 决策内容 | 协调方法的判定标准：方法体包含 2+ 子服务调用 OR 包含事务逻辑 OR 包含数据聚合/转换（非 trivial 映射） |
| 决策理由 | 单一标准的边界情况太多。例如 `getNovel(id)` 是纯委托，`getNovelWithChapters(id)` 调了 2 个子服务并聚合数据，是协调方法 |
| 备选方案 | 仅用"方法行数 >1"判断 — 不准确，有的 2 行方法只是加了 log 或类型转换 |
