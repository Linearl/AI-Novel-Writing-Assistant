# REQ-7021 决策日志

## D1: 迁移方向为 server → shared，禁止反向

**日期**：2026-07-10
**决策**：只允许 server schema 迁移到 shared，禁止 shared 依赖 server
**原因**：shared 是 server 和 client 的共同依赖，架构上 shared → server 是自然方向。反方向会造成循环依赖和构建顺序混乱。
**影响**：迁移后 shared/types/ 文件增多/变大，但这是可接受的代价。

## D2: 纯 server 内部 DTO/配置 schema 不迁移

**日期**：2026-07-10
**决策**：仅 server 内部使用的请求校验、中间件配置、内部 DTO 等 schema 保留在 server
**原因**：这些 schema 与 server 的 Express/Prisma 实现细节耦合，迁移到 shared 无复用价值，反而增加 shared 的维护负担。
**影响**：审计分类标准需明确区分"纯内部"和"可迁移"。

## D3: 优先 .extend() 而非复制粘贴

**日期**：2026-07-10
**决策**：server 侧需要 shared schema 的变体时，使用 `.extend()` / `.pick()` / `.omit()` / `.merge()` 等 zod 内置方法
**原因**：保持单一数据源，避免分叉（fork）后的独立演化。
**影响**：所有变体逻辑显式表达，变更追溯清晰。

## D4: 50% 消除率为目标而非硬性指标

**日期**：2026-07-10
**决策**：目标将 server 侧独立 zod 调用从 407 减少至 200 以下（50% 减少率），但如果实际可合并的不足 50%，接受实际结果
**原因**：审计前无法精确预知可合并比例，50% 是合理估计。强行追求数字可能导致低价值迁移。
**影响**：审计阶段完成后评估实际可达目标，必要时调整预期。
