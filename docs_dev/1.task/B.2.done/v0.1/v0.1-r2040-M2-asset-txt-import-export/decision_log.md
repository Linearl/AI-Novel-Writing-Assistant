# REQ-2040 决策日志

## 决策记录

### D1: 文本格式选型 — TXT 而非 JSON/XML

**日期**: 2026-07-03

**决策**: 各模块资产导出使用纯 TXT 格式，不使用 JSON 或 XML。

**备选方案**:
1. 纯 TXT 格式（自定义分隔符）
2. JSON 格式（结构化但人类可读性差）
3. XML 格式（严格但冗长）
4. TXT + JSON 双格式导出

**理由**:
- TXT 可读性最好，用户可直接用记事本查看和编辑
- 跨工具兼容性最佳，任何文本编辑器都能打开
- 学习成本最低，格式直观
- 不需要 JSON/XML 解析器，降低用户使用门槛
- 防 vendor lock-in 的核心目的是给用户最大自由度，TXT 最符合

**风险**: TXT 格式不含类型信息，导入时需要约定规则
**缓解**: 严格定义字段名合法列表，导入时校验字段合法性

### D2: 文件编码 — UTF-8 无 BOM

**日期**: 2026-07-03

**决策**: 所有 TXT 文件统一使用 UTF-8 编码，不含 BOM。

**备选方案**:
1. UTF-8 无 BOM
2. UTF-8 with BOM
3. GBK/GB2312（兼容中文 Windows 老工具）

**理由**:
- UTF-8 是 Web 标准，Node.js 原生支持
- 无 BOM 避免部分编辑器（如旧版记事本）出现多余字符
- GBK 虽兼容部分中文 Windows 工具，但增加编码检测复杂度，不利于跨平台

**风险**: 极少数旧版 Windows 记事本默认用 ANSI 打开 UTF-8 文件可能乱码
**缓解**: 新版 Windows 10+ 记事本已默认 UTF-8，且导出文件名带 `.txt` 后缀不依赖 BOM 识别

### D3: 导入模式设计 — 覆盖/合并/追加

**日期**: 2026-07-03

**决策**: 不同模块使用不同的导入模式组合。

**模式分配**:

| 模块 | 可用模式 | 说明 |
|------|---------|------|
| 世界设定 | 覆盖 / 合并 | 覆盖=全量替换；合并=新字段写入，已有字段更新 |
| 大纲 | 覆盖 / 追加 | 覆盖=全量替换；追加=在现有大纲末尾添加 |
| 关系网 | 覆盖 / 合并 | 覆盖=全量替换；合并=新角色/关系写入，已有保留 |

**理由**:
- 大纲有明确顺序，"合并"语义模糊，用"追加"更直观
- 设定和关系网的"合并"允许用户只更新部分数据，更灵活
- 默认模式统一为"覆盖"，行为确定性高

### D4: 导入传输方式 — text/plain body

**日期**: 2026-07-03

**决策**: 导入端点使用 `text/plain` Content-Type 接收原始文本内容。

**备选方案**:
1. `text/plain` body（直接传文本）
2. `multipart/form-data`（文件上传）
3. Base64 编码嵌入 JSON

**理由**:
- TXT 文件通常很小（几十KB以内），直接传文本足够
- 减少前端文件处理步骤，减少 FormData 构造
- 与导出的 `text/plain` 对称，概念简单
- 保留 multipart 作为后续扩展点（如大文件场景）

**风险**: 部分代理/网关可能过滤 `text/plain` POST body
**缓解**: 实际项目内部调用，不受外部代理影响
