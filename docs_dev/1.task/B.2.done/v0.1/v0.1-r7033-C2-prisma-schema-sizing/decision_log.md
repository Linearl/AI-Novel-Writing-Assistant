### D-01：审计先行，实施后行

| 项目 | 内容 |
| ---- | ---- |
| 决策日期 | 2026-07-10 |
| 决策者 | AI 建议 |
| 决策内容 | 将任务分为审计评估（T1~T3）+ 实施（T4~T6）两个阶段，审计不通过不进入实施 |
| 决策理由 | 宽表审计和多 schema 评估的结果直接影响实施方案。先审计可避免在不了解全貌的情况下做出不可逆的 schema 变更 |

### D-02：JSON 收敛优先于多 schema 拆分

| 项目 | 内容 |
| ---- | ---- |
| 决策日期 | 2026-07-10 |
| 决策者 | AI 建议 |
| 决策内容 | 推荐混合方案（选项 C）：先执行低风险的宽表收敛，再评估高影响的多 schema 拆分 |
| 决策理由 | 宽表收敛仅改动个别列的类型（独立列 → JSON 属性），不改变 Prisma 构建流程。多 schema 拆分涉及 Prisma Client 生成路径变化，需充分验证 |
| 备选方案 | 直接做多 schema 拆分 — 风险高，可能阻塞其他并行任务 |

### D-03：迁移历史精简需保守操作

| 项目 | 内容 |
| ---- | ---- |
| 决策日期 | 2026-07-10 |
| 决策者 | AI 建议 |
| 决策内容 | 迁移 squash 仅当迁移数 >60 或可合并为基线迁移且不丢失回滚能力时才执行 |
| 决策理由 | 迁移历史是数据库变更的唯一可追溯记录。过早 squash 会丧失"回到任意时间点"的能力 |
| 备选方案 | 无条件 squash 所有迁移 — 丧失回滚能力，风险不可接受 |
