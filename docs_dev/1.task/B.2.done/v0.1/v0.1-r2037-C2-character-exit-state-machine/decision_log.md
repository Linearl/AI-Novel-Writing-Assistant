# REQ-2037 决策日志

## D1: 退场 ≠ 死亡，需要分开管理

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-07-03 |
| 决策人 | 用户 + AI |
| 决策 | 使用四态状态机（active / exited / dead / frozen），exited 和 dead 作为独立状态 |
| 原因 | 死亡和退场在叙事功能上不同：死亡角色可能在回忆、倒叙、幻觉中出现，而退场角色仅可能被提及名字。合并为一个"不在场"状态会丢失这一语义差异，影响后续 prompt 构建。 |
| 影响 | 状态枚举增加一个值；状态转换规则需要两个入口路径；但整体复杂度可控。 |
| 备选方案 | 三态（active / inactive / frozen）— 过于粗粒度，无法区分退场和死亡 |

## D2: frozen 是终态，由 auto-director 自动判定

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-07-03 |
| 决策人 | AI |
| 决策 | frozen 状态仅由 auto-director 根据提及频率自动判定，不暴露给用户手动操作 |
| 原因 | frozen 的语义是"auto-director 已确认该角色不再需要"，这是系统层面的判断，而非用户意图。用户可能误操作冻结仍有用的角色，而系统基于全局上下文的判断更准确。 |
| 影响 | 用户无法手动冻结角色；如果 auto-director 判断有误，用户只能通过等待（角色被再次提及时自动解冻）或联系 AI 辅助。首版接受此限制。 |
| 备选方案 | 同时暴露手动冻结 — 增加用户操作复杂度，且可能导致误操作。后续迭代可考虑。 |

## D3: 自动推断使用 LLM 而非规则匹配

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-07-03 |
| 决策人 | AI |
| 决策 | 角色退场/死亡推断通过 LLM 结构化输出实现，不使用关键词匹配或手写规则 |
| 原因 | 退场判断高度依赖剧情语义理解。"张三完成使命独自离开"是退场，"张三暂时离开去搬救兵"不是退场——两者在文本层面可能很相似。项目遵循 AI-First 原则，核心决策路径以 AI 结构化理解为主。 |
| 影响 | 每章确认后增加一次 LLM 调用（退场推断），增加少量延迟和 token 消耗。通过结构化输出和精简 prompt 控制成本。 |
| 备选方案 | 规则匹配（关键词 + 章节停留时数）— 实现简单但覆盖率低，容易误判。可作为 LLM 失败时的 fallback，首版不做。 |

## D4: 冻结阈值可配置（默认 5 章）

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-07-03 |
| 决策人 | 用户 + AI |
| 决策 | 自动冻结阈值（连续未提及章数）默认 5，可通过项目设置配置 |
| 原因 | 不同小说节奏差异大：快节奏小说 3 章不出场可能就是退场，而慢节奏长篇中角色可能因剧情线交替暂时不出场。固定阈值无法适应所有场景。 |
| 影响 | 需要在项目设置中增加一个配置项。首版可通过数据库直接配置，后续迭代增加 UI 配置入口。 |
| 备选方案 | 固定阈值 5 — 简单但不灵活，可能导致误冻结或延迟冻结 |

## D5: frozen 角色数据不删除

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-07-03 |
| 决策人 | AI |
| 决策 | frozen 角色保留在数据库中，仅从生成上下文排除 |
| 原因 | 数据完整性是底线。用户可能需要回顾角色设定、在角色管理页面查看、或将角色重新激活（后续迭代）。删除数据是不可逆操作，与项目数据保护原则冲突。 |
| 影响 | 数据库中会积累越来越多的 frozen 角色记录，但 Character 表本身不大（百级），不影响性能。 |
| 备选方案 | frozen 后归档到独立表 — 过度设计，增加查询复杂度 |
