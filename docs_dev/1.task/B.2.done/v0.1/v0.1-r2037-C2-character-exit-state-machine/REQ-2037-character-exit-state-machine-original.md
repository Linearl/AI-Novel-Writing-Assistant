## 1. 背景与问题

长篇小说（尤其 50 章以上）中后期常见角色膨胀问题：

- 大量边缘角色在完成核心使命后仍然留在生成上下文中
- auto-director 为每章构建角色上下文时，所有 `active` 角色均被纳入，导致 token 浪费和角色信息稀释
- 已死亡角色与已完成使命角色混为一谈，无法区分"可能在回忆中出现"和"使命完成不再出场"
- 缺乏机制让 auto-director 自动识别并冻结不再需要的角色，依赖用户手动管理

竞品"游蜂写作"已实现角色退场管理，包含退场标记和退场后上下文过滤，是本需求的直接触发点。

---

## 2. 目标与范围

### 2.1 目标

1. 为 Character 引入四态状态机：`active` → `exited` / `dead` → `frozen`
2. auto-director 执行章节后，通过 LLM 自动推断哪些角色在该章退场或死亡
3. `frozen` 角色不参与正文生成上下文，但仍保留在数据库中可查看
4. 客户端角色管理面板展示退场状态，支持手动标记退场/死亡

### 2.2 In Scope

**共享层（shared）**：
- `Character` 模型增加 `exitStatus` 字段，枚举类型：`active` | `exited` | `dead` | `frozen`
- 新增 `CharacterExitStatus` TypeScript 枚举

**后端（server）**：
- Prisma schema 变更 + 数据库迁移（SQLite + PostgreSQL）
- 角色上下文构建时过滤 `frozen` 角色（不纳入正文生成 prompt）
- auto-director 章节执行后新增"退场推断"步骤（新 prompt + LLM 调用）
- 新增角色退场状态变更 API（手动标记 + 自动推断结果写入）

**客户端（client）**：
- 角色管理面板展示退场状态标签
- 支持按 `exitStatus` 筛选角色
- 支持手动将角色标记为 `exited` 或 `dead`

### 2.3 Out of Scope

- 退场角色的自动召回机制（如"伏笔回收"场景 — 后续迭代）
- 角色退场的时间线回溯（退场后发现需要，暂不支持回退状态）
- `frozen` 状态的用户手动操作（`frozen` 仅由 auto-director 自动判定）
- 退场角色在回忆/倒叙中的自动激活（dead 角色在回忆场景中出现由 prompt 处理，不做代码级控制）

## 3. 需求详情

### 3.1 状态定义

| 状态 | 含义 | 生成上下文 | 数据库保留 | 可手动标记 | 自动推断目标 |
| ---- | ---- | ---------- | ---------- | ---------- | ------------ |
| `active` | 正常活跃角色 | 参与 | 是 | — (默认) | — |
| `exited` | 完成使命不再出场，可能被提及 | 不参与 | 是 | 是 | 是 |
| `dead` | 明确死亡，可能在回忆/倒叙中出现 | 不参与 | 是 | 是 | 是 |
| `frozen` | auto-director 确认不再需要 | 不参与 | 是 | 否 | 是（由 exited/dead 转入） |

### 3.2 状态转换规则

```
active ──→ exited   （手动或自动推断：角色完成使命）
active ──→ dead     （手动或自动推断：角色死亡）
exited ──→ frozen   （auto-director 确认后自动冻结）
dead   ──→ frozen   （auto-director 确认后自动冻结）
```

不可逆转换：
- `exited` / `dead` 不可回退到 `active`（使命完成或死亡是确定性事件）
- `frozen` 为终态，不可回退（auto-director 已确认该角色不再需要）

自动冻结条件：
- `exited` 角色在连续 N 章（默认 5 章）未被任何文本提及时，自动转为 `frozen`
- `dead` 角色在连续 N 章未被回忆/倒叙提及时，自动转为 `frozen`
- N 值可通过项目设置配置

### 3.3 自动推断机制

**触发时机**：auto-director 执行完一章正文生成并确认后。

**推断流程**：
1. 收集本章正文内容
2. 将正文内容 + 当前所有 `active` 角色列表提交给 LLM
3. LLM 判断：本章中是否有角色退场（完成使命不再出场）或死亡（明确死亡情节）
4. 对于标记为退场/死亡的角色，LLM 同时提供推断置信度和证据（引用章内文本）
5. 置信度达到阈值（默认 0.7）时自动标记；低于阈值时写入待确认列表供用户审核

**LLM 输出格式**（结构化 JSON）：

```json
{
  "exitEvents": [
    {
      "characterId": "clx...",
      "characterName": "张三",
      "exitType": "exited",
      "confidence": 0.85,
      "evidence": "张三完成了护送任务后独自离开，表示'我的使命到此为止'",
      "chapterContext": "第 45 章末尾"
    }
  ]
}
```

### 3.4 上下文过滤

auto-director 构建章节生成上下文时：
1. 查询小说所有角色
2. 过滤掉 `exitStatus = frozen` 的角色
3. `active` 角色正常纳入
4. `exited` / `dead` 角色：默认不纳入，但如果当前章节大纲涉及回忆/倒叙且与该角色相关，则纳入（此逻辑首版不做，留给 prompt 层处理）

### 3.5 客户端交互

**角色列表展示**：
- 每个角色卡片增加状态标签：`活跃`（绿色）/ `已退场`（灰色）/ `已死亡`（红色）/ `已冻结`（蓝色）
- 角色列表顶部增加筛选器：全部 / 活跃 / 已退场 / 已死亡 / 已冻结

**手动标记**：
- 角色详情页增加"标记退场"和"标记死亡"按钮
- 点击后弹出确认对话框，说明影响（不再参与后续生成上下文）
- 标记后角色状态立即变更

## 4. 验收标准

### AC-1: 数据模型

- [ ] Prisma schema 中 Character 模型包含 `exitStatus` 字段，默认值 `active`
- [ ] TypeScript 类型 `CharacterExitStatus` 在 shared 包中导出
- [ ] 数据库迁移成功执行（SQLite 和 PostgreSQL）

### AC-2: 上下文过滤

- [ ] auto-director 构建角色上下文时，`frozen` 角色不被纳入
- [ ] `active` 角色正常纳入
- [ ] 过滤逻辑不影响现有章节生成质量

### AC-3: 自动推断

- [ ] 章节确认后自动触发退场推断 LLM 调用
- [ ] LLM 返回结构化退出事件
- [ ] 高置信度（>= 0.7）事件自动写入数据库
- [ ] 低置信度事件写入待确认列表

### AC-4: 冻结机制

- [ ] `exited` / `dead` 角色在连续 N 章未被提及时自动转为 `frozen`
- [ ] 冻结操作可配置 N 值

### AC-5: 客户端

- [ ] 角色列表展示退场状态标签
- [ ] 支持按 `exitStatus` 筛选
- [ ] 手动标记退场/死亡流程完整（按钮 → 确认 → 状态变更）

### AC-6: 数据安全

- [ ] `frozen` 角色数据保留在数据库中，不被删除
- [ ] 状态转换不可逆（active→exited/dead→frozen）
- [ ] 退场推断结果可追溯（包含证据引用）
