# REQ-2009 决策日志

## D1: 复用 isRunnableProviderConfig 而非新增过滤

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-06-27 |
| 决策人 | AI |
| 决策 | 复用 `client/src/lib/llmSelection.ts` 中已有的 `isRunnableProviderConfig()` 函数 |
| 原因 | 该函数已被 `LLMSelector` 组件使用且经过验证，判定逻辑完整（`isConfigured && isActive && models.length > 0`）。新增过滤函数会导致行为不一致。 |
| 影响 | 零新增代码，仅在 `ModelRoutesPage` 中引入已有函数 |

## D2: 客户端过滤而非服务端

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-06-27 |
| 决策人 | AI |
| 决策 | 在客户端 `ModelRoutesPage` 中过滤，不新增服务端 API 端点 |
| 原因 | `/settings/api-keys` 已返回 `isConfigured`/`isActive` 字段，客户端过滤零成本。新增服务端端点需要额外的路由、测试和维护。 |
| 影响 | 网络传输仍包含未配置厂商的数据（可忽略，数据量极小） |

## D3: 保护已有路由引用

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-06-27 |
| 决策人 | AI |
| 决策 | 编辑已有路由时，当前引用的厂商始终可见（即使不可用），显示为灰色 + "未配置"标签 |
| 原因 | 如果直接隐藏不可用厂商，用户编辑已有路由时会看不到当前厂商，导致困惑。保留可见但禁用，让用户知道需要先配置。 |
| 影响 | 下拉框可能包含少量灰色选项，不影响可用厂商的选择体验 |
