# REQ-2041 决策日志

## D1: 伏笔状态分为 4 种

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-07-03 |
| 决策人 | AI |
| 决策 | 伏笔状态收敛为 4 种：planted / active / resolved / expired |
| 原因 | 现有 `payoffLedgerStatusSchema` 定义了 6 种状态（setup/hinted/pending_payoff/paid_off/failed/overdue），其中 setup 和 hinted 语义接近（都是已埋设未回收），paid_off 和 failed 都可视为已处理。对用户而言，4 种状态更直观：已埋设、进行中、已回收、过期。通过 `normalizedStatus` 字段在新旧枚举之间建立映射，保持向后兼容。 |
| 影响 | shared 层新增 `normalizedStatus` 字段，server 层伏笔检测和过期检测使用新状态枚举，UI 面板使用 4 种状态展示。 |
| 备选方案 | 直接修改现有枚举值（破坏性变更）— 会导致大量现有代码需要同步修改，风险较高 |

## D2: 过期阈值可配置（默认 20 章）

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-07-03 |
| 决策人 | AI |
| 决策 | 伏笔过期告警的章节数阈值可配置，默认值 20 章 |
| 原因 | 不同类型的小说节奏差异很大。快节奏网文可能 5-10 章就需要回收伏笔，而慢节奏长篇可能 50 章后才到回收时机。硬编码一个固定阈值无法适应所有场景，可配置是必要灵活性。默认 20 章是基于主流网文的平均节奏选取的平衡值。 |
| 影响 | NovelSettings 增加 `payoffExpiryThreshold` 字段，server 层过期检测读取该配置，client 层增加配置入口。 |
| 备选方案 | 固定阈值（如硬编码 20）— 简单但不灵活，无法适应不同小说类型 |

## D3: AI 检测 + 用户确认模式

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-07-03 |
| 决策人 | AI |
| 决策 | 伏笔埋设检测由 LLM 自动完成，低置信度结果标记为 suggested，需用户确认 |
| 原因 | AI 检测伏笔的准确性有限，尤其是隐含式伏笔（如角色行为暗示而非直接描述）。全自动写入可能产生大量误判，导致用户对系统失去信任。"AI 检测 + 用户确认"模式在效率和准确性之间取平衡。 |
| 影响 | 伏笔条目可能有 confirmed/suggested 两种可信度状态，UI 面板区分展示。 |
| 备选方案 | 全自动写入（不需确认）— 效率更高但误判率难控；纯手动录入 — 准确但用户负担重 |

## D4: 回收提醒上限 5 条

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-07-03 |
| 决策人 | AI |
| 决策 | auto-director 生成新章节时，最多注入 5 条最紧迫的过期伏笔作为回收提醒 |
| 原因 | 注入过多伏笔信息会显著增加 token 消耗，且对 AI 生成质量产生干扰（过多约束导致生成内容不自然）。5 条是基于实践的经验值：足够覆盖最紧急的伏笔，同时不过度影响生成质量。 |
| 影响 | 按逾期章节数排序取前 5 条，注入格式为结构化列表。 |
| 备选方案 | 全量注入 — 简单但 token 消耗不可控；不注入 — 无法实现自动提醒 |
