## D2: Context Block 从 reference 提升到 preferred

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-06-30 |
| 决策人 | AI |
| 决策 | 将 `existing_beat_sheet` 和 `existing_chapter_details` 的 group 从 `reference` 提升到 `preferred` |
| 原因 | `reference` group 不在 requiredGroups 或 preferredGroups 中，token 预算紧张时会被优先裁剪，导致结构保持约束失去实际参考依据。提升到 `preferred` 确保这些信息在大多数情况下保留在 prompt 中。 |
| 影响 | 可能略微增加 prompt token 消耗，但 preferred group 在 `required` 之后才被裁剪 |

## D3: 重试次数限制为 1 次

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-06-30 |
| 决策人 | AI |
| 决策 | 结构保持验证失败时最多重试 1 次，重试仍失败则降级使用首次输出 |
| 原因 | 重试增加延迟和 token 消耗。1 次重试已足够给 LLM 第二次机会遵守约束。多次重试的边际收益递减。降级策略确保流程不被阻断。 |
| 影响 | 最坏情况下多 1 次 LLM 调用的延迟和成本 |

## D4: 验证范围仅限已有章节关联的 beat

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-06-30 |
| 决策人 | AI |
| 决策 | 验证函数只检查"有已关联章节的 beat"的结构一致性，不检查无章节的 beat |
| 原因 | 无章节关联的 beat 没有需要保护的细化数据，LLM 可以自由调整。只保护有实际数据的 beat，减少不必要的约束。 |
| 影响 | 验证更精准，不会因无关 beat 的结构调整触发误报 |
