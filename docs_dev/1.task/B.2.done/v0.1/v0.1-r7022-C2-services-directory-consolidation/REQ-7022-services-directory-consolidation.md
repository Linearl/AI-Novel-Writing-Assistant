# REQ-7022 Services 目录收敛与大文件拆分

## 基本信息

| 字段 | 内容 |

| 需求编号 | REQ-7022 |
| 优先级 | P1 |
| 版本 | 0.1 |

---

## 1. 背景与问题

services/ 目录规模失控：
- 516 个文件、62 个子目录
- 17 个文件 >650 行，接近强制重构线（>700 行必须重构）
- 最大文件 719 行，其他 8 个 >680 行
- services/novel/director/ 下 11 个子目录，其中 runtime/ 含 36 个文件
- 部分子目录文件数少且职能重叠（debug/ 2 文件 + commands/ 3 文件）

**不改会怎样**：超大文件持续增长，目录结构随新需求继续扁平扩张，新开发者定位代码越来越困难。

---

## 2. 目标与范围

### 2.1 目标

1. 对 8 个 >680 行的文件制定并执行拆分计划
2. 评估 director 子目录收敛：debug/ + commands/ -> operations/
3. 审计 services/novel/ 下 34 个子目录的模块内聚性，标记可合并项
4. 对 director state 文件去重（director/StateXxx.ts vs director/state/Xxx.ts）

### 2.2 In Scope

- `server/src/services/` 目录树下的文件拆分与子目录重组
- 8 个大文件的拆分（>680 行）
- director 子目录评估与合并
- state 文件去重

### 2.3 Out of Scope

- 不涉及 services/ 之外的目录重构
- 不改 service 层的功能逻辑
- 不重构非超大文件（<650 行的维持现状）

---

## 3. 需求详情

### 3.1 大文件拆分（>680 行，共 8 个）

| # | 文件路径 | 行数 | 拆分方向 |
|---|---------|------|---------|
| 1 | 最大文件 1 | ~719 | 按职责拆分为 2-3 个文件 |
| 2 | 大文件 2 | ~700 | 提取纯函数到 utils/ |
| ... | ... | ... | ... |

（注：具体文件名由 Phase 1 审计阶段确认）

**拆分原则**：
- 按职责边界拆，不按行数机械切
- 提取的可复用逻辑落到 `utils/` 或 `shared/`
- 拆分后通过 facade `index.ts` 维持原导入路径兼容

### 3.2 Director 子目录收敛

**目标**：评估将 `director/debug/`（2 文件）和 `director/commands/`（3 文件）合并到 `director/operations/`。

**评估标准**：
- 文件数 <5 的子目录 → 检查与兄弟目录的语义重叠
- 职能相近（debug 和 commands 都提供 director 操作入口）→ 可合并
- 合并后 operations/ 文件数 <15 可接受

### 3.3 Novel 子目录内聚性审计

审计 `services/novel/` 下 34 个子目录：
- 每个子目录的文件数、职责
- 与兄弟目录的职能重叠检测
- 标记可合并项（文件数 <3 且职能重叠的优先）

### 3.4 Director State 文件去重

检查 `director/StateXxx.ts` 与 `director/state/Xxx.ts` 之间的重复：
- 若两者定义同一 state，合并到 `director/state/` 下
- 若职责不同，保留但重命名以消除歧义

## 4. 验收标准

- [ ] 8 个 >680 行的文件均已拆分至 <600 行
- [ ] director 子目录收敛评估完成（合并或明确决策不合并）
- [ ] state 文件去重完成（合并或明确保留理由）
- [ ] `pnpm typecheck` 通过
- [ ] `pnpm test` 通过
- [ ] 无新增 import 路径断裂（原导入路径通过 facade 兼容）

---

## 5. 风险与缓解

| 风险 | 缓解 |
|------|------|
| 拆分后循环依赖 | 拆分前分析 import 图，按依赖方向调整文件边界 |
| facade index.ts 维护负担 | 仅对外部消费多的模块提供 facade |
| 合并子目录后路径大量变更 | 先合并→再全局搜索替换导入路径 |
