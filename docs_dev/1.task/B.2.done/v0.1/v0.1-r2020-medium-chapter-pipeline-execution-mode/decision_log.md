# REQ-2020: 自动导演流水线执行模式 - 决策日志

## 决策记录

### D.1 流水线模式作为可选配置

**日期**：2026-06-27

**背景**：
用户希望引入流水线执行模式，但担心影响现有功能。

**决策**：
将流水线模式作为可选配置，默认不启用。

**理由**：
1. 降低风险：现有功能不受影响
2. 用户选择：用户可以根据需要启用
3. 渐进式 rollout：可以逐步推广

**影响**：
- 需要在 UI 中添加配置选项
- 需要在后端支持两种模式

---

### D.2 细化与写作交错执行

**日期**：2026-06-27

**背景**：
当前批量模式下，所有章节细化完成后才开始写作，存在等待浪费。

**决策**：
实现细化与写作的交错执行：章节 A 细化完成后立即开始写作，同时章节 B 继续细化。

**理由**：
1. 提高效率：减少等待时间
2. 更早看到内容：用户可以更早看到已完成的章节
3. 资源利用率：CPU 和 API 调用更均衡

**影响**：
- 需要修改结构化大纲阶段的逻辑
- 需要修改流水线执行服务的逻辑
- 需要处理并行执行的同步问题

---

### D.3 人工审核阻塞机制

**日期**：2026-06-27

**背景**：
流水线模式下，如果某一章的审校未通过且需要人工审核，后续章节的写作应该怎么办？

**决策**：
阻塞后续章节的写作，但细化流水线继续执行。

**理由**：
1. 保证质量：人工审核是质量保证的重要环节
2. 避免问题扩散：如果继续写作，可能会基于有问题的内容
3. 细化继续：细化不依赖写作结果，可以继续执行

**影响**：
- 需要实现阻塞机制
- 需要实现恢复机制
- 需要在 UI 中显示阻塞状态

---

### D.4 暂停后可修改配置

**日期**：2026-06-27

**背景**：
用户希望在自动导演暂停时可以修改流水线模式配置。

**决策**：
允许在暂停状态下修改流水线模式配置。

**理由**：
1. 灵活性：用户可以根据实际情况调整
2. 用户体验：不需要取消重新开始

**影响**：
- 需要在任务中心添加配置修改入口
- 需要处理配置变更的逻辑

---

### D.5 前置检查机制

**日期**：2026-06-27

**背景**：
流水线模式下，如何确保写作时 task sheet 已经生成？

**决策**：
在开始某一章的写作前，必须确保该章节的细化已完成。

**理由**：
1. 数据完整性：task sheet 是写作的必要输入
2. 避免错误：没有 task sheet 会导致写作失败

**影响**：
- 需要添加前置检查逻辑
- 需要处理细化未完成的情况

---

### D.6 状态管理方案

**日期**：2026-06-27

**背景**：
流水线模式下，如何管理细化和写作的状态？

**决策**：
在 DirectorAutoExecutionState 中添加 pipelineState 字段，追踪细化和写作的进度。

**理由**：
1. 状态可追踪：可以实时了解执行进度
2. 支持暂停/恢复：可以保存和恢复状态
3. 支持错误处理：可以根据状态决定处理策略

**影响**：
- 需要扩展数据模型
- 需要实现状态持久化
- 需要在 UI 中显示状态

---

### D.7 错误处理策略

**日期**：2026-06-27

**背景**：
流水线模式下，细化失败或写作失败应该怎么处理？

**决策**：
- 细化失败：跳过该章节的写作，继续下一个章节
- 写作失败：尝试自动修复，修复失败则记录质量债务，继续下一个章节

**理由**：
1. 不阻塞流水线：一个章节的失败不应影响其他章节
2. 记录问题：失败的章节会被记录，可以后续处理
3. 用户选择：用户可以选择修复或跳过

**影响**：
- 需要实现错误处理逻辑
- 需要实现质量债务记录
- 需要在 UI 中显示失败状态

---

### D.8 并行度控制

**日期**：2026-06-27

**背景**：
流水线模式下，细化和写作可以并行执行，如何控制并行度？

**决策**：
- 最大并行细化数：1
- 最大并行写作数：1
- 细化与写作可并行

**理由**：
1. 避免资源竞争：过多并行会导致 API 调用竞争
2. 简化逻辑：单线程执行更容易理解和调试
3. 保证连续性：写作需要按顺序执行

**影响**：
- 需要实现并行度控制
- 需要实现队列管理
