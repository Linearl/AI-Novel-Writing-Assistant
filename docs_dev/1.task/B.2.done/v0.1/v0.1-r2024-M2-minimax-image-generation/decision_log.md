# REQ-2024: 决策日志

## 决策 1: MiniMax API 兼容性策略

**决策点**: 如何处理 MiniMax 与 OpenAI 的 API 差异？

| 方案 | 优点 | 缺点 |
|------|------|------|
| A: 在现有函数中添加分支 | 代码复用，改动小 | 逻辑复杂度增加 |
| B: 新建 MiniMax 专用函数 | 逻辑清晰 | 代码重复 |

**选择**: 方案 A（在现有函数中添加分支）

**理由**:
1. MiniMax 与 OpenAI 格式差异不大
2. 复用现有错误处理和超时逻辑
3. 保持代码结构一致性

**日期**: 2026-06-29

## 决策 2: MiniMax baseURL 处理

**决策点**: MiniMax 的 baseURL 如何处理？

| 方案 | 优点 | 缺点 |
|------|------|------|
| A: 硬编码 MiniMax baseURL | 简单直接 | 不灵活 |
| B: 从配置读取 | 灵活 | 需要额外配置 |
| C: 从 APIKey 表读取 | 复用现有机制 | 需要用户配置 |

**选择**: 方案 C（从 APIKey 表读取）

**理由**:
1. 复用现有的 `resolveProviderSecret` 函数
2. 用户可以在设置页面配置 MiniMax API Key
3. 与现有 Provider 配置机制一致

**日期**: 2026-06-29

## 决策 3: MiniMax 错误处理

**决策点**: MiniMax 的错误码如何处理？

| 方案 | 优点 | 缺点 |
|------|------|------|
| A: 统一错误处理 | 代码简单 | 错误信息不精确 |
| B: MiniMax 专用错误处理 | 错误信息精确 | 代码复杂度增加 |

**选择**: 方案 B（MiniMax 专用错误处理）

**理由**:
1. MiniMax 有特定的错误码（如 1026 敏感内容）
2. 需要给用户更精确的错误提示
3. 提升用户体验

**日期**: 2026-06-29
