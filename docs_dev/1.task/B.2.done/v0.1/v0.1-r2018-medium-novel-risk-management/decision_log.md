# REQ-2018 决策日志

## DEC-001: 风险数据模型设计策略

- **决策**: 新建 `NovelRisk` + `RiskAuditLog` 两个 Prisma 模型，不扩展现有模型
- **替代方案**: 扩展现有 `PipelineJobNotice` 或 `DirectorQualityRepairRisk` 来存续风险
- **理由**: 现有模型侧重瞬时通知和局部评级，缺乏生命周期管理和完整审计追踪。新建模型不再受限于原有设计约束
- **影响**: 需要 Prisma 迁移，新增 shared 类型文件 `novelRisk.ts`
- **日期**: 2026-06-26

## DEC-002: 风险面板位置

- **决策**: 面板放在步骤 7（质量修复）下方，作为可折叠区域
- **替代方案**: 独立页面/侧边栏/弹窗
- **理由**: 用户需求明确指定"建议放在步骤7质量修复下面"，与工作流上下文紧密关联，独立页面会增加导航成本
- **影响**: 步骤 7 页面需要新增面板区域，可能需要调整页面布局
- **日期**: 2026-06-26

## DEC-003: 回溯影响评估策略

- **决策**: 首版实现"受影响章节列表 + 预估修复代价"提示，不自动执行修复
- **替代方案**: 自动级联修改受影响章节
- **理由**: 自动修改风险高，应当有人工确认环节。精细化自动修复可作为后续迭代
- **影响**: 回溯操作需要人工介入，但安全性更高
- **日期**: 2026-06-26

## DEC-004: 风险评估引擎

- **决策**: 首版使用规则引擎（基于风险数量和严重程度的简单规则）生成警示，不引入 LLM 评估
- **替代方案**: 调用 LLM 做深度剧情影响分析
- **理由**: 规则引擎确定性高、无延迟、不消耗 token；LLM 评估可作为后续增强
- **影响**: 警示可能不如 LLM 评估细致，但足以覆盖常见风险模式
- **日期**: 2026-06-26

## DEC-005: 导出格式

- **决策**: 支持 Markdown（人类可读）和 JSON（程序化处理）两种格式
- **替代方案**: 仅 Markdown；仅 JSON；CSV 格式
- **理由**: Markdown 方便人工审查和存档，JSON 方便程序化处理和系统集成
- **影响**: 需实现两种序列化逻辑
- **日期**: 2026-06-26

## DEC-006: 任务编号分配

- **决策**: 使用 REQ-2018（2xxx 核心功能开发段）
- **替代方案**: 3918（3xxx 用户界面段）
- **理由**: 此需求涉及完整的数据模型 + 后端 API + 前端面板 + 导出功能，不仅是 UI 变更，归类为"核心功能开发"更准确
- **影响**: 任务包编号为 v0.1-r2018
- **日期**: 2026-06-26
