# REQ-2005 决策日志

## D1: 软删除而非物理删除

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-06-27 |
| 决策人 | AI |
| 决策 | 使用 `deletedAt` 时间戳做软删除，不物理删除章节数据 |
| 原因 | 用户误操作后可恢复；保留删除时间信息便于后续清理策略；不影响其他表的外键引用 |
| 影响 | 查询层需要统一过滤 `deletedAt IS NULL` |
| 备选方案 | 物理删除 — 不可恢复，且需要处理关联数据（自动导演任务、章节批次等） |

## D2: deletedAt 字段而非 isDeleted 布尔

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-06-27 |
| 决策人 | AI |
| 决策 | Chapter 模型新增 `deletedAt DateTime?` 而非 `isDeleted Boolean` |
| 原因 | 时间戳保留删除时间信息，可用于后续"自动清理 N 天前的已删除章节"策略；Prisma 的 `null` 过滤比布尔值过滤更自然 |
| 影响 | 需要新增 migration |
| 备选方案 | `isDeleted Boolean` — 更简单但丢失时间信息 |

## D3: Undo Toast 10 秒窗口

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-06-27 |
| 决策人 | AI |
| 决策 | 删除后显示 10 秒 undo toast，超时后章节仍可通过 API 恢复 |
| 原因 | 10 秒足够用户意识到误操作并撤销；超时后 toast 自动消失不干扰正常使用；软删除数据保留，即使 toast 消失也可通过其他方式恢复 |
| 影响 | 需要 toast 组件支持自定义持续时间和操作按钮 |
| 备选方案 | 永久显示撤销按钮 — 干扰正常使用；不提供 undo — 用户体验差 |

## D4: 删除按钮放在 AI 执行台区域

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-06-27 |
| 决策人 | AI |
| 决策 | 删除按钮放在右侧 AI 执行台（ChapterExecutionActionPanel）中，而非左侧队列 |
| 原因 | 与现有操作按钮（写本章、生成执行计划等）放在一起，用户在选中章节后自然看到；左侧队列空间有限，不适合放破坏性操作按钮 |
| 影响 | 需要在 ChapterExecutionActionPanel 中新增按钮区域 |
| 备选方案 | 左侧队列右键菜单 — 可发现性差；左侧队列滑动删除 — 移动端模式不适合桌面端 |
