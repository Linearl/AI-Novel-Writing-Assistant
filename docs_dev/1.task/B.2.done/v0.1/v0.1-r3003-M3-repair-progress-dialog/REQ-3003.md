# REQ-3003 章节修复进度感知弹窗

## 1. 元信息

| 字段 | 值 |
|------|-----|
| 编号 | REQ-3003 |
| 类型 | UI/UX 增强（3xxx） |
| 复杂度 | medium |
| 优先级 | p2 |
| 目标版本 | 0.1 |
| 创建日期 | 2026-06-26 |

## 2. 背景

用户在"AI 执行台"点击"自动修复问题"后，章节修复在后台进行，SSE 流式输出藏在右侧面板的"资料诊断 → 修复"子标签里。用户零进度感知，不知道修复是否在进行、还要等多久。

## 3. 范围

### 3.1 在范围

1. **"查看详情"按钮**：修复进行中时，在 `ChapterExecutionActionPanel` 的"自动修复问题"按钮下方动态出现
2. **RepairProgressDialog 弹窗组件**：Dialog 弹窗，内嵌修复过程实时展示
3. **流式输出展示**：复用现有 `StreamOutput` 组件展示修复 SSE 文本
4. **状态信息展示**：弹窗标题区显示 `repairRunStatus` 的阶段名（phase）和消息（message）
5. **流式输出完成态**：修复完成后弹窗保持可查看，支持关闭

### 3.2 不在范围

- 不修改后端修复 API
- 不修改 SSE 事件格式
- 不在弹窗内提供"停止修复"按钮（已在 ActionPanel 和 ReferencePanel 中有"
- 不支持同时查看多个章节的修复进度（当前只有一个 repair stream）

## 4. EARS 验收

1. **当**用户点击"自动修复问题"且修复正在进行中，**应**在修复按钮下方出现"查看详情"按钮
2. **当**用户点击"查看详情"按钮，**应**弹出 `RepairProgressDialog`，显示修复流式输出
3. **当**弹窗打开且修复正在进行，**应**实时滚动跟随流式输出
4. **当**修复完成（`repairRunStatus.phase === "finalizing"` 或 `isRepairStreaming === false`），**应**保持弹窗可查看，并显示完整修复内容
5. **当**修复未启动或已结束，**应**隐藏"查看详情"按钮
6. **当**弹窗关闭后再次打开，**应**显示最新的修复内容

## 5. 风险与未决项

| 风险 | 等级 | 缓解 |
|------|------|------|
| 修复流式输出量很大，弹窗性能 | 低 | `StreamOutput` 已处理大文本渲染 |
| 用户切换章节后弹窗行为 | 低 | 弹窗关闭时检测 chapterId 变化，自动关弹窗 |
