# REQ-2002 决策日志

## D1: 流式层检测 vs 事后批量检测

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-06-26 |
| 决策人 | 用户 + AI |
| 决策 | 在流式 chunk 级别实时检测，而非等 LLM 调用结束后再分析 |
| 原因 | 核心目标是减少 token 流费。事后检测虽然更准（可分析全文），但模型已经消耗了全部 token。流式检测可以提前打断，将浪费从 80k~150k 降到 3k~5k。 |
| 影响 | 需要在 `ChapterWritingGraph` 的流式循环中插入检测逻辑 |
| 备选方案 | 事后检测 + 下次重试 — 无法节省已消耗的 token |

## D2: n-gram 重复率算法

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-06-26 |
| 决策人 | AI |
| 决策 | 使用 n-gram 重复率（`1 - unique/total`）作为检测指标 |
| 原因 | n-gram 计算的时间复杂度为 O(n)，适合流式场景。编辑距离 O(n²) 和 LCS O(n²) 在长文本上太慢。n-gram 重复率对"段落循环"模式非常敏感——相同段落重复时，n-gram 唯一率会急剧下降。 |
| 影响 | 无法检测语义级重复（不同措辞表达相同意图），但这属于 prompt 优化范畴 |
| 备选方案 | 编辑距离 / MinHash / 语义相似度 — 精度更高但计算开销不适合流式场景 |

## D3: AbortController 中止 vs 消费丢弃

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-06-26 |
| 决策人 | AI |
| 决策 | 使用 `AbortController.abort()` 直接断开 LLM 流，而非消费但不转发 |
| 原因 | "消费但丢弃"仍然会让 LLM 服务器继续生成和传输 token，只是客户端不使用。`AbortController.abort()` 会关闭 HTTP 连接，LLM 服务器检测到连接关闭后会停止生成。这是真正意义上的"停止消耗"。 |
| 影响 | 需要将 AbortSignal 传递到 `streamTextPrompt()` 的调用链中 |
| 备选方案 | 消费但丢弃 — 服务端仍在消耗算力，网络仍在传输，节省有限 |

## D4: 截断回退到段落边界

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-06-26 |
| 决策人 | AI |
| 决策 | 截断时回退到最近的段落边界（双换行符 `\n\n`），而非句子边界 |
| 原因 | 小说文本以段落为语义单元。句子级截断可能导致一个段落的最后一句缺失但格式完整，读者看不出问题；但段落级截断更安全——不会出现一个对话只说了上半句的情况。中文小说中段落分隔比句子分隔更容易可靠检测。 |
| 影响 | 可能多丢失一个不完整段落的内容（通常 100~300 字），可接受 |
| 备选方案 | 句子边界截断 — 保留更多内容，但中文句号判断不可靠（省略号、引号等干扰） |

## D5: 循环信号与 usageAnomaly 共享计数器

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-06-26 |
| 决策人 | AI |
| 决策 | `streamingLoopDetected` 信号计入现有的 `usageAnomalyCount` 计数器 |
| 原因 | 循环打断本质上是一种"使用异常"——模型行为异常导致资源浪费。与 token 超限共享计数器意味着：2 次循环打断或 1 次循环打断 + 1 次 token 超限，都会触发熔断。这比新增独立计数器更合理——异常信号应该聚合而非分散。 |
| 影响 | 不需要新增独立的熔断器阈值字段 |
| 备选方案 | 独立计数器 — 增加配置复杂度，且两种异常的应对措施相同（停止执行），没必要区分 |

## D6: 首版不做自动重试

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-06-26 |
| 决策人 | AI |
| 决策 | 首版仅截断 + 标记，不自动重试被打断的章节生成 |
| 原因 | 自动重试涉及重试策略（几次？换 prompt？降温度？）、与现有 quality repair 流程的协调、以及防止"截断→重试→再截断"的无限循环。复杂度高，需要单独的需求来设计。首版先做到"截断并保留有效内容"，后续迭代再加重试。 |
| 影响 | 截断后章节状态为 `loop_truncated`，需要用户手动决定下一步（重写/编辑/跳过） |
| 备选方案 | 首版就做自动重试 — 复杂度翻倍，且可能引入新的循环问题 |
