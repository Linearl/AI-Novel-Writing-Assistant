# REQ-2042 决策日志

## D1: soft target + 事后修复，不做 hard constraint

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-07-03 |
| 决策人 | AI |
| 决策 | 字数控制采用 soft target + 事后检测修复方案，不在生成过程中做硬性字数限制 |
| 原因 | 在 LLM 生成过程中强制字数限制（如截断、重复追加指令）会导致生成质量下降、叙事连贯性被破坏。生成后检测 + compress/expand 的方式更灵活：AI 可以自然完成内容，系统再做调整。虽然增加了 LLM 调用次数，但保证了生成质量。 |
| 影响 | 章节生成后增加字数检测步骤，可能触发 compress/expand 额外 LLM 调用。需要设计循环保护防止无限修复。 |
| 备选方案 | 在 generation prompt 中做强约束（如"必须写满 XXX 字"或"不得超过 XXX 字"）— 简单但 AI 遵循度不稳定，且硬截断破坏叙事完整性 |

## D2: 水文检测用 LLM 而非规则

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-07-03 |
| 决策人 | AI |
| 决策 | 水文检测基于 LLM 语义理解，判断每个段落是否与推进剧情/塑造角色/建立世界观相关 |
| 原因 | 水文的本质是"与剧情推进无关的冗余内容"，这需要理解段落在上下文中的语义作用。基于规则的方法（如关键词匹配、句长统计、描写词比例）无法区分"细腻的氛围营造"和"无意义的环境堆砌"。LLM 可以理解剧情语义，做出更准确的判断。 |
| 影响 | 每章增加一次 LLM 调用（水文检测），token 消耗增加。可通过配置开关控制是否启用。 |
| 备选方案 | 基于规则的水文检测（如描写词密度 > 40% 则判定为水文）— 简单但误判率高；用户手动标注 — 准确但用户负担重 |

## D3: 基准字数范围可配置（默认 3000~4000）

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-07-03 |
| 决策人 | AI |
| 决策 | 基准字数范围（min/max）可在小说 settings 中配置，默认值 3000~4000 |
| 原因 | 不同类型的小说对字数的需求差异很大。短篇网文可能单章 2000 字足够，而慢节奏长篇可能需要 5000+ 字。固定的基准值无法适应所有场景。可配置提供了必要的灵活性。默认 3000~4000 是基于主流网络小说的平均章节长度。 |
| 影响 | NovelSettings 增加 baseWordCountMin / baseWordCountMax 字段，settings 页面增加配置入口。 |
| 备选方案 | 硬编码固定基准值 — 简单但不灵活，无法适应不同小说类型 |

## D4: compress/expand 最多 2 轮循环

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-07-03 |
| 决策人 | AI |
| 决策 | compress/expand 后重新检测字数，最多执行 2 轮修复，超过则标记 warning 并继续 |
| 原因 | compress 和 expand 本质上是 LLM 操作，结果不确定性较高——compress 可能删得不够，expand 可能加得不够。理论上可能出现"compress 后仍超限 → 再 compress → 仍超限"的循环。设置 2 轮上限是安全阀：给系统足够的自我修复机会，同时防止无限循环消耗资源。 |
| 影响 | 超过 2 轮后章节标记为 warning 状态，用户可手动处理。 |
| 备选方案 | 无限循环直到达标 — 风险高，可能消耗大量 token 且无保证收敛；单轮修复 — 可能不够充分 |

## D5: 字数达标时才执行水文检测

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-07-03 |
| 决策人 | AI |
| 决策 | 水文检测仅在字数在目标范围内时执行，字数不达标时不执行水文检测 |
| 原因 | 字数不达标（超限或不足）时，系统已经需要执行 compress/expand 修复。在修复后的章节上再执行水文检测才有意义——否则检测结果会因字数异常而失真（如超限章节可能本身就包含大量冗余，检测结果不具参考价值）。这也减少了不必要的 LLM 调用。 |
| 影响 | 水文检测的触发条件增加前置判断，减少 LLM 调用次数。 |
| 备选方案 | 无论字数是否达标都执行水文检测 — 结果可能不准确，且增加 LLM 开销 |
