## 1. 背景与问题

长篇小说创作中，不同章节承担不同的叙事功能，但当前系统对所有章节使用统一的字数限制，导致双重问题：

1. **字数膨胀（水文）**：部分章节的 AI 生成内容存在过度描写、环境铺陈、无关对话等"水文"现象，字数远超必要范围。用户反馈"有些章节读起来很拖沓"。
2. **核心章节字数不足**：高潮章、转折章等关键节点需要足够的篇幅展开冲突、刻画情感变化，但受统一字数限制，内容被压缩，叙事张力不足。
3. **无语义级水文检测**：现有机制仅依赖字数阈值做硬性截断或提醒，无法判断段落是否与剧情推进相关。一篇描写细腻但推动情节的章节可能被误判为"水文"，而真正水的章节可能因字数刚好达标而逃脱检测。

---

## 2. 目标与范围

### 2.1 目标

1. 根据章节角色（普通/过渡 vs 高潮/转折）自动分配不同的目标字数范围
2. 章节角色由 auto-director 在 volume planning / beat sheet 阶段自动标注
3. 生成后检测实际字数与目标范围的偏差，超限自动触发 compress，不足触发 expand
4. 水文检测基于 LLM 语义理解，检测"无效描写密度"（与推进剧情无关的段落比例）

### 2.2 In Scope

**共享层（shared）**：
- Chapter 类型增加 wordCountTarget 字段（min/max/role）

**后端（server）**：
- auto-director 在 volume planning / beat sheet 阶段标注章节角色
- generation pipeline 集成字数约束（prompt 注入）
- 生成后字数检测 + 自动 compress/expand
- 水文检测（LLM 判断无效描写密度）

**前端（client）**：
- 章节列表显示字数目标 vs 实际字数对比

### 2.3 Out of Scope

- 用户手动调整字数系数（仅支持全局基准字数范围配置）
- 水文检测结果的自动修复（仅标记，compress/expand 由字数检测触发）
- 实时字数统计（生成过程中不中断，仅生成后检测）

## 3. 需求详情

### 3.1 章节角色分类与字数系数

| 章节角色 | 系数范围 | 字数范围（基准 3000~4000） |
| ---- | ---- | ---- |
| normal（普通章） | 1.0 ~ 1.25 | 3000 ~ 5000 字 |
| transition（过渡章） | 1.0 ~ 1.25 | 3000 ~ 5000 字 |
| climax（高潮章） | 1.0 ~ 2.5 | 3000 ~ 10000 字 |
| turning_point（转折章） | 1.0 ~ 2.5 | 3000 ~ 10000 字 |

章节角色由 auto-director 在 volume planning / beat sheet 阶段自动标注，标注结果写入 Chapter 的 `wordCountTarget.role` 字段。

### 3.2 字数控制流程

**生成前**：
1. auto-director 读取章节的 `wordCountTarget`，确定目标字数范围（min/max）
2. 将字数范围信息注入生成 prompt，作为 soft target 引导 AI 生成

**生成后**：
1. 检测实际字数 vs 目标范围
2. 超出 max：触发 compress 流程（AI 精简冗余描写）
3. 低于 min：触发 expand 流程（AI 补充缺失内容）
4. 在 min~max 范围内：跳过，进入水文检测

### 3.3 水文检测

水文检测基于 LLM 语义理解，检测"无效描写密度"：

- **定义**：章节中与推进剧情、塑造角色、建立世界观无关的段落占比
- **检测方式**：LLM 对章节内容进行段落级分析，判断每个段落的有效性
- **输出**：无效描写密度百分比（0%~100%）
- **阈值**：超过 30% 视为水文超标，触发精简提醒或自动精简

WHEN 实际字数在目标范围内
THE SYSTEM SHALL 执行水文检测，评估无效描写密度。

WHEN 无效描写密度超过 30%
THE SYSTEM SHALL 标记该章节为水文超标，并建议或自动执行精简。

### 3.4 基准字数范围配置

基准字数范围（默认 3000~4000）可在小说设置中配置，所有系数计算基于该基准。

### 3.5 处理流程图

```
章节生成完成
    ↓
读取 wordCountTarget (min/max/role)
    ↓
计算实际字数
    ↓
实际字数 > max ?
  → YES: 触发 compress → 重新检测字数
  → NO ↓
实际字数 < min ?
  → YES: 触发 expand → 重新检测字数
  → NO ↓
执行水文检测（LLM 语义分析）
    ↓
无效描写密度 > 30% ?
  → YES: 标记水文超标 + 建议精简
  → NO: 通过，结束
```

- [ ] 章节列表显示字数目标 vs 实际字数对比
- [ ] 基准字数范围可在 settings 中配置（默认 3000~4000）
- [ ] 所有测试通过

## 5. 风险与约束

| 风险 | 缓解 |
| ---- | ---- |
| LLM 水文检测准确性有限 | 检测结果作为辅助标记，用户可忽略或手动调整；阈值可配置 |
| compress/expand 可能影响叙事连贯性 | compress/expand 操作附带上下文摘要，保持叙事连贯 |
| 水文检测增加 LLM 调用开销 | 仅对字数在目标范围内的章节执行水文检测；可配置为可选 |
| 章节角色标注不准确 | auto-director 标注结果可由用户手动覆盖 |

---

## 6. 关联与边界

- 与现有章节生成的关系：字数控制嵌入现有 generation pipeline，在生成前注入约束、生成后触发检测
- 与 auto-director 的关系：auto-director 负责章节角色标注（volume planning 阶段）和字数检测触发
- 与 compress/expand 的关系：字数超限或不足时调用现有/新增的 compress/expand 能力
