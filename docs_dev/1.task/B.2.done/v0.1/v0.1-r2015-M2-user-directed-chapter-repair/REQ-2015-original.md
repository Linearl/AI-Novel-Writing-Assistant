# REQ-2015 用户指导式章节修复

> 状态：⏳ 进行中

## 需求元信息

| 字段 | 内容 |
| ---- | ---- |
| 需求编号 | REQ-2015 |
| 优先级 | P2 |
| 来源 | 用户反馈 — 审计后修复选项不足 |
| 关联需求 | 无 |

---

## 1. 背景与问题

当前章节审计（步骤 7）发现高风险问题后，系统提供三种修复方案：

1. **一键补丁（autoRepair）**：AI 根据审计发现的 issues 自动生成局部文本替换
2. **重写整章（rewrite）**：清除内容，从头重新生成整章
3. **手动编辑器**：用户自己打开编辑器修改

**痛点**：这三种方案缺少一个中间地带——用户经常知道问题出在哪里、想怎么改，但不想自己动手写，也不想完全交给 AI 自由发挥或推倒重来。例如：

- "这段节奏太慢了，请加快节奏，删掉冗余描写"
- "这个角色的动机不够，请补充他为什么要做这个决定"
- "结尾太突兀，请加一段心理活动过渡"
- "对话太书面了，请改成口语化"

用户需要的是**提出方向，让 AI 执行**。

---

## 2. 目标与范围

### 2.1 目标

1. 在审计结果面板新增第四种修复入口："按指导修复"
2. 用户输入修改方向文本后，AI 根据用户指示对章节进行定向修改
3. 复用现有 SSE 修复管线，不新增独立端点

### 2.2 In Scope

**后端**：

- 修复请求 schema 扩展 `userInstruction` 可选字段
- 修复运行时识别 `userInstruction` 并强制走 `heavy_repair` 模式
- 修复 prompt 注入用户指导作为最高优先级指令

**前端**：

- 审计结果面板新增"按指导修复"按钮和输入区域
- 输入区域支持自由文本，带字数统计和示例提示
- 修复进行中显示 SSE 流式进度

### 2.3 Out of Scope

- 不新增独立 API 端点，复用现有 `POST /:id/chapters/:chapterId/repair`
- 不修改现有三种修复方案的行为
- 不涉及 auto-director 全自动链路的修改（仅手动触发场景）
- 不涉及审计 issue 自动解析或结构化

---

## 3. 需求详情

### 3.1 用户交互流

**WHEN** 用户在章节执行面板看到审计结果标记为 `needs_repair`
**THEN** 系统在现有修复按钮区域下方显示"按指导修复"折叠面板

**WHEN** 用户展开"按指导修复"面板
**THEN** 系统显示一个 textarea 输入框，包含：

- placeholder 提示："描述你希望怎么修改，例如：'这段节奏太慢，请加快，删掉冗余的环境描写'"
- 字数统计（最多 2000 字）
- 预设示例快捷标签（可点击填入）：节奏加快、补充动机、口语化对话、加强冲突、丰富心理描写

**WHEN** 用户输入修改方向并点击"按指导修复"按钮
**THEN** 系统将用户文本作为 `userInstruction` 发送到修复端点，启动 SSE 修复流

**WHEN** 修复端点收到带 `userInstruction` 的请求
**THEN** 系统构造一个 `ReviewIssue`，其 `fixSuggestion` 为用户指令，`category` 为 `"user_directed"`，并强制使用 `heavy_repair` 模式执行

**WHEN** 修复完成
**THEN** 与现有修复流程一致：显示修复后内容、更新章节状态、可选择接受或继续修改

### 3.2 EARS 验收条目

1. **WHEN** 章节状态为 `needs_repair`，审计面板 SHALL 显示"按指导修复"入口
2. **WHEN** 用户在 textarea 中输入修改方向并提交，系统 SHALL 将文本作为 `userInstruction` 发送至修复端点
3. **WHEN** 修复端点收到带 `userInstruction` 的请求，系统 SHALL 跳过 patch 修复，直接进入 `heavy_repair` 模式
4. **WHEN** `heavy_repair` 执行时，系统 SHALL 将 `userInstruction` 作为最高优先级指令注入修复 prompt 上下文
5. **WHEN** 修复完成，系统 SHALL 以 SSE 流式输出修复过程，与现有修复体验一致
6. **WHEN** 用户未输入任何文字就点击提交，系统 SHALL 显示提示"请输入修改方向"，不允许空提交

---

## 4. 验收标准

- [ ] 审计结果面板出现"按指导修复"入口
- [ ] textarea 输入修改方向后可提交
- [ ] 空输入被拦截，显示错误提示
- [ ] 提交后进入 SSE 修复流，界面显示流式进度
- [ ] 修复结果基于用户方向进行定向修改（非随机重写）
- [ ] 修复完成后章节状态正确更新
- [ ] 现有三种修复方案不受影响

---

## 5. 风险与约束

| 风险 | 缓解 |
| ---- | ---- |
| 用户指导过于模糊导致 AI 修改偏离预期 | 在 placeholder 和示例标签中引导用户给出具体方向 |
| 用户指导与审计 issues 冲突 | `userInstruction` 优先级最高，审计 issues 作为辅助上下文 |
| heavy_repair 模式耗时较长 | SSE 流式输出保持用户感知，不阻塞 UI |

---

## 6. 关联与边界

- 复用现有 `POST /:id/chapters/:chapterId/repair` 端点，不新增端点
- 与 REQ-2004（Creative Hub 可观测性）无交集
- 不影响 auto-director 全自动链路的 `autoRepair` 逻辑
