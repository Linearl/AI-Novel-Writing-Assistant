# 决策日志：服务器日志系统

## 决策 1：日志库选择

**问题**：选择哪个日志库？

**选项**：
1. Winston - 最流行的 Node.js 日志库，功能全面
2. Pino - 高性能日志库，JSON 格式
3. Bunyan - 简单的日志库

**决策**：选择 **Winston**

**理由**：
- 社区活跃，文档完善
- 支持多种传输方式（文件、控制台、HTTP等）
- TypeScript 类型支持良好
- 支持日志轮转和归档（通过 winston-daily-rotate-file）
- 适合企业级应用

**权衡**：
- Pino 性能更好，但功能较少
- Winston 更灵活，适合复杂场景

---

## 决策 2：日志文件轮转策略

**问题**：如何管理日志文件大小和数量？

**选项**：
1. 按日期轮转，保留固定天数
2. 按大小轮转，保留固定文件数
3. 按日期+大小双重轮转

**决策**：选择 **按日期轮转，保留 30 天**

**理由**：
- 简单易懂，便于查询
- 30 天足够回溯大多数问题
- 避免单个文件过大

**实现**：
- 使用 `winston-daily-rotate-file`
- 文件名格式：`app-YYYY-MM-DD.log`
- 超过 7 天自动压缩（.gz）

## 决策 3：日志查询接口设计

**问题**：如何设计日志查询 API？

**选项**：
1. RESTful API（GET /api/logs）
2. GraphQL API
3. WebSocket 实时流

**决策**：选择 **RESTful API**

**理由**：
- 简单直接，易于使用
- 支持缓存和 CDN
- 前后端解耦

**实现**：
- GET `/api/logs`
- 支持参数：level, since, until, limit, offset, keyword, module
- 返回分页结果

---

## 决策 4：日志存储位置

**问题**：日志文件存放在哪里？

**选项**：
1. `server/logs/`（项目内）
2. `/var/log/`（系统日志目录）
3. 云存储（S3、Azure Blob 等）

**决策**：选择 **`server/logs/`（项目内）**

**理由**：
- 便于本地开发和调试
- 无需额外配置
- Git 可忽略（添加到 .gitignore）

**权衡**：
- 生产环境可能需要配置到系统日志目录
- 云存储适合分布式部署

---

## 决策 5：日志安全策略

**问题**：如何保护日志中的敏感信息？

**选项**：
1. 不过滤（最简单）
2. 正则过滤（替换敏感词）
3. 白名单过滤（只记录允许的字段）

**决策**：选择 **正则过滤 + 白名单**

**实现**：
- 自动过滤常见敏感词：password, token, secret, key
- 只记录必要的请求/响应字段
- 敏感操作记录审计日志

**理由**：
- 平衡安全性和可调试性
- 避免意外泄露敏感信息

## 决策 6：日志查询权限控制

**问题**：谁可以查询日志？

**选项**：
1. 无限制（任何人都能查）
2. 管理员权限
3. 认证用户

**决策**：选择 **管理员权限**

**实现**：
- 日志查询接口需要管理员 token
- 普通用户只能查看自己的操作日志
- 管理员可以查看所有日志

**理由**：
- 日志包含敏感信息，需要访问控制
- 防止信息泄露
- 符合安全最佳实践

---

## 决策 7：实现优先级

**问题**：应该先实现哪个功能？

**选项**：
1. 日志写入和轮转（基础功能）
2. 日志查询 API（核心功能）
3. 前端日志查看器（增强功能）
4. 实时日志流（高级功能）

**决策**：按照 **1 → 2 → 3 → 4** 的顺序实现

**理由**：
- 先有数据（写入），才能查询
- 查询 API 是最核心的功能
- 前端查看器是锦上添花
- 实时流是高级功能，优先级最低

---

## 决策 8：错误处理策略

**问题**：日志系统本身出错时如何处理？

**选项**：
1. 静默失败（不通知）
2. 降级到控制台输出
3. 抛出异常，终止服务

**决策**：选择 **降级到控制台输出**

**实现**：
- 日志文件写入失败时，自动降级到控制台
- 记录降级事件（便于排查）
- 不影响正常服务运行

**理由**：
- 日志系统不应影响核心业务
- 控制台输出是最后的保障
- 便于快速定位日志系统问题
