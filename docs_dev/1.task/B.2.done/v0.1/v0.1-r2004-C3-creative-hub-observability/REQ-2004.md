## 1. 背景与问题

### 1.1 现状

在 Creative Hub 中发出指令后，界面上只能看到"规划中..."或"执行中..."等笼统状态，用户无法得知：

1. AI 正在执行哪个具体步骤
2. 当前调用了什么工具、传了什么参数
3. 预计还有多少步骤完成
4. 如果卡住了，卡在哪个环节

### 1.2 后端已有基础设施

后端已经有完整的追踪基础设施，但前端未充分利用：

| 组件 | 文件 | 已有能力 |
| ---- | ---- | ---- |
| AgentTraceStore | `server/src/agents/traceStore.ts` | AgentRun / AgentStep / AgentApproval 全量记录到 SQLite，含 provider、model、tokenUsage、costUsd、durationMs |
| CreativeHubLangGraph | `server/src/creativeHub/CreativeHubLangGraph.ts` | SSE 帧推送 run_status、tool_call、tool_result、interrupt、approval_resolved、turn_summary |
| LLM Usage Tracking | `server/src/llm/usageTracking.ts` | 每次 LLM 调用的 token 用量、延迟、模型、promptAssetKey、modelRoute |
| Agent Runs API | `server/src/routes/agentRuns.ts` | GET `/api/agent-runs/:id` 返回完整 steps + approvals + metrics |

### 1.3 关键差距

**后端有但前端未消费的数据：**

- `AgentStep` 级别的 token 用量、延迟、费用、模型信息 — SSE 帧中不携带
- `AgentRunMetrics`（stepCount、successCount、failureCount、totalDurationMs、totalCostUsd）— 前端从未调用此 API
- `DirectorLlmUsageRecord`（每次 LLM 调用的详细记录）— 无查询 API，前端无法访问
- `creative_hub/tool_call` 和 `creative_hub/tool_result` 帧 — 被 ActivityFeed 显式过滤掉（lines 156-161）
- `CreativeHubDebugTraceEntry.meta` 只是 `string[]`，丢失了结构化数值信息

### 1.4 影响

- 用户对 AI 执行过程是黑盒体验
- 卡住时无法定位是哪个步骤、哪个工具调用出了问题
- 无法评估 token 消耗和成本
- 调试效率低，需要查后端日志才能定位问题

## 2. 目标与范围

### 2.1 目标

1. 用户在 Creative Hub 中能实时看到 AI 执行的步骤列表（类似 CI/CD pipeline 可视化）
2. 用户能看到当前步骤的详情（工具名、参数摘要、使用的模型）
3. 失败时显示具体错误类型（模型超时 / token 超限 / 结构化输出解析失败）
4. 用户能看到本次对话已消耗的 token 数和估算费用

### 2.2 In Scope

**后端：**
- SSE 帧增强：`tool_call` 和 `tool_result` 帧增加 `durationMs`、`model`、`tokenUsage` 字段
- 新增 API：`GET /api/agent-runs/:id/llm-usage` 查询 DirectorLlmUsageRecord
- CreativeHubLangGraph 在每个节点执行时推送更丰富的 `run_status` 帧（含当前步骤名、步骤序号、总步骤数）

**前端：**
- Agent 步骤面板组件（可折叠，在对话区底部或侧栏）
- 步骤列表实时更新（基于 SSE 帧）
- 当前步骤详情展示（prompt ID、模型、token 预估）
- 错误详情展示（区分错误类型）
- Token 消耗仪表（实时累计展示）

**共享类型：**
- 扩展 `CreativeHubStreamFrame` 类型定义

### 2.3 Out of Scope

- 历史 Run 浏览器 UI（仅关注当前 Run 的实时可观测性）
- AgentRun replay 功能（API 已有，UI 不在本期）
- DirectorLlmUsageRecord 的完整 dashboard（本期仅做 token 消耗摘要）
- Creative Hub 线程历史时间线 UI

## 3. 需求详情

### 3.1 Agent 步骤面板

WHEN 用户在 Creative Hub 中发出指令并 AI 开始执行
THE SYSTEM SHALL 在界面中显示一个可折叠的"执行追踪"面板，实时展示：

- 当前 Run 的步骤列表，每个步骤显示：步骤序号、步骤名称、状态图标（进行中/成功/失败/等待审批）
- 当前正在执行的步骤高亮
- 步骤之间用连线表示顺序关系

面板位置：Creative Hub 右侧栏（CreativeHubSidebar）中新增一个 Tab 或折叠区域。

### 3.2 当前步骤详情

WHEN 用户展开某个步骤或查看当前执行步骤
THE SYSTEM SHALL 显示：

- 工具名称（如 `novel_browse`、`chapter_write`）
- 输入参数摘要（截断到 200 字符）
- 使用的模型（如 `gpt-4o`、`claude-sonnet-4-6`）
- 步骤开始时间
- 步骤耗时（进行中时实时更新）

### 3.3 错误详情

WHEN 某个步骤执行失败
THE SYSTEM SHALL 显示具体的错误分类：

| errorCode | 用户可见文案 | 建议操作 |
| ---- | ---- | ---- |
| `model_timeout` | 模型响应超时 | 请稍后重试，或切换到更快的模型 |
| `token_limit` | 输入超出模型上下文限制 | 尝试缩短指令或分步执行 |
| `output_parse_failed` | AI 输出格式异常 | 系统将自动重试 |
| `tool_error` | 工具执行失败 | 检查相关数据是否完整 |
| `approval_expired` | 审批超时 | 请重新发起操作 |
| 其他 | 执行异常 | 联系管理员并提供错误详情 |

### 3.4 Token 消耗仪表

WHEN 一个 Run 完成（或进行中）
THE SYSTEM SHALL 在步骤面板底部显示：

- 本次 Run 已消耗的总 token 数（prompt + completion 分开显示）
- 估算费用（USD）
- 使用的模型列表
- 总耗时

数据来源：优先从 SSE 帧的 `tool_result` 中提取 tokenUsage；Run 结束后从 `/api/agent-runs/:id` 获取完整 metrics 做最终校准。

### 3.5 SSE 帧增强

WHEN 后端推送 `creative_hub/tool_call` 或 `creative_hub/tool_result` 帧
THE SYSTEM SHALL 在帧数据中增加以下字段：

```typescript
// tool_call 帧增强
{
  event: "creative_hub/tool_call";
  data: {
    runId?: string;
    stepId?: string;
    toolName: string;
    inputSummary: string;
    // 新增
    model?: string;          // 当前步骤使用的模型
    stepSeq?: number;        // 步骤序号
    totalSteps?: number;     // 预估总步骤数（如有）
  };
}

// tool_result 帧增强
{
  event: "creative_hub/tool_result";
  data: {
    runId?: string;
    stepId?: string;
    toolName: string;
    outputSummary: string;
    success: boolean;
    output?: Record<string, unknown>;
    errorCode?: string;
    // 新增
    durationMs?: number;     // 步骤耗时
    tokenUsage?: {           // token 用量
      prompt?: number;
      completion?: number;
      total?: number;
    };
    costUsd?: number;        // 估算费用
  };
}
```

### 3.6 run_status 帧增强

WHEN 后端推送 `creative_hub/run_status` 帧
THE SYSTEM SHALL 在帧数据中增加步骤进度信息：

```typescript
{
  event: "creative_hub/run_status";
  data: {
    runId?: string;
    status: string;
    message?: string;
    // 新增
    currentStepName?: string;  // 当前步骤名称（如 "coordinator_plan"、"tool_execute"）
    currentStepSeq?: number;   // 当前步骤序号
    completedSteps?: number;   // 已完成步骤数
  };
}
```

## 4. 验收标准

### 4.1 功能验收

- [ ] Creative Hub 侧栏显示"执行追踪"折叠面板
- [ ] 面板中实时显示当前 Run 的步骤列表
- [ ] 每个步骤显示工具名称、状态、耗时
- [ ] 当前执行步骤高亮显示
- [ ] 步骤详情可展开查看输入参数摘要和模型信息
- [ ] 失败步骤显示具体错误类型和建议操作
- [ ] 面板底部显示 token 消耗摘要（prompt/completion/total/费用/耗时）
- [ ] Run 完成后 token 数据从 API 最终校准

### 4.2 技术验收

- [ ] SSE 帧类型定义与后端实现一致
- [ ] `/api/agent-runs/:id/llm-usage` 端点返回正确的 LLM 调用记录
- [ ] ActivityFeed 不再过滤 `tool_call` 和 `tool_result`（或提供开关）
- [ ] 面板折叠/展开状态持久化（Zustand store）

### 4.3 回归验收

- [ ] 现有 Creative Hub 对话流不受影响
- [ ] 现有 debug trace 卡片仍正常工作
- [ ] turn summary 卡片仍正常工作
- [ ] SSE 帧兼容旧版客户端（新增字段均为 optional）

## 5. 风险与约束

| 风险 | 缓解 |
| ---- | ---- |
| SSE 帧增加数据可能增大传输量 | 仅增加少量元数据字段（model、durationMs、tokenUsage），不传完整 input/output |
| 步骤序号/总数在流式执行中可能不准确 | `totalSteps` 标记为 optional，前端用 `?` 展示 |
| ActivityFeed 过滤逻辑变更可能影响现有 UI | 提供 store 开关让用户选择是否显示工具调用 |
| DirectorLlmUsageRecord 查询可能有性能问题 | 按 runId 索引查询，限制返回条数 |

---

## 6. 关联与边界

- 与 `AgentTraceStore` 的关系：复用已有的 Run/Step 记录，不新建追踪表
- 与 `CreativeHubLangGraph` 的关系：在现有 SSE 推送逻辑中增加字段，不改变图结构
- 与 `CreativeHubSidebar` 的关系：在侧栏中新增执行追踪区域，与现有内容并列
- 与 `CreativeHubActivityFeed` 的关系：调整过滤逻辑，让 tool_call/tool_result 可见
- 与 `CreativeHubDebugTraceCard` 的关系：执行追踪面板是更丰富的替代方案，debug trace 保留为兜底
