## 1. 背景与问题

### 1.1 工作台步骤映射

| 步骤 | Key | 标签 |
| ---- | ---- | ---- |
| 1 | `basic` | 项目设定 |
| 2 | `story_macro` | 故事宏观规划 |
| 3 | `character` | 角色准备 |
| 4 | `outline` | 卷战略 / 卷骨架 |
| 5 | `structured` | 节奏 / 拆章（章节细化） |
| 6 | `chapter` | 章节执行 |
| 7 | `pipeline` | 质量修复 |

### 1.2 当前缺陷

步骤5章节细化完成后，系统自动触发 `syncToChapterExecution: true` 将章节同步到步骤6。同步逻辑中存在缺陷：

- 对于**已有内容**的章节（`hasContent = true`），`preserveWorkflowState = true`，状态被保留 ✓
- 对于**已细化但尚未生成正文**的章节（`hasContent = false`），`preserveWorkflowState = false`，`chapterStatus` 被重置为 `"unplanned"` ✗

这导致虽然章节已获得 taskSheet、sceneCards 等执行合同，但在步骤6的"待写作"队列中仍显示为"待准备"而非"待写作"。

### 1.3 根因分析

**关键文件：** `server/src/services/novel/volume/VolumeChapterSyncService.ts`

在更新已有章节时（lines 127-149）：

```typescript
...(!item.preserveWorkflowState
  ? {
    generationState: "planned",
    chapterStatus: "unplanned",    // <-- 问题所在
  }
  : {}),
```

`preserveWorkflowState` 的判定逻辑在 `volumePlanChangeDetection.ts`（lines 302-303）：

```typescript
clearContent: hasContent && !options.preserveContent,
preserveWorkflowState: hasContent && options.preserveContent,
```

`preserveWorkflowState` 仅在章节有内容时才为 `true`。对于步骤5细化完成但尚未生成正文的章节，`hasContent = false`，导致 `preserveWorkflowState = false`，进而重置状态。

### 1.4 影响

- 步骤6"待写作"过滤器排除 `chapterStatus: "unplanned"` 的章节
- 用户完成章节细化后无法在步骤6看到待写作章节
- 需要手动触发额外操作才能将章节状态推进到 `pending_generation`

## 2. 目标与范围

### 2.1 目标

1. 修正同步逻辑，使已具备执行合同的章节在同步后自动进入 `pending_generation` 状态
2. 步骤6"待写作"队列正确显示已细化章节
3. 不影响已有正文章节的工作流状态保留逻辑

### 2.2 In Scope

**服务端：**
- 修正 `VolumeChapterSyncService.ts` 中的章节状态判定逻辑
- 修正 `volumePlanChangeDetection.ts` 中的 `preserveWorkflowState` 判定
- 新建章节同步时根据执行合同设置初始状态

**验证：**
- 步骤6"待写作"队列过滤逻辑兼容性验证

### 2.3 Out of Scope

- 步骤5同步面板 UI 修改（面板本身已正常工作）
- `chapterStatus` 枚举值新增或修改
- 其他同步场景（如手动同步）的逻辑修改（应保持一致行为）

---

## 3. 需求详情

### 3.1 同步状态保留逻辑修正

WHEN 步骤5章节细化完成后自动同步到步骤6
THE SYSTEM SHALL 按以下规则设置 `chapterStatus`：

| 条件 | chapterStatus |
| ---- | ---- |
| 章节有内容（`hasContent = true`） | 保留现有状态 |
| 章节无内容但有执行合同（taskSheet 或 sceneCards） | `pending_generation` |
| 章节无内容且无执行合同 | `unplanned`（默认） |

### 3.2 新建章节初始状态

WHEN 同步过程中创建新章节
THE SYSTEM SHALL：
- 若新章节携带 taskSheet 或 sceneCards，设置 `chapterStatus = "pending_generation"`
- 若新章节无执行合同，保持默认 `"unplanned"`

### 3.3 已有正文章节状态保护

WHEN 已有正文章节参与同步
THE SYSTEM SHALL 保留其现有 `chapterStatus`，不因同步操作而重置。

---

## 4. 验收标准

- [ ] 步骤5章节细化完成后，自动同步不再将已有执行合同的章节重置为 `unplanned`
- [ ] 已细化（有 taskSheet 或 sceneCards）但无正文的章节，同步后 `chapterStatus` 为 `pending_generation`
- [ ] 已有正文的章节，同步后工作流状态保持不变
- [ ] 步骤6"待写作"队列正确显示已细化章节
- [ ] 新创建的章节在同步时也正确设置初始状态
- [ ] 手动同步与自动同步行为一致

---

## 5. 风险与约束

| 风险 | 缓解 |
| ---- | ---- |
| 修改可能影响已有正文章节的状态保留 | 仅修改无内容章节的逻辑分支，不触碰 `hasContent = true` 路径 |
| 新建章节状态设置可能遗漏边界情况 | 以 taskSheet/sceneCards 非空为判定依据，与现有 plannerPersistence 逻辑对齐 |

---

## 6. 关联与边界

- 与 `plannerPersistence.ts` 的关系：后者在章节规划持久化时已支持将 `unplanned` 推进为 `pending_generation`，本次修复使同步逻辑与其对齐
- 与步骤6队列过滤器的关系：`chapterExecution.shared.tsx` 的"待写作"过滤器已正确支持 `pending_generation`，无需修改
