## 1. 背景与问题

`client/src/pages/novels/NovelEdit.tsx` 达 2731 行，是项目约束（单文件 >700 必须重构）的近 4 倍。该文件是小说编辑主页面，承担了过多职责：布局编排、标签页切换、数据加载、状态管理、子面板渲染等。

不改的影响：
- 新功能开发时冲突率高
- AI 辅助编辑时上下文窗口浪费
- 代码审查困难，bug 定位慢

---

## 2. 目标与范围

### 2.1 目标

1. NovelEdit.tsx 从 2731 行降至 ≤ 400 行（仅保留页面编排逻辑）
2. 拆分出的每个子文件 ≤ 600 行
3. 功能零回归

### 2.2 In Scope

**前端**：
- `client/src/pages/novels/NovelEdit.tsx` — 拆分主体
- 新建 `client/src/pages/novels/components/novelEdit/` — 子组件目录

### 2.3 Out of Scope

- 已有独立组件（如 `NovelAutoDirectorDialog.tsx`、`OutlineTab.tsx` 等）
- 后端代码
- 样式重构（仅结构拆分）

---

## 3. 需求详情

### 3.1 拆分策略

基于文件内容分析，预期拆分为以下模块（实际以 design.md 为准）：

| 模块 | 职责 | 预估行数 |
|------|------|----------|
| `NovelEdit.tsx`（主文件） | 路由、标签页编排、全局状态 | ≤ 400 |
| `NovelEditHeader.tsx` | 顶部工具栏、项目信息 | ~150 |
| `NovelEditTabContent.tsx` | 标签页内容分发 | ~200 |
| `useNovelEditState.ts` | 页面级状态管理 hook | ~300 |
| `useNovelEditActions.ts` | 页面操作（保存、切换等） | ~200 |
| `novelEdit.constants.ts` | 常量、配置 | ~50 |

### 3.2 约束

- 纯结构重构，不改任何业务逻辑
- 每步拆分后 `pnpm typecheck` 必须通过
- 保持现有的 prop drilling / context 模式，不引入新状态管理库

---
