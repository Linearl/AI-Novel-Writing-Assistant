## 1. 背景与问题

### 1.1 现状

系统**完全不感知**模型的上下文窗口大小：

| 层级 | 当前行为 | 问题 |
| ---- | ---- | ---- |
| 数据模型 | `ModelRouteConfig` 只有 `maxTokens`（输出限制） | 无输入上下文窗口配置 |
| Prompt Registry | `ContextPolicy.maxTokensBudget` 是每个 PromptAsset 的硬编码值 | 不随模型变化 |
| Chat | `.slice(-20)` 固定条数截断 | 无 token 感知 |
| CreativeHub | 消息无限累积 | 长对话必然超限 |
| 错误处理 | 搜索 `context_length_exceeded` 零匹配 | 超限时原始 API 错误直接传播 |

### 1.2 核心差距

1. **没有 `contextWindow` 字段** — 系统不知道每个模型能处理多少输入 token
2. **没有自动压缩机制** — 上下文超限时没有降级策略
3. **`maxTokens` 只是输出限制** — 与输入上下文窗口是完全不同的概念

## 2. 目标与范围

### 2.1 目标

1. 每个模型路由可配置上下文窗口大小（默认 1M，可取消勾选降级到 256K）
2. 上下文使用超出阈值（默认 85%）时自动执行压缩
3. 压缩后上下文占用降至 40% 以下
4. 覆盖三条路径：Prompt Registry、Chat、CreativeHub

### 2.2 In Scope

**数据层：**
- `ModelRouteConfig` 新增 `contextWindow` 字段
- `ProviderConfig` 新增 provider 级默认 `contextWindow`
- `ResolvedModel` / `ResolvedLLMClientOptions` 透传 `contextWindow`

**后端：**
- 上下文压缩服务（估算 + 截断 + 摘要）
- Prompt Runner 集成模型感知的上下文预算
- Chat 路由替换 `.slice(-20)` 为 token 感知截断
- CreativeHub 路由增加消息压缩

**前端：**
- 模型路由设置 UI 新增上下文窗口配置（1M Checkbox，默认勾选）

### 2.3 Out of Scope

- AI 摘要压缩（本期只做行级/消息级截断，不做 LLM 调用摘要）
- 精确 tokenizer（本期使用 `字符数/4` 估算）
- 压缩日志 UI 展示
- 模型上下文窗口自动检测（从 provider API 获取）

## 3. 需求详情

### 3.1 模型上下文窗口配置

#### 3.1.1 数据模型

`ModelRouteConfig` 新增字段：

```prisma
contextWindow Int?  // 输入上下文窗口大小（token 数），null 表示使用 provider 默认值
```

默认值：1M（1048576 token）。取消勾选时为 256K（262144 token）。

#### 3.1.2 Provider 默认值

`ProviderConfig` 新增 `defaultContextWindow`：

| Provider | 默认上下文窗口 |
| ---------- | -------------- |
| openai | 128000 |
| anthropic | 200000 |
| deepseek | 65536 |
| google | 1048576 |
| moonshot | 128000 |
| 其他 | 1048576 (1M，项目默认) |

#### 3.1.3 设置 UI

WHEN 用户在模型路由设置页面编辑路由
THE SYSTEM SHALL 显示上下文窗口配置：

```
上下文窗口
☑ 1M 上下文（推荐，适合长篇写作）
```

默认勾选 1M。取消勾选时为 256K。使用 Checkbox 而非 Radio，因为本项目以长篇小说写作为主，1M 是绝大多数场景的首选。

### 3.2 上下文使用估算

#### 3.2.1 Token 估算

复用现有 `estimateTextTokens()` 函数（`contextBudget.ts`）：

```typescript
function estimateTextTokens(text: string): number {
  return Math.max(1, Math.ceil(text.length / 4));
}
```

#### 3.2.2 上下文使用率计算

```typescript
function calculateContextUsage(
  systemPrompt: string,
  messages: Array<{ content: string }>,
  contextBlocks: Array<{ content: string }>,
  contextWindow: number
): { usedTokens: number; totalWindow: number; usageRatio: number } {
  const systemTokens = estimateTextTokens(systemPrompt);
  const messageTokens = messages.reduce((sum, m) => sum + estimateTextTokens(m.content), 0);
  const blockTokens = contextBlocks.reduce((sum, b) => sum + estimateTextTokens(b.content), 0);
  const usedTokens = systemTokens + messageTokens + blockTokens;
  return {
    usedTokens,
    totalWindow: contextWindow,
    usageRatio: usedTokens / contextWindow,
  };
}
```

### 3.3 自动压缩触发

#### 3.3.1 阈值配置

| 参数 | 默认值 | 说明 |
| ---- | ---- | ---- |
| `CONTEXT_COMPRESS_THRESHOLD` | 0.85 (85%) | 触发压缩的使用率阈值 |
| `CONTEXT_COMPRESS_TARGET` | 0.40 (40%) | 压缩后的目标使用率 |

#### 3.3.2 压缩策略

WHEN 上下文使用率超过 `CONTEXT_COMPRESS_THRESHOLD`
THE SYSTEM SHALL 执行以下压缩（按优先级）：

**阶段1：丢弃低优先级上下文块**
- 丢弃 `priority < 5` 的 optional 上下文块
- 丢弃 `dropOrder` 最高的块
- 每丢弃一块后重新计算使用率

**阶段2：截断消息历史**
- 保留最近 N 条消息（按时间倒序）
- 丢弃更早的消息
- 保留 system message 不动
- 目标：使用率降至 `CONTEXT_COMPRESS_TARGET` 以下

**阶段3：截断剩余上下文块**
- 对剩余的 optional 块执行 `summarizeContextBlock()`（行级截断）
- 从最低优先级的块开始截断
- 目标：使用率降至 `CONTEXT_COMPRESS_TARGET` 以下

**阶段4：截断 required 块（最后手段）**
- 对 required 块执行 `summarizeContextBlock()`
- 保留首行 + 尽可能多的内容

#### 3.3.3 压缩日志

每次压缩后记录：

```typescript
interface CompressionLog {
  trigger: "threshold_exceeded";
  beforeTokens: number;
  afterTokens: number;
  contextWindow: number;
  usageBefore: number;
  usageAfter: number;
  droppedBlocks: string[];
  truncatedBlocks: string[];
  droppedMessages: number;
}
```

日志输出到 console.warn（开发模式），便于调试。

### 3.4 Prompt Registry 集成

WHEN `promptRunner.ts` 执行 prompt
THE SYSTEM SHALL：

1. 从 `ResolvedLLMClientOptions` 读取 `contextWindow`
2. 计算 `effectiveBudget = min(promptAsset.contextPolicy.maxTokensBudget, contextWindow * 0.7)`
3. 将 `effectiveBudget` 传递给 `selectContextBlocks()`
4. 若使用率超过阈值，触发压缩

### 3.5 Chat 路由集成

WHEN Chat 路由处理消息
THE SYSTEM SHALL：

1. 从当前模型路由读取 `contextWindow`
2. 估算所有消息的总 token 数
3. 若超过阈值，从最早的消息开始丢弃（保留 system message）
4. 目标：使用率降至 `CONTEXT_COMPRESS_TARGET` 以下
5. 替换现有的 `.slice(-20)` 硬截断

### 3.6 CreativeHub 集成

WHEN CreativeHub 处理线程消息
THE SYSTEM SHALL：

1. 从当前模型路由读取 `contextWindow`
2. 估算 checkpoint 消息 + 新消息的总 token 数
3. 若超过阈值，从最早的 checkpoint 消息开始丢弃
4. 保留最近 3 个 checkpoint 的消息
5. 在 `buildSeedMessages()` 中执行压缩

## 4. 验收标准

### 4.1 配置

- [ ] 模型路由设置页面显示 1M 上下文 Checkbox（默认勾选）
- [ ] 默认值为 1M，取消勾选时为 256K
- [ ] 配置保存到 `ModelRouteConfig.contextWindow`
- [ ] Provider 级默认值正确回退

### 4.2 自动压缩

- [ ] 上下文使用率超过 85% 时自动触发压缩
- [ ] 压缩后使用率降至 40% 以下
- [ ] 压缩过程对调用方透明（不报错）
- [ ] 压缩日志正确记录丢弃/截断的块和消息数

### 4.3 路径覆盖

- [ ] Prompt Registry 路径：上下文块按模型预算选择/截断
- [ ] Chat 路径：消息按 token 感知截断（替换 `.slice(-20)`）
- [ ] CreativeHub 路径：checkpoint 消息按 token 感知压缩

### 4.4 回归

- [ ] 现有 prompt 执行不受影响（budget=0 的 PromptAsset 仍无限制）
- [ ] 现有 Chat 行为兼容（20 条以内不触发压缩）
- [ ] 现有 CreativeHub 行为兼容（短对话不触发压缩）

## 5. 风险与约束

| 风险 | 缓解 |
| ---- | ---- |
| 字符数/4 估算不精确 | 作为 v1 估算，后续可引入 tokenizer |
| 压缩可能丢弃重要上下文 | 优先丢弃低优先级块，required 块最后处理 |
| Chat 压缩可能丢失对话连贯性 | 保留最近消息，丢弃最早消息 |
| CreativeHub 压缩可能丢失 checkpoint 上下文 | 保留最近 3 个 checkpoint |
