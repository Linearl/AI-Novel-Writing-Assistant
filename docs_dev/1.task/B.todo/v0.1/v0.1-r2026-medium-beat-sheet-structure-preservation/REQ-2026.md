## 1. 背景与问题

### 1.1 现状

REQ-2007 为节奏板重生成添加了 `referenceExisting` 选项，将旧 beat sheet 摘要和已细化章节摘要作为 `reference` group（priority 55-60）注入 prompt。但存在三个缺口：

| 缺口 | 影响 |
| ---- | ---- |
| prompt 中无结构保持指令 | LLM 可自由改变 beat 数量、key、章节数分配 |
| `reference` group 优先级低，token 紧张时被 drop | 参考信息可能根本不会出现在最终 prompt 中 |
| 无后处理验证 | LLM 输出结构不一致时无法感知，直接接受 |

### 1.2 根因

`beatSheet.prompts.ts` 的 system prompt 只约束了 beat 数量（5-8 条）和 chapterSpanHint 覆盖范围，没有约束"重新生成时必须保持与已有结构的一致性"。`contextBlocks.ts` 中 `reference` group 不在 `requiredGroups` 或 `preferredGroups` 中，token 预算紧张时会被优先裁剪。

### 1.3 影响链路

```
节奏板全量替换（mergeBeatSheet）
  ↓ 新 beats 可能与旧结构完全不同
章节列表生成时 buildExistingBeatBlocks()
  ↓ 旧章节按 beatKey/span 匹配到新 beat
resolveFullVolumeResumeState() 逐 beat 比对章节数
  ↓ 大概率：新旧结构不一致 → 从不一致处开始全部重生成
已有章节的 purpose/taskSheet/sceneCards 丢失或错位
```

### 1.4 关键文件

| 文件 | 作用 |
| ---- | ---- |
| `server/src/prompting/prompts/novel/volume/contextBlocks.ts` | 上下文块构建，reference group 优先级 |
| `server/src/prompting/prompts/novel/volume/beatSheet.prompts.ts` | Beat sheet prompt 模板 |
| `server/src/services/novel/volume/volumeBeatSheetGeneration.ts` | Beat sheet 生成函数 |
| `server/src/services/novel/volume/volumeGenerationHelpers.ts` | mergeBeatSheet、mergeChapterList、resolveVolumeChapterBeatKey |

## 2. 目标与范围

### 2.1 目标

1. 当 `referenceExisting=true` 时，prompt 中包含结构保持硬性指令（beat 数量、key、章节数）
2. LLM 输出后验证新旧 beat 结构一致性
3. 验证失败时自动重试（最多 1 次），将失败原因注入 guidance
4. 与现有 `resolveFullVolumeResumeState()` 形成两层保护

### 2.2 In Scope

**Prompt 层：**
- `contextBlocks.ts`：`existing_beat_sheet` 和 `existing_chapter_details` 从 `reference` group 提升到 `preferred` group
- `beatSheet.prompts.ts`：system prompt 新增结构保持约束段（仅 `referenceExisting=true` 时生效）

**后处理层：**
- `volumeBeatSheetGeneration.ts`：新增 `validateBeatStructurePreservation()` 验证函数
- 验证失败时注入 guidance 重试（最多 1 次）

### 2.3 Out of Scope

- 强制保留已有 beat 结构的 Beat 锁定机制（方案 C，留作后续需求）
- 章节列表层的匹配逻辑改造
- 前端 UI 变更

## 3. 需求详情

### 3.1 Prompt 结构保持约束

WHEN `referenceExisting=true` 且存在已有 beat sheet
THE SYSTEM SHALL 在 system prompt 中追加结构保持约束段：

```
【结构保持要求】
当上下文中包含"已有节奏板结构"时，你必须遵守：
1. beats 数量必须与已有结构一致
2. 每个 beat 的 key 必须沿用已有结构的 key
3. 每个 beat 的 chapterSpanHint 必须覆盖相同数量的章节
4. 只能调整 summary 和 mustDeliver 的内容
```

### 3.2 Context Block 优先级提升

WHEN `referenceExisting=true`
THE SYSTEM SHALL 将 `existing_beat_sheet` 和 `existing_chapter_details` 的 group 从 `reference` 改为 `preferred`，确保不被 token 预算裁剪。

### 3.3 后处理验证

WHEN LLM 返回新 beats 且 `referenceExisting=true` 且存在已有 beat sheet
THE SYSTEM SHALL 验证以下条件：

| 条件 | 验证方式 |
| ---- | ---- |
| beat 数量一致 | `newBeats.length === existingBeatSheet.beats.length` |
| key 保持 | 每个已有 beat 的 key 在新 beats 中存在 |
| 章节数一致 | 每个 beat 的 chapterSpanHint 解析后的章节数与旧的一致 |

### 3.4 重试机制

WHEN 验证失败
THE SYSTEM SHALL 将失败原因注入 guidance 触发重试，最多重试 1 次。

重试 prompt 中追加：
```
结构保持约束未满足: {reason}。请严格保持已有 beat 结构，只修改 summary 和 mustDeliver。
```

WHEN 重试仍失败
THE SYSTEM SHALL 使用重试前的最佳输出继续流程，记录 warning 日志。

## 4. 验收标准

- [ ] `referenceExisting=true` 时，prompt 中包含结构保持指令段
- [ ] `existing_beat_sheet` context block 提升到 preferred group
- [ ] LLM 输出 beats 后，验证函数检查 beat 数量、key、章节数
- [ ] 验证失败时自动重试，重试 prompt 中包含失败原因
- [ ] 重试仍失败时正常返回（降级为 LLM 最佳输出），记录 warning 日志
- [ ] `referenceExisting=false` 时行为不变
- [ ] 首次生成（无已有 beat sheet）时行为不变
- [ ] 现有 `postValidate`（章节覆盖验证）仍正常工作

## 5. 风险与约束

| 风险 | 缓解 |
| ---- | ---- |
| LLM 不遵守结构保持约束 | 后处理验证 + 重试兜底 |
| 重试增加延迟和 token 消耗 | 最多重试 1 次；验证通过时零额外成本 |
| 提升 context block 优先级可能挤占其他上下文 | `preferred` group 在 `required` 之后裁剪，影响有限 |
| 章节数验证依赖 chapterSpanHint 解析 | 已有 `parseBeatChapterSpan` 函数，复用现有逻辑 |
