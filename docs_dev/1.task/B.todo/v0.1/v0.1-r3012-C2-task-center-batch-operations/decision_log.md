# 任务中心批量操作功能 - 决策日志

## 决策 1: 选中状态管理方式

**日期**: 2026-07-11

**决策**: 使用 React useState 在 TaskCenterPage 组件中管理选中状态

**理由**:
- 选中状态是临时的，不需要持久化
- 页面刷新后清除选中状态符合用户预期
- 避免增加全局状态管理复杂度

**替代方案**:
- 方案 B: 使用 Zustand 全局 store
  - 优点: 跨页面保持状态
  - 缺点: 增加复杂度，临时状态不需要跨页面共享

**结论**: 选择方案 A (useState)

## 决策 2: 复选框位置

**日期**: 2026-07-11

**决策**: 复选框放在任务项左侧，与任务标题同行

**理由**:
- 符合常见 UI 设计模式（如邮件列表、文件管理器）
- 用户可以快速扫描选中状态
- 不增加任务项高度

**替代方案**:
- 方案 B: 复选框放在任务项右侧
  - 优点: 无
  - 缺点: 不符合用户习惯，难以快速扫描

**结论**: 选择方案 A (左侧)

---

## 决策 3: 全选框实现方式

**日期**: 2026-07-11

**决策**: 使用 HTML 原生 checkbox 的 indeterminate 属性实现部分选中状态

**理由**:
- 浏览器原生支持，无需额外实现
- 语义正确， accessibility 友好
- 实现简单

**实现**:
```typescript
const checkboxRef = useRef<HTMLInputElement>(null);
useEffect(() => {
  if (checkboxRef.current) {
    checkboxRef.current.indeterminate = isPartiallySelected;
  }
}, [isPartiallySelected]);
```

**结论**: 选择原生 indeterminate 属性

## 决策 4: 批量取消策略

**日期**: 2026-07-11

**决策**: 逐个调用 API 取消，非并发执行

**理由**:
- 避免服务器压力过大
- 可以准确追踪每个任务的取消结果
- 失败时可以重试单个任务

**替代方案**:
- 方案 B: 并发执行所有取消请求
  - 优点: 速度快
  - 缺点: 服务器压力大，难以追踪失败任务

**结论**: 选择方案 A (逐个执行)

---

## 决策 5: 不可取消任务处理

**日期**: 2026-07-11

**决策**: 自动跳过不可取消的任务，并在操作完成后提示用户

**理由**:
- 避免向用户展示不可操作的任务
- 减少用户困惑
- 符合"静默处理可预期情况"原则

**实现**:
- 操作前过滤出可取消的任务
- 操作后显示"成功取消 X 个，跳过 Y 个不可取消的任务"

**结论**: 选择自动跳过策略
