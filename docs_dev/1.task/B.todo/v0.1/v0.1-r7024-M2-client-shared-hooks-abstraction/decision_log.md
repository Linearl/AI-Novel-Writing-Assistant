# REQ-7024 决策留痕

## 决策记录

### D-01：useApiMutation 定位为便利层，不替代底层 useMutation

| 项目 | 内容 |
| ---- | ---- |
| 决策日期 | 2026-07-10 |
| 决策者 | AI 建议 |
| 决策内容 | useApiMutation 封装常用模式（toast + invalidate），但保留底层 useMutation 的直通能力 |
| 决策理由 | 不是所有 mutation 都需要 toast（如后台轮询）或 invalidate（如纯副作用操作）。强制统一会引入不必要的抽象泄漏 |
| 备选方案 | 强制所有 mutation 改用 useApiMutation — 会导致特殊场景需要传空回调绕过默认行为 |

### D-02：拆分策略用功能边界，不机械按行数切割

| 项目 | 内容 |
| ---- | ---- |
| 决策日期 | 2026-07-10 |
| 决策者 | AI 建议 |
| 决策内容 | 大文件拆分按组件的功能职责边界进行，而不是按代码行数机械切割 |
| 决策理由 | 机械切割（如前 300 行一个文件、后 300 行一个文件）会破坏逻辑内聚性，导致跨文件的状态共享和 props drilling |
| 备选方案 | 按行数平均切割 — 会破坏组件内聚性 |

### D-03：NovelWorkspaceRail 拆分为 Shell + DataProvider + Nav 三层

| 项目 | 内容 |
| ---- | ---- |
| 决策日期 | 2026-07-10 |
| 决策者 | AI 建议 |
| 决策内容 | NovelWorkspaceRail（678 行 god component）拆分为布局壳 + 数据加载层 + 导航控制器 |
| 决策理由 | 当前文件混合了 layout JSX、数据获取、tab 状态管理三类职责。三层拆分后每个文件 <300 行，职责清晰 |
| 备选方案 | 按视觉区域拆分（sidebar / main / header）— 数据获取逻辑仍需跨组件共享，不如按职责拆分清晰 |

### D-04：useApiQuery 在 T1 调研后决定是否实施

| 项目 | 内容 |
| ---- | ---- |
| 决策日期 | 2026-07-10 |
| 决策者 | AI 建议 |
| 决策内容 | useApiQuery 留到 T1 调研阶段根据实际重复模式数量决定，不预设实施 |
| 决策理由 | useQuery 的样板代码通常比 useMutation 少（主要是 staleTime/cacheTime 配置），可能不值得抽象。先调研再决定 |
| 备选方案 | 预设必须实施 — 可能创建无实际价值的抽象 |
