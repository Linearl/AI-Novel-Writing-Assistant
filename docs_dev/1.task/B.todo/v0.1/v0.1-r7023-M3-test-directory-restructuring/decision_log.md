# REQ-7023 决策日志

## D1: 覆盖率工具选 c8 而非 Node 22+ 内置方案

**日期**：2026-07-10
**决策**：使用 c8 作为覆盖率工具
**原因**：项目测试跑在 CJS dist/ 上，c8 对 CJS + source map 的支持最成熟。Node 22+ `--experimental-test-coverage` 仍在实验阶段，且要求原生 ESM。
**影响**：需新增一个 devDependency（c8），但体积小、零配置。

## D2: 增量迁移而非一次性重组

**日期**：2026-07-10
**决策**：按领域逐批迁移测试文件，每批独立提交和验证
**原因**：一次性移动 220 个文件风险大（import 断裂难以排查，review 困难）。分批迁移可及时发现问题，也降低与其他并行开发的冲突概率。
**影响**：迁移周期变长（多个 batch），但安全可控。

## D3: 迁移顺序按文件数从少到多

**日期**：2026-07-10
**决策**：优先迁移文件数少的领域（tools/ → routes/ → ... → runtime/）
**原因**：文件数少的领域迁移风险最低，先积累经验再处理大领域。每批成功后对后续领域更有信心。
**影响**：runtime/（可能文件最多）最后迁移。

## D4: CI 覆盖率门禁初期设为仅报告不阻断

**日期**：2026-07-10
**决策**：覆盖率门禁初期 50%，设置为 warning 模式（仅报告不阻断 CI）
**原因**：初期覆盖率可能低于 50%，阻断 CI 会阻塞正常开发。先获取基线数据，再逐步提高阈值并改为 blocking。
**影响**：需在 CI 配置中设置 `fail-on-low-coverage: false` 或等效配置。

## D5: tsx 评估为可选任务，不影响主流程

**日期**：2026-07-10
**决策**：tsx + node --test 评估独立进行，不阻塞 Phase 2 增量迁移
**原因**：tsx 方案是优化（减少构建步骤），但即使不可行也不影响覆盖率接入和目录重构。独立评估避免成为前置阻塞项。
**影响**：Phase 1.3 可延后执行，Phase 2 不依赖其结果。
