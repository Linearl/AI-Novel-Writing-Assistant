# REQ-7044: 检查点管理

## 1. 需求背景

### 1.1 问题描述

系统现有的检查点机制仅支持创建和恢复检查点，缺乏对检查点的生命周期管理能力。随着长篇小说创作的推进（160章+），检查点数量快速增长，占用大量存储空间，且无法自动清理过期检查点，导致系统维护成本增加。

### 1.2 现状分析

**已有功能**：
- 检查点创建（章节完成时自动保存）
- 检查点恢复（支持从指定检查点恢复执行）
- 检查点持久化（SQLite存储）

**缺失功能**：
- ❌ 自动清理旧检查点（按策略保留最近N个）
- ❌ 检查点列表查询API
- ❌ 检点手动删除API
- ❌ 检查点元信息管理（标签、说明）

### 1.3 目标用户

所有使用系统进行长篇小说创作的用户，特别是：
- 创作100章+长篇的用户
- 需要频繁回滚到特定章节的用户
- 关注存储空间的用户

## 2. 需求定义

### 2.1 功能需求

#### FR-1: 自动清理旧检查点

**描述**：系统自动按策略清理过期检查点，仅保留最近的检查点。

**规则**：
- 默认保留最近20个检查点
- 清理时机：每次创建新检查点时触发
- 可配置保留数量
- 清理时不删除"标记保留"的检查点

**验收标准**：
- [ ] 创建第21个检查点时自动删除第1个
- [ ] 标记保留的检查点不被清理
- [ ] 保留数量可通过API配置

#### FR-2: 检查点列表查询API

**描述**：提供API查询指定小说的所有检查点列表。

**规则**：
- 按创建时间倒序排列
- 支持分页查询
- 返回检查点基本信息（ID、章节号、创建时间、标签）

**验收标准**：
- [ ] API返回正确的检查点列表
- [ ] 支持分页参数
- [ ] 列表按时间倒序排列

#### FR-3: 检查点删除API

**描述**：支持手动删除指定检查点。

**规则**：
- 支持单个删除
- 支持批量删除
- 已标记保留的检查点需确认后删除

**验收标准**：
- [ ] 可删除单个检查点
- [ ] 可批量删除多个检查点
- [ ] 标记保留的检查点删除时返回确认提示

#### FR-4: 检查点标记保留

**描述**：支持标记重要检查点为"保留"状态，防止被自动清理。

**验收标准**：
- [ ] 可将检查点标记为保留
- [ ] 可取消保留标记
- [ ] 保留状态在列表中可见

### 2.2 非功能需求

#### NFR-1: 性能

- 查询API响应时间<100ms
- 自动清理操作不阻塞主流程

#### NFR-2: 存储

- 清理操作释放磁盘空间
- 记录清理日志便于审计

#### NFR-3: 兼容性

- 不影响现有检查点创建和恢复功能
- 向后兼容现有数据

## 3. 技术约束

### 3.1 架构约束

- 必须复用现有的Prisma Checkpoint模型
- API路由遵循模块化设计（`modules/novel/`）
- 前端页面在检查点管理界面操作

### 3.2 依赖约束

- 依赖：无
- 被依赖：REQ-7046（批量任务队列，需要检查点清理配合）

### 3.3 数据约束

- Checkpoint表已有字段：id, novelId, chapterIndex, data, createdAt
- 需要新增字段：isPinned（是否保留）, label（标签）

## 4. 验收标准

### 4.1 功能验收

- [ ] 自动清理保留最近20个检查点
- [ ] 列表API返回正确数据
- [ ] 删除API正常工作
- [ ] 标记保留的检查点不被自动清理

### 4.2 性能验收

- [ ] 列表查询<100ms
- [ ] 自动清理<500ms

### 4.3 测试验收

- [ ] 单元测试覆盖率>80%
- [ ] 集成测试通过

## 5. 风险评估

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|----------|
| 清理误删重要检查点 | 用户数据丢失 | 中 | 标记保留机制 + 清理日志 |
| 数据库字段迁移失败 | 功能不可用 | 低 | Prisma migrate安全执行 |
| 存储释放不及时 | 磁盘空间不足 | 低 | 后台异步清理 |

## 6. 工作量评估

- **开发时间**：0.5天
- **测试时间**：0.5天
- **总计**：1天

- 不影响核心生成流程
