# REQ-7062: LLM 基础设施增强 — 决策日志

## 决策1: Token 追踪传播机制

**决策日期**：2026-07-14

**背景**：Token 用量追踪如何在调用链路中传播上下文？

**选项**：
- **方案A**: AsyncLocalStorage（Node.js 内置，自动传播）
- **方案B**: 显式参数传递（每个函数都传递追踪上下文）
- **方案C**: 全局单例存储（简单但不支持并发请求隔离）

**决策**：✅ 方案A

**理由**：
1. AsyncLocalStorage 是 Node.js 原生方案，无外部依赖
2. 自动传播到所有异步调用，无需修改每个函数签名
3. 支持并发请求隔离（每个请求有独立上下文）
4. 上游已验证可用（usageTracking.ts）

**影响**：
- 需要在调用链入口使用 `usageTrackingStorage.run()`
- 需要确保 AsyncLocalStorage 上下文正确传播
- 内存开销可忽略（每个请求一个上下文对象）

## 决策2: Token 数据持久化策略

**决策日期**：2026-07-14

**背景**：Token 用量数据如何持久化？

**选项**：
- **方案A**: 实时写入数据库（简单但 IO 开销大）
- **方案B**: 批量写入（30 秒间隔 + 缓冲区上限）（平衡）
- **方案C**: 仅写入日志文件，不入数据库（最简但查询困难）

**决策**：✅ 方案B

**理由**：
1. 实时写入对高频 LLM 调用场景 IO 压力大
2. 批量写入减少 DB 连接开销
3. 30 秒间隔 + 100 条上限平衡了数据安全和性能
4. 写入失败时数据在内存中保留，下次重试

**影响**：
- 最多丢失 30 秒数据（极端崩溃场景）
- 查询延迟最多 30 秒（数据可能未刷写）
- 缓冲区内存占用 < 100KB

## 决策3: 验收状态枚举 vs 字符串

**决策日期**：2026-07-14

**背景**：验收状态应该用 TypeScript 枚举还是字符串字面量？

**选项**：
- **方案A**: TypeScript `enum`（运行时对象）
- **方案B**: `as const` 字符串字面量（类型安全 + 轻量）
- **方案C**: 保持现有字符串散布（不改变）

**决策**：✅ 方案B

**理由**：
1. `as const` 比 `enum` 更轻量，无运行时对象开销
2. 类型安全与 `enum` 等效
3. `normalizeAcceptanceStatus()` 处理旧数据兼容
4. 未来可轻松扩展新状态

**影响**：
- 所有验收状态引用点需更新为枚举常量
- 旧数据通过 `normalizeAcceptanceStatus()` 自动兼容
- Prisma schema 可选使用原生 enum 或 String 类型

## 决策4: 热重载方式

**决策日期**：2026-07-14

**背景**：请求限制器热重载如何实现？

**选项**：
- **方案A**: 模块缓存清除（`evictSharedLimiters()`）
- **方案B**: 文件监听 + 自动重载（chokidar/watch）
- **方案C**: 事件驱动重载（EventBus 通知）

**决策**：✅ 方案A

**理由**：
1. 最简实现，一个函数清除 Map 即可
2. 不需要文件监听或事件系统
3. 与上游实现一致
4. 后续如需自动重载可在此基础上扩展

**影响**：
- 需要 API 触发（非自动）
- 进行中的请求使用旧限制器（无影响）
- 下次请求自动创建新限制器
