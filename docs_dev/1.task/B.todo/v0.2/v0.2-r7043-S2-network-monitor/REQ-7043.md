# REQ-7043: 网络状态监控

## 1. 需求背景

### 1.1 问题描述

当前系统在网络断开时，LLM API 调用会直接失败，正在执行的 Director 任务和 Side Effect Job 被迫中断，用户需要手动重启任务。在长篇小说创作场景中（160章+），一次断网可能导致整个流水线停滞，恢复后还需人工介入继续。

### 1.2 现状分析

**已有功能**：
- `server/src/llm/connectivity.ts` — 手动触发的 LLM 连通性探测（plain + structured probe）
- `invokeWithRetry`（REQ-7040）— LLM 调用层的指数退避重试
- SQLite withSqliteRetry — 数据库层瞬态错误重试

**缺失功能**：
- ❌ 后台自动心跳检测（无定时、无状态持久化）
- ❌ 网络状态变化事件发布（其他模块无法订阅）
- ❌ 断网时自动暂停任务（Director/SideEffect 无感知网络状态）
- ❌ 网络恢复后自动继续（无状态恢复机制）

### 1.3 目标用户

所有使用系统进行长篇小说创作的用户，特别是：
- 长时间运行自动导演任务的用户
- 网络环境不稳定的用户（移动热点、公共 WiFi）
- 无人值守批量生成场景的用户

## 2. 需求定义

### 2.1 功能需求

#### FR-1: 后台心跳检测

**描述**：系统以可配置的间隔周期性发送轻量级 LLM 探测请求，检测网络状态。

**规则**：
- 默认心跳间隔：30 秒（可配置）
- 探测方式：复用现有 `testPlainConnection`，发送 16 token 请求
- 连续失败 3 次判定为断网（避免单次抖动误判）
- 连续成功 1 次判定为恢复（快速恢复）

**验收标准**：
- [ ] 后台定时器按配置间隔执行心跳
- [ ] 连续失败 3 次触发断网状态变更
- [ ] 连续成功 1 次触发恢复状态变更
- [ ] 心跳探测失败不影响其他 LLM 调用

#### FR-2: 网络状态事件发布

**描述**：网络状态变化时通过 EventBus 发布事件，供 Director 和 SideEffect 订阅。

**事件类型**：
- `network:online` — 网络恢复在线
- `network:offline` — 网络断开

**事件负载**：
```typescript
{
  timestamp: string;        // ISO 时间戳
  reason: string;           // 断网原因/恢复依据
  lastSuccessAt?: string;   // 最近一次成功探测时间
  probeLatency?: number;    // 最近一次探测延迟(ms)
}
```

**验收标准**：
- [ ] 断网时发布 `network:offline` 事件
- [ ] 恢复时发布 `network:online` 事件
- [ ] 事件负载包含诊断信息
- [ ] 事件通过 `novelEventBus` 发布

#### FR-3: 网络状态查询接口

**描述**：提供 API 接口查询当前网络状态和最近心跳历史。

**返回数据**：
```typescript
{
  isOnline: boolean;
  lastCheckAt: string;
  lastSuccessAt: string | null;
  consecutiveFailures: number;
  recentProbes: Array<{
    timestamp: string;
    ok: boolean;
    latency: number | null;
    error: string | null;
  }>;
}
```

**验收标准**：
- [ ] GET `/api/network/status` 返回当前网络状态
- [ ] 返回最近 10 次心跳探测记录
- [ ] 状态数据可序列化为 JSON

#### FR-4: 心跳生命周期管理

**描述**：心跳定时器随服务启动/停止，避免内存泄漏。

**规则**：
- 服务启动时自动启动心跳
- 服务停止时清理定时器
- 心跳探测使用轻量级请求，不占用 LLM 调用配额

**验收标准**：
- [ ] 服务启动后心跳自动开始
- [ ] 服务停止时定时器正确清理
- [ ] 心跳探测不影响正常 LLM 调用队列

### 2.2 非功能需求

#### NFR-1: 性能

- 心跳探测使用最小 token（16 maxTokens）
- 心跳间隔不低于 15 秒（避免 API 限流）
- 心跳逻辑不应阻塞其他请求

#### NFR-2: 可靠性

- 断网判定容忍单次抖动（连续 3 次失败）
- 恢复判定快速响应（连续 1 次成功）
- 心跳失败不影响系统其他功能

#### NFR-3: 可观测性

- 心跳探测结果记录日志
- 网络状态变化记录 warn/error 日志
- 可通过 API 查询状态

## 3. 技术约束

### 3.1 架构约束

- 基于现有 `connectivity.ts` 的 `testPlainConnection` 扩展
- 事件发布使用现有 `novelEventBus`
- 心跳定时器使用 Node.js `setInterval`

### 3.2 依赖约束

- 依赖：REQ-7040（API失败自动重试 — 重试逻辑在心跳探测时复用）
- 被依赖：REQ-7052（后台任务管理器 — 需要订阅网络状态事件暂停/恢复任务）

### 3.3 数据约束

- 网络状态仅在内存中维护（不持久化）
- 心跳探测记录仅保留最近 10 次（环形缓冲）

## 4. 验收标准

### 4.1 功能验收

- [ ] 心跳按配置间隔自动执行
- [ ] 断网/恢复状态正确切换
- [ ] 事件正确发布到 EventBus
- [ ] API 接口返回正确的网络状态
- [ ] 心跳生命周期与服务一致

### 4.2 性能验收

- [ ] 心跳探测延迟 < 5 秒（单次）
- [ ] 心跳逻辑 CPU 占用可忽略
- [ ] 内存占用 < 1MB（心跳相关）

### 4.3 测试验收

- [ ] 单元测试覆盖率 > 80%
- [ ] 集成测试通过
- [ ] 边界用例覆盖（断网/恢复/抖动场景）

## 5. 风险评估

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|----------|
| 心跳探测被限流 | 误判为断网 | 低 | 使用最小 token，间隔 >= 15 秒 |
| 状态判定过于敏感 | 频繁状态切换 | 中 | 连续失败 3 次才判定断网 |
| 定时器内存泄漏 | 服务内存增长 | 低 | 服务停止时正确清理 |
| LLM 提供商临时不可用 | 误判为断网 | 中 | 连续 3 次失败才判定，避免单次误判 |

## 6. 工作量评估

- **开发时间**：0.3 天
- **测试时间**：0.2 天
- **总计**：0.5 天

## 7. 优先级

- 直接影响长篇小说创作的无人值守场景

- 技术风险低，基于现有 connectivity 扩展
