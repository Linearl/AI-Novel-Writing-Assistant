# REQ-7047: 进度可视化

## 1. 需求背景

### 1.1 问题描述

当前系统在执行章节生成时，用户只能看到最终结果，无法实时了解生成进度。对于需要生成100+章的长篇小说，用户不知道当前执行到第几章、预计还需要多久完成，导致无法合理安排时间，也缺乏对系统状态的感知。

### 1.2 现状分析

**已有功能**：
- 章节任务状态管理（pending/running/completed/failed）
- 检查点机制（可恢复执行）
- 批量任务队列（REQ-7046，支持排队执行）

**缺失功能**：
- ❌ 实时进度显示
- ❌ 当前章节指示
- ❌ 预计剩余时间计算
- ❌ 进度可视化界面

### 1.3 目标用户

所有使用系统进行长篇小说创作的用户，特别是：
- 创作100+章长篇的用户
- 使用批量任务队列的用户
- 需要监控生成进度的用户

## 2. 需求定义

### 2.1 功能需求

#### FR-1: 实时进度显示

**描述**：实时显示当前生成进度百分比。

**规则**：
- 进度百分比：已完成章节数 / 总章节数 * 100%
- 每秒更新一次
- 进度条可视化显示

**验收标准**：
- [ ] 进度百分比实时更新
- [ ] 进度条平滑动画
- [ ] 进度准确（无跳变）

#### FR-2: 当前章节指示

**描述**：显示当前正在生成的章节编号和标题。

**规则**：
- 显示当前章节编号（如"第15章"）
- 显示章节标题（如果已生成）
- 显示当前状态（生成中/暂停/完成）

**验收标准**：
- [ ] 正确显示当前章节编号
- [ ] 显示章节标题
- [ ] 显示当前状态

#### FR-3: 预计剩余时间

**描述**：根据历史执行速度估算剩余时间。

**规则**：
- 基于最近10章的平均执行时间估算
- 显示格式："预计还需 X小时Y分钟"
- 执行速度变化时动态调整

**验收标准**：
- [ ] 估算时间合理（误差<30%）
- [ ] 显示格式清晰
- [ ] 速度变化时动态调整

#### FR-4: 进度可视化界面

**描述**：提供独立的进度监控界面。

**规则**：
- 显示整体进度条
- 显示当前批次进度（如果是批量任务）
- 显示已用时间
- 显示预计剩余时间
- 显示失败任务数量

**验收标准**：
- [ ] 界面布局清晰
- [ ] 所有必要信息都显示
- [ ] 响应式设计（适配不同屏幕）

### 2.2 非功能需求

#### NFR-1: 性能

- 进度更新延迟<1秒
- 界面渲染流畅（60fps）

#### NFR-2: 可观测性

- 进度数据可通过API查询
- 支持WebSocket实时推送（可选）

#### NFR-3: 兼容性

- 不影响现有章节生成功能
- 与批量任务队列（REQ-7046）集成

## 3. 技术约束

### 3.1 架构约束

- 前端使用React组件实现
- 后端提供进度API
- 复用现有的任务状态管理

### 3.2 依赖约束

- 依赖：REQ-7046（批量任务队列，提供进度数据源）
- 被依赖：无

### 3.3 数据约束

- 进度数据实时计算，无需持久化
- 历史执行时间可缓存

## 4. 验收标准

### 4.1 功能验收

- [ ] 进度百分比实时更新
- [ ] 当前章节正确显示
- [ ] 剩余时间估算合理
- [ ] 界面布局清晰

### 4.2 性能验收

- [ ] 进度更新延迟<1秒
- [ ] 界面渲染流畅

### 4.3 测试验收

- [ ] 单元测试覆盖率>80%
- [ ] 集成测试通过
- [ ] UI测试通过

## 5. 风险评估

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|----------|
| 进度计算不准确 | 用户困惑 | 中 | 基于历史数据估算 + 动态调整 |
| 界面卡顿 | 用户体验差 | 低 | 优化渲染 + 虚拟列表 |
| WebSocket连接失败 | 无法实时更新 | 低 | 降级到轮询模式 |

## 6. 工作量评估

- **开发时间**：1天
- **测试时间**：0.5天
- **总计**：1.5天

## 7. 优先级

**P1** - 高优先级

**理由**：
- 直接影响用户体验
- 是批量任务功能的必要补充
- 技术复杂度低，可快速实现
