# REQ-7047: 进度可视化 — 决策日志

## 决策1: 实时更新方式

**决策日期**：2026-07-11

**背景**：前端如何获取实时进度数据？

**选项**：
- **方案A**: 轮询（每秒请求一次API）
- **方案B**: WebSocket（服务端主动推送）
- **方案C**: Server-Sent Events（SSE）

**决策**：✅ 方案A（轮询）

**理由**：
1. 实现简单，无需额外依赖
2. 1秒轮询间隔足够（进度更新不频繁）
3. 与现有架构兼容
4. 可配置轮询频率，平衡实时性和性能

**影响**：
- 前端实现setInterval轮询
- 后端提供Progress API
- 轮询频率可配置（默认1秒）

## 决策2: 进度计算基准

**决策日期**：2026-07-11

**背景**：进度百分比应该基于什么计算？

**选项**：
- **方案A**: 基于章节数（已完成/总章节）
- **方案B**: 基于时间（已用时间/预计总时间）
- **方案C**: 基于任务数（已完成任务/总任务数）

**决策**：✅ 方案A（基于章节数）

**理由**：
1. 用户最关心"完成了多少章"
2. 章节数是明确的度量单位
3. 时间估算可能不准确，不适合做进度基准
4. 与用户心智模型一致

**影响**：
- 进度 = 已完成章节数 / 总章节数 * 100%
- 剩余时间单独估算

## 决策3: 剩余时间估算方法

**决策日期**：2026-07-11

**背景**：如何估算剩余时间？

**选项**：
- **方案A**: 基于最近N章的平均时间
- **方案B**: 基于所有历史章节的平均时间
- **方案C**: 使用指数加权移动平均（EWMA）

**决策**：✅ 方案A（基于最近10章的平均时间）

**理由**：
1. 最近的数据更能反映当前执行速度
2. 避免早期慢速数据影响估算
3. 实现简单，逻辑清晰
4. 10章是合理的窗口大小

**影响**：
- 使用最近10章的平均执行时间
- 无历史数据时使用默认值（每章2分钟）

## 决策4: 进度显示位置

**决策日期**：2026-07-11

**背景**：进度信息应该显示在哪里？

**选项**：
- **方案A**: 独立页面
- **方案B**: 嵌入到现有页面（如队列管理页面）
- **方案C**: 浮动面板（始终可见）

**决策**：✅ 方案B（嵌入到队列管理页面）

**理由**：
1. 与批量任务队列（REQ-7046）集成
2. 用户在同一页面查看队列和进度
3. 避免页面跳转，体验更好
4. 复用现有的页面布局

**影响**：
- 在队列管理页面添加ProgressVisualization组件
- 组件可复用（未来可嵌入其他页面）

## 决策5: 数据更新频率

**决策日期**：2026-07-11

**背景**：进度数据应该多久更新一次？

**选项**：
- **方案A**: 每秒更新（高实时性）
- **方案B**: 每5秒更新（低服务器压力）
- **方案C**: 可配置（默认1秒）

**决策**：✅ 方案C（可配置，默认1秒）

**理由**：
1. 不同用户对实时性要求不同
2. 默认1秒足够大多数场景
3. 可调整以平衡实时性和性能
4. 避免过度轮询导致服务器压力

**影响**：
- refreshInterval参数可配置
- 默认1000ms

## 决策6: 失败任务显示

**决策日期**：2026-07-11

**背景**：失败任务应该如何显示？

**选项**：
- **方案A**: 仅显示失败数量
- **方案B**: 显示失败数量 + 详情列表

**决策**：✅ 方案B（显示数量 + 详情）

**理由**：
1. 用户需要知道哪些任务失败了
2. 详情列表便于排查问题
3. 可点击查看详情
4. 与批量任务队列的失败处理集成

**影响**：
- 显示失败任务数量
- 可展开查看失败详情（章节号、错误信息）
