# REQ-7061: 中文本地化 — 决策日志

## 决策1: i18n 框架 vs 集中常量

**决策日期**：2026-07-14

**背景**：中文化应该引入 i18n 框架（如 react-i18next）还是仅用集中常量管理？

**选项**：
- **方案A**: 引入 react-i18next / i18next（标准 i18n 方案）
- **方案B**: 集中式常量文件 + `getContextGroupLabel()` 函数
- **方案C**: 直接在各处硬编码中文字符串

**决策**：✅ 方案B

**理由**：
1. 当前仅有中文需求，无多语言切换需求
2. i18n 框架引入成本高于收益
3. 集中常量文件简单直接，维护成本低
4. 后续如需多语言，可从常量文件迁移

**影响**：
- 新增 `contextGroupLabels.ts` 集中管理标签
- 标签变更只需修改一处
- 不引入新依赖

## 决策2: LLM 对中文标签的理解

**决策日期**：2026-07-14

**背景**：prompt 上下文块标签切换为中文后，LLM 的理解能力是否会受影响？

**选项**：
- **方案A**: 保持英文标签（确保 LLM 理解）
- **方案B**: 切换为中文（面向用户可读性优先）
- **方案C**: 双语标签（如 "章节任务 / Chapter Task"）

**决策**：✅ 方案B

**理由**：
1. 主流 LLM（GPT-4、Claude、Gemini）对中文理解能力充足
2. 上下文块标签是结构标识，LLM 主要关注内容而非标签文本
3. 简洁的中文标签提升 prompt 可读性
4. 双语标签增加不必要的 token 消耗

**影响**：
- 所有块标签切换为中文
- 无需双语兼容
- Token 消耗可能微降（中文标签通常更短）

## 决策3: toListBlock 空兜底值

**决策日期**：2026-07-14

**背景**：`toListBlock()` 在输入为空时应该返回什么？

**选项**：
- **方案A**: 返回空字符串 `""`
- **方案B**: 返回 `"无"`
- **方案C**: 不输出该块（跳过空块）

**决策**：✅ 方案B

**理由**：
1. 空字符串在 prompt 中无意义，LLM 可能忽略该块
2. 显式的"无"告知 LLM 该信息已确认为空，而非遗漏
3. 与上游实现一致
4. 跳过空块可能导致上下文结构不完整

**影响**：
- `toListBlock()` 空输入返回"无"
- prompt 中空块会显示"无"而非空白

| 空兜底值 | 方案B："无" | 显式告知 LLM 信息为空 |
