# REQ-7059: Prompt 模板系统 — 决策日志

## D1: Token 语法格式

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-07-14 |
| 决策人 | AI |
| 决策 | 采用 Mustache 风格双大括号 token 语法：`{{context.xxx}}`、`{{input.xxx}}`、`{{slot.xxx}}` |
| 原因 | 上游已验证该语法简洁直观；与 Mustache/Handlebars 生态兼容；正则解析简单可靠 |
| 影响 | 需要实现 TOKEN_PATTERN 和 BRACE_TOKEN_PATTERN 两个正则 |
| 备选方案 | 方案B: `${context.xxx}` 模板字符串 — 与 JS 语法冲突；方案C: `<context:xxx>` XML 风格 — 可读性差 |

---

## D2: 编译诊断策略

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-07-14 |
| 决策人 | AI |
| 决策 | 编译失败返回诊断对象而不抛异常，blocking 诊断阻止保存 |
| 原因 | 诊断是用户体验的关键部分；不抛异常确保编译器始终可用；blocking 机制确保模板安全性 |
| 影响 | 调用方需要检查 diagnostics 而非 try-catch |
| 备选方案 | 方案B: 编译失败抛异常 — 用户体验差；方案C: 无诊断 — 无法定位问题 |

---

## D3: 版本存储方式

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-07-14 |
| 决策人 | AI |
| 决策 | 每次保存创建新版本记录（versionNo 递增），模板 JSON 序列化存储 |
| 原因 | 版本化支持回滚和对比；序列化存储简单直接；上游已验证该模式 |
| 影响 | 新增 PromptTemplateVersion 表，数据量随保存次数线性增长 |
| 备选方案 | 方案B: diff 存储 — 实现复杂；方案C: 仅保留当前版本 — 不支持回滚 |

---

## D4: 必需上下文组定义

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-07-14 |
| 决策人 | AI |
| 决策 | 沿用上游的 10 个 WRITER_REQUIRED_CONTEXT_GROUPS |
| 原因 | 上游已验证这 10 个上下文组是章节生成的核心输入；缺少任一组都会显著降低生成质量 |
| 影响 | 用户编辑模板时不可删除这 10 个组的引用 |
| 备选方案 | 方案B: 更少的必需组 — 可能遗漏关键上下文；方案C: 更多的必需组 — 限制灵活性 |

---

## D5: 运行时集成方式

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-07-14 |
| 决策人 | AI |
| 决策 | 在 runStructuredPrompt 调用链中插入模板解析层，对现有调用方透明 |
| 原因 | 最小化对现有代码的修改；调用方无需感知模板系统的存在 |
| 影响 | 需要修改 runStructuredPrompt 的内部实现，但接口不变 |
| 备选方案 | 方案B: 调用方显式调用模板系统 — 侵入性大 |

---

## D6: 覆盖粒度

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-07-14 |
| 决策人 | AI |
| 决策 | 覆盖粒度为 novelId + promptId，每本小说每个 prompt 独立覆盖 |
| 原因 | 不同小说需要不同策略，同一小说的不同 prompt 也可能需要不同定制 |
| 影响 | Override 表使用 @@unique([scope, novelId, promptId]) 约束 |
| 备选方案 | 方案B: 仅 novelId 粒度 — 不够灵活；方案C: 全局覆盖 — 不支持小说级定制 |

---

## 总结

| 决策 | 选择 | 核心理由 |
| ---- | ---- | -------- |
| Token 语法 | `{{context.xxx}}` 双大括号 | 上游验证，生态兼容 |
| 编译诊断 | 返回诊断 + blocking 阻止 | 用户体验 + 安全 |
| 版本存储 | 递增版本号 + JSON 序列化 | 简单可靠，支持回滚 |
| 必需组 | 沿用上游 10 个组 | 上游验证核心性 |
| 运行时集成 | 透明插入调用链 | 最小化侵入 |
| 覆盖粒度 | novelId + promptId | 灵活且合理 |
