# REQ-7056: 跨章节人物一致性 — 决策日志

## 决策1: 角色状态存储方式

**决策日期**：2026-07-11

**背景**：CharacterState应该如何存储？

**选项**：
- **方案A**: 独立数据库表（character_states），每章一条记录
- **方案B**: JSON字段存储在角色表中
- **方案C**: 单条记录覆盖更新（只保留最新状态）

**决策**：✅ 方案A

**理由**：
1. 独立表支持历史查询（按章节查看状态变化）
2. 结构化字段便于查询和分析
3. 不增加角色表的字段膨胀
4. 支持后续的矛盾检测和评分计算

**影响**：
- 新增character_states表
- 每章生成后写入一条记录
- 查询时需要按章节筛选

## 决策2: 矛盾检测方法

**决策日期**：2026-07-11

**背景**：如何检测人物特征矛盾？

**选项**：
- **方案A**: 纯LLM检测（语义分析）
- **方案B**: 纯规则检测（硬编码规则）
- **方案C**: 规则+LLM双层检测

**决策**：✅ 方案C

**理由**：
1. 规则检测快速准确（如身高数值矛盾），覆盖硬矛盾
2. LLM检测深层语义（如性格不一致），覆盖软矛盾
3. 双层检测提高准确率，降低漏检率
4. 规则检测成本低，LLM检测按需调用

**影响**：
- 规则检测实现简单，无需LLM调用
- LLM检测需要设计prompt和schema
- 两层结果合并去重

## 决策3: 检测时机

**决策日期**：2026-07-11

**背景**：矛盾检测应该在什么时候执行？

**选项**：
- **方案A**: 章节生成后立即检测（同步）
- **方案B**: 章节生成后异步检测（后台任务）
- **方案C**: 定期批量检测

**决策**：✅ 方案B

**理由**：
1. 不阻塞章节生成流程
2. LLM检测需要时间，异步执行更合适
3. 利用REQ-7052的后台任务管理
4. 检测结果异步推送通知

**影响**：
- 集成到REQ-7052的后台任务系统
- 检测完成后通过SSE推送结果
- 章节发布不等待检测完成

## 决策4: 矛盾严重度分级

**决策日期**：2026-07-11

**背景**：如何分级矛盾的严重度？

**选项**：
- **方案A**: 二级（硬矛盾/软矛盾）
- **方案B**: 三级（严重/一般/轻微）
- **方案C**: 连续评分（0-1严重度）

**决策**：✅ 方案A

**理由**：
1. 二级分类足够区分关键问题和可接受偏差
2. 实现简单，判定规则清晰
3. 硬矛盾直接冲突（如性别改变），软矛盾模糊不一致（如性格偏移）
4. 后续可扩展为三级

**影响**：
- 硬矛盾：直接冲突，必须修复
- 软矛盾：模糊不一致，建议修复
- 评分计算时硬矛盾扣分更多

## 决策5: LLM检测Prompt设计

**决策日期**：2026-07-11

**背景**：如何设计矛盾检测的LLM prompt？

**选项**：
- **方案A**: 单次prompt检测所有维度
- **方案B**: 分维度多次prompt检测
- **方案C**: 分角色多次prompt检测

**决策**：✅ 方案A

**理由**：
1. 单次prompt减少LLM调用次数
2. 所有维度在同一个上下文中分析更准确
3. 输出结构化结果，解析简单
4. 成本和时间最优

**影响**：
- Prompt需要包含完整的历史状态和新章节内容
- 输出schema包含所有矛盾维度
- 上下文长度可能较大（多角色长章节）

| 检测时机 | 方案B：异步后台执行 | 不阻塞生成流程 |
