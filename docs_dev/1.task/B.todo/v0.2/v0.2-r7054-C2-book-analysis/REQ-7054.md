# REQ-7054: Book Analysis 拆书系统

## 1. 需求背景

### 1.1 问题描述

用户在创作长篇小说前，往往需要分析已有作品（参考书、对标作品、灵感来源）来提炼写作策略。当前系统缺乏系统化的书籍分析能力，用户只能手动阅读和记录，效率低下且容易遗漏关键信息。需要一个 AI 驱动的拆书工作台，自动从导入文档中提取多维度分析结果。

### 1.2 现状分析

**已有功能**：
- 小说项目管理（创建、编辑、删除）
- 角色管理（基本信息、关系、弧光）
- 世界观设定管理
- RAG 检索基础能力

**缺失功能**：
- 文档导入与分段处理
- AI 自动分段分析（情节、人物、世界观、主题、风格）
- 角色 12 维画像提取
- 角色外貌追踪与变化分析
- 角色肖像生成
- 分析结果的结构化存储与浏览
- 拆书结果发布到小说项目

### 1.3 目标用户

- 需要分析对标作品的网文作者
- 希望从参考书中提炼写作策略的新手作者
- 需要系统化整理阅读笔记的创作者

## 2. 需求定义

### 2.1 功能需求

#### FR-1: 文档导入与分段

**描述**：支持导入 TXT/DOCX/PDF 格式的文档，系统自动按章节或段落分段。

**规则**：
- 支持 TXT、DOCX、PDF 三种格式
- 自动识别章节边界（按标题、空行、数字编号）
- 支持手动调整分段范围
- 分段后显示段落预览
- 每个分析项目可包含多个源文档

**验收标准**：
- [ ] 可导入 TXT/DOCX/PDF 文件
- [ ] 自动识别章节边界并分段
- [ ] 支持手动调整分段范围（SourceRangePicker）
- [ ] 分段后显示段落预览

#### FR-2: AI 多维度分析

**描述**：对导入文档进行 5 大维度的 AI 分析：情节、人物、世界观、主题、风格。

**规则**：
- 每个维度独立分析，可单独触发
- 分析结果以结构化 JSON 存储
- 支持增量分析（仅分析新增段落）
- 分析过程有进度显示
- 分析结果可浏览和编辑

**验收标准**：
- [ ] 情节分析：提取核心冲突、转折点、节奏
- [ ] 人物分析：提取角色列表、性格特征、关系
- [ ] 世界观分析：提取设定、规则、地理
- [ ] 主题分析：提取核心主题、隐喻、象征
- [ ] 风格分析：提取叙事视角、语言特征、节奏

#### FR-3: 角色 12 维画像

**描述**：从分析结果中提取角色的 12 维度详细画像。

**规则**：
- 12 维度：外貌、性格、背景、动机、关系、成长弧、语言风格、行为模式、象征意义、弱点、技能、社会地位
- 每个维度有结构化字段
- 支持从多个段落聚合信息
- 画像可编辑和手动补充

**验收标准**：
- [ ] 自动提取 12 维角色画像
- [ ] 画像数据结构化存储
- [ ] 支持手动编辑和补充
- [ ] 角色候选卡展示（CharacterCandidateCard）

#### FR-4: 角色外貌追踪

**描述**：追踪角色在不同章节/段落中的外貌描述变化。

**规则**：
- 提取每次出场的外貌描述
- 记录外貌变化时间线
- 外貌术语标准化（AppearanceTermService）
- 支持外貌对比查看

**验收标准**：
- [ ] 提取每次出场的外貌描述
- [ ] 外貌变化时间线可视化
- [ ] 外貌术语标准化
- [ ] 外貌面板展示（CharacterAppearancePanel）

#### FR-5: 角色肖像生成

**描述**：基于外貌描述生成角色肖像图片。

**规则**：
- 基于外貌描述自动生成提示词
- 调用图像生成 API 生成肖像
- 支持多种风格选择
- 生成结果可保存和替换

**验收标准**：
- [ ] 基于外貌描述生成提示词
- [ ] 调用图像 API 生成肖像
- [ ] 肖像面板展示（CharacterImagePanel）
- [ ] 支持保存和替换

#### FR-6: 分析工作台

**描述**：提供完整的拆书工作台界面，支持双栏布局、导航、导出。

**规则**：
- 双栏布局：左侧导航 + 右侧内容（DualPaneLayout）
- 分段导航（ChapterNavigator）
- 章节阅读器（ChapterReader）
- 结构化摘要展示（StructuredSummary）
- 分析结果导出
- 发布到小说项目（PublishingService）

**验收标准**：
- [ ] 双栏布局正常工作
- [ ] 分段导航可跳转
- [ ] 结构化摘要展示各维度分析
- [ ] 支持导出分析结果
- [ ] 可发布到已有小说项目

#### FR-7: 预算与并发控制

**描述**：分析任务的 token 预算管理和并发控制。

**规则**：
- 预算调整对话框（BudgetAdjustDialog）
- 并发限制防止 API 过载
- 分析进度追踪（进度条）
- 任务失败重试

**验收标准**：
- [ ] 预算调整对话框可用
- [ ] 并发限制生效
- [ ] 进度追踪正常
- [ ] 失败任务可重试

### 2.2 非功能需求

#### NFR-1: 性能

- 单段分析响应时间 < 30s（LLM 调用除外）
- 支持 100MB 以内的文档导入
- 分析结果查询 < 500ms

#### NFR-2: 可靠性

- 分析任务支持断点续传
- LLM 调用失败自动重试（最多 3 次）
- 分析结果持久化存储

#### NFR-3: 可扩展性

- 分析维度可配置
- 支持自定义分析 prompt

## 3. 技术约束

### 3.1 架构约束

- 服务端采用模块化架构：`server/src/modules/bookAnalysis/`（HTTP）+ `server/src/services/bookAnalysis/`（业务逻辑）
- 前端页面：`client/src/pages/bookAnalysis/`
- Prompt 必须在 `server/src/prompting/` 注册
- 使用 Prisma 持久化分析结果

### 3.2 依赖约束

- 依赖 RAG 基础设施（文档索引）
- 依赖 LLM 调用能力（`runStructuredPrompt`）
- 依赖图像生成 API（角色肖像）
- 被依赖：小说项目可导入拆书结果

### 3.3 数据约束

- 需要新增约 7 个 Prisma 模型
- 分析结果为结构化 JSON，单条可较大

## 6. 风险评估

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|----------|
| LLM 分析质量不稳定 | 分析结果不可用 | 中 | 多轮 Prompt 优化 + 人工审核 |
| 大文档处理超时 | 用户体验差 | 中 | 分段处理 + 异步队列 |
| Prisma 模型复杂度高 | 开发周期长 | 中 | 参考上游实现，逐步迭代 |
| 图像生成 API 不稳定 | 肖像生成功能不可用 | 中 | 降级为纯文本外貌描述 |
| 上下游代码差异大 | 移植困难 | 低 | 保持架构兼容，适配层隔离 |

## 7. 工作量评估

- **开发时间**：6-8天
- **测试时间**：2天
- **总计**：8-10天

## 8. 优先级

**P2** — 高复杂度，建议后期开发

**理由**：
- 功能覆盖面广，涉及前后端大量新增代码
- 依赖 RAG 基础设施和图像生成 API
- 可借鉴上游实现，降低设计风险
- 对核心写作流程非阻塞，可作为独立模块开发
