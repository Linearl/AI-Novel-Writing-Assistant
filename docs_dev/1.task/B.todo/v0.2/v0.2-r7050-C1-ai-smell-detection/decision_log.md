# REQ-7050: AI味自动检测 — 决策日志

## 决策1: 检测方式——规则 vs LLM vs 混合

**决策日期**：2026-07-11

**背景**：AI味检测应该采用什么技术方案？

**选项**：
- **方案A**: 纯规则检测（正则+统计）
- **方案B**: 调用LLM分析AI味
- **方案C**: 规则检测为主，高分时LLM辅助确认

**决策**：方案A（纯规则检测）

**理由**：
1. 规则检测速度快（<3秒），适合实时检查
2. 无额外LLM调用成本
3. AI味特征（高频词、句式模式）可通过规则有效捕捉
4. 预留LLM辅助接口，后续可升级

**影响**：
- 检测模块完全本地运行，无外部依赖
- 词汇库可动态更新
- 检测精度依赖词汇库和规则质量

## 决策2: 评分模型——维度设计

**决策日期**：2026-07-11

**背景**：AI味检测应该包含哪些维度？

**选项**：
- **方案A**: 仅词汇维度
- **方案B**: 词汇 + 句式两个维度
- **方案C**: 词汇 + 句式 + 情感 + 多样性四个维度

**决策**：方案C（四个维度）

**理由**：
1. 词汇维度捕捉用词AI味（基础）
2. 句式维度捕捉句式单调性（节奏）
3. 情感维度捕捉情感模式化（深度）
4. 多样性维度捕捉重复和单调（整体）
5. 多维度综合比单一维度更准确

**影响**：
- 四个独立检测器，各自独立评分
- 加权平均得到综合评分
- 各维度权重可配置

## 决策3: 评分等级划分

**决策日期**：2026-07-11

**背景**：AI味评分如何划分等级？

**选项**：
- **方案A**: 三级（低/中/高）
- **方案B**: 四级（自然/轻微/明显/重度）
- **方案C**: 五级（极低/低/中/高/极高）

**决策**：方案B（四级）

**理由**：
1. 三级太粗，区分度不够
2. 五级太细，等级间差异不明显
3. 四级与业务逻辑对应：
   - 自然（0-30）：无需处理
   - 轻微（31-60）：可接受，可选优化
   - 明显（61-80）：需关注，调整参数
   - 重度（81-100）：需重写

**影响**：
- 评分等级与自动调整策略对应
- 等级阈值可配置
- 报告中显示等级标签

## 决策4: 超阈值调整策略

**决策日期**：2026-07-11

**背景**：AI味评分超阈值时如何调整？

**选项**：
- **方案A**: 仅通知用户
- **方案B**: 自动调整生成参数
- **方案C**: 分级处理——轻度调整参数，重度触发重写

**决策**：方案C（分级处理）

**理由**：
1. 轻度问题通过参数调整即可改善
2. 重度问题需要彻底重写
3. 分级处理平衡了效率和效果
4. 与REQ-7049（自动重新生成）协同

**影响**：
- 轻度（61-80）：提高温度参数
- 重度（81-100）：触发REQ-7049重新生成
- 在AiSmellConfig中配置阈值和策略

## 决策5: 词汇库管理方式

**决策日期**：2026-07-11

**背景**：AI味检测词汇库如何管理？

**选项**：
- **方案A**: 硬编码在代码中
- **方案B**: 数据库存储，支持动态更新
- **方案C**: 配置文件存储

**决策**：方案B（数据库存储）

**理由**：
1. 数据库存储支持运行时更新
2. 可按类别组织（词汇/情感/内心独白）
3. 支持用户自定义扩展
4. 与项目现有配置管理方式一致

**影响**：
- 新增 `AiSmellDictionary` 数据库表
- 提供词汇库管理API
- 默认预置常用AI词汇

## 决策6: 检测结果存储位置

**决策日期**：2026-07-11

**背景**：AI味检测结果如何存储？

**选项**：
- **方案A**: 独立的AI味检测结果表
- **方案B**: 存储在ChapterQualityReport的dimensions JSON字段
- **方案C**: 两者都存

**决策**：方案B（复用质量报告存储）

**理由**：
1. AI味检测是质量检查的一个维度
2. 与REQ-7048的质量报告结构一致
3. 避免数据冗余
4. 查询时一次获取完整质量报告

**影响**：
- 在QualityReport.dimensions中新增 'aiSmell' 维度
- 查询质量报告时可一并获取AI味检测结果
- 不需要额外的数据库表

| 评分等级 | 四级 | 与业务逻辑对应 |
