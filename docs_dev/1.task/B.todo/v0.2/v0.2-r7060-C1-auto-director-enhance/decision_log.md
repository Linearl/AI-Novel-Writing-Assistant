# REQ-7060: Auto-Director 增强 — 决策日志

## 决策1: 创建向导实现方式

**决策日期**：2026-07-14

**背景**：5 步创建向导应直接移植上游实现还是重新设计？

**选项**：
- **方案A**: 直接移植上游 `autoDirector/` 目录下 9 个文件，适配接口差异
- **方案B**: 参考上游思路但重新设计 UI 和状态管理
- **方案C**: 使用现成的多步表单库（如 react-step-wizard）

**决策**：✅ 方案A

**理由**：
1. 上游实现完整（9 个文件、约 2000 行），经验证可用
2. 本项目与上游使用相同技术栈（React + TypeScript）
3. 直接移植成本最低，只需适配 API 调用和路由差异
4. 避免引入新依赖

**影响**：
- 需要适配本项目的 API 层和路由结构
- 上游 UI 样式可能需要调整以适配本项目设计系统

## 决策2: 通知轮询策略

**决策日期**：2026-07-14

**背景**：桌面通知的轮询频率和策略如何选择？

**选项**：
- **方案A**: 固定 15 秒轮询（简单）
- **方案B**: 基于 SSE/WebSocket 实时推送（复杂但精确）
- **方案C**: 15 秒轮询 + 页面不可见时降频（平衡）

**决策**：✅ 方案C

**理由**：
1. SSE/WebSocket 需要服务端改造，成本过高
2. 固定轮询在页面不可见时浪费资源
3. 页面不可见时增加通知频率确保用户回来时能收到最新状态
4. 与上游实现一致

**影响**：
- 轮询间隔可配置，默认 15 秒
- 需要处理 `visibilitychange` 事件

## 决策3: 散文检测实现方式

**决策日期**：2026-07-14

**背景**：散文质量检测应该用 regex 还是 LLM？

**选项**：
- **方案A**: 纯 regex 检测（零成本、无 LLM 依赖）
- **方案B**: LLM 语义分析（精确但成本高）
- **方案C**: regex 初筛 + LLM 复核（平衡）

**决策**：✅ 方案A

**理由**：
1. 上游实现为 regex-based，经验证效果可接受
2. regex 检测零 LLM 成本，可高频调用
3. 9 种问题码中大部分适合模式匹配
4. 后续可迭代加入 LLM 复核

**影响**：
- 检测结果可能存在误报
- 需要可调阈值控制灵敏度

## 决策4: 待审自动提升的双重确认机制

**决策日期**：2026-07-14

**背景**：待审超时自动提升如何防止误操作？

**选项**：
- **方案A**: 仅系统校验（自动推进）
- **方案B**: 双重确认（系统校验 + 用户确认）
- **方案C**: 仅通知用户手动处理

**决策**：✅ 方案B

**理由**：
1. 纯自动推进风险太高，可能推进未准备好审核的章节
2. 仅通知不推进无法解决流水线停滞问题
3. 双重确认平衡了自动性和安全性
4. 与上游实现一致

**影响**：
- 需要 EventBus 支持请求-响应模式
- 用户确认超时（7 天）自动拒绝

## 决策5: 冲突等级曲线与 Tension Curve 的关系

**决策日期**：2026-07-14

**背景**：FR-5 冲突等级曲线与 REQ-7063 Tension Curve 功能重叠，如何处理？

**选项**：
- **方案A**: 合并为一个任务包
- **方案B**: FR-5 仅实现后端约束注入，UI 由 REQ-7063 实现
- **方案C**: 两个任务包独立实现，各自包含完整前后端

**决策**：✅ 方案B

**理由**：
1. REQ-7063 专注于可视化交互（React Flow + d3-shape），功能更完整
2. FR-5 的核心价值在于后端约束注入，前端 UI 可复用 REQ-7063
3. 避免重复实现
4. FR-5 先行提供基础版本，REQ-7063 迭代增强

**影响**：
- FR-5 实现基础的曲线数据模型和后端约束注入
- 前端可视化延后到 REQ-7063
- 两者共享数据模型定义
