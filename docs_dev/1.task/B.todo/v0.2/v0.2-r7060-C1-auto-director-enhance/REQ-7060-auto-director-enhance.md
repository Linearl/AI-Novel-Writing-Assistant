# REQ-7060: Auto-Director 增强

## 1. 需求背景

### 1.1 问题描述

当前 Auto-Director 在以下维度存在体验缺口：
- 创建新小说项目时缺少引导式向导，用户需手动逐步配置
- 自动导演执行过程中用户离开页面后无法及时获知进度和暂停状态
- 待审核章节长时间无人处理导致流水线停滞
- 缺少散文级质量检测能力，仅有章节级审校
- 冲突强度（conflictLevel）为硬编码值，缺少用户可调控的曲线编辑
- 待审状态切换时上下文不足，审校结果质量受限
- 资源上下文散落在多处，组装逻辑不统一

### 1.2 现状分析

**已有功能**：
- Auto-Director 基本流程（规划→生成→审校→循环）
- 手动创建小说项目的表单页面
- 章节级审校（chapter acceptance）
- EventBus 事件系统

**缺失功能**：
- ❌ 分步创建向导（5 步引导式配置）
- ❌ 桌面通知（Browser Notification API 集成）
- ❌ 待审自动提升（超时自动推进 + 双重确认）
- ❌ 散文质量检测器（regex-based 问题检测）
- ❌ 冲突等级曲线编辑（用户锚定硬约束）
- ❌ 待审上下文增强注入
- ❌ 资源上下文统一重构

### 1.3 目标用户

所有使用 Auto-Director 功能的用户，特别是：
- 首次使用系统的新用户（需要创建向导引导）
- 长时间挂机运行自动导演的用户（需要通知能力）
- 追求散文质量的作者（需要质量检测反馈）

## 2. 需求定义

### 2.1 功能需求

#### FR-1: 5 步创建向导

**描述**：将小说项目创建过程拆分为 5 个步骤，以向导式 UI 引导用户完成配置。

**步骤**：
1. **基础设置**（StageBasicSetup）— 书名、类型、简介、目标字数
2. **创意想法**（StageIdea）— 核心创意、灵感来源、风格偏好
3. **世界观与风格**（StageWorldStyle）— 世界观设定、写作风格、参考作品
4. **候选方案**（StageCandidates）— AI 生成多个方案供用户选择
5. **模型运行**（StageModelRun）— 选择模型、确认配置、启动创建

**状态机**：
- 支持步骤间前进/后退
- 每步验证通过后才可前进
- 支持从候选方案中选择并应用

**验收标准**：
- [ ] 5 步向导 UI 完整实现，步骤间可前进/后退
- [ ] 每步有独立的验证逻辑
- [ ] 候选方案步骤支持多选和预览
- [ ] 创建完成后自动跳转到小说编辑页
- [ ] 状态机支持保存/恢复中间状态

#### FR-2: 桌面通知系统

**描述**：利用 Browser Notification API，在自动导演执行过程中向用户发送桌面通知，确保用户及时获知任务状态变化。

**通知场景**：
- 自动导演任务暂停（pause）
- 自动导演任务恢复（resume）
- 自动导演任务完成（complete）
- 自动导演任务出错（error）

**轮询机制**：
- 每 15 秒轮询一次任务状态
- 状态变化时触发通知
- 页面不可见时（visibilitychange）增加通知频率

**验收标准**：
- [ ] 首次使用时请求通知权限
- [ ] 任务暂停/恢复/完成/出错时发送桌面通知
- [ ] 15 秒轮询间隔可配置
- [ ] 页面不可见时通知不丢失
- [ ] 通知点击后跳转到对应页面

**上游参考**：
- `client/src/lib/autoDirectorPauseNotifications.ts` — 通知管理器（114 行）

#### FR-3: 待审自动提升

**描述**：当章节处于待审核状态超过一定时间（默认 14 天），自动提升审核优先级并触发双重确认机制。

**规则**：
- 待审超过 14 天触发自动提升
- 提升前需双重确认（用户确认 + 系统校验）
- 确认后自动推进审核流程

**验收标准**：
- [ ] 14 天超时可配置
- [ ] 超时后触发提升事件
- [ ] 双重确认流程完整
- [ ] 确认后自动推进审核

**上游参考**：
- `server/src/services/novel/state/PendingReviewAutoPromotionService.ts` — 594 行

#### FR-4: 散文质量检测器

**描述**：基于 regex 的散文质量问题检测，识别 9 种常见问题码，为审校提供细粒度反馈。

**问题码类型**：
- `repetitiveSentence` — 重复句式
- `excessiveAdjective` — 过多形容词
- `passiveVoiceOveruse` — 被动语态过度使用
- `tellingInsteadShowing` — 以叙述代替展示
- `dialogueTag` — 对话标签问题
- `pacing` — 节奏问题
- `vaguePronoun` — 模糊代词
- `clichePhrase` — 陈词滥调
- `infoDump` — 信息倾泻

**验收标准**：
- [ ] 9 种问题码全部实现
- [ ] 每种问题码有对应 regex 模式
- [ ] 检测结果包含问题位置和建议
- [ ] 支持自定义检测规则配置

**上游参考**：
- `server/src/services/novel/runtime/proseQuality/ProseQualityDetector.ts` — 450 行

#### FR-5: 冲突等级曲线

**描述**：支持用户通过交互式曲线编辑器设定章节冲突强度（0-100），作为自动导演生成的硬约束。

**验收标准**：
- [ ] 冲突等级可按章节设置（0-100）
- [ ] 曲线可视化展示
- [ ] 作为硬约束注入生成上下文
- [ ] 支持批量调整（卷级别）

**上游参考**：
- `client/src/components/tensionCurve/` — 10 文件（与其他任务包共享）

#### FR-6: 待审上下文注入

**描述**：章节进入待审状态时，自动注入增强上下文（前文摘要、角色状态、世界变化等），提升审校质量。

**验收标准**：
- [ ] 待审上下文包含前文摘要
- [ ] 包含当前角色状态
- [ ] 包含世界设定变更
- [ ] 上下文大小可配置

#### FR-7: 资源上下文重构

**描述**：将散落在多处的资源上下文组装逻辑统一重构为独立模块，消除重复代码。

**验收标准**：
- [ ] 统一的资源上下文组装接口
- [ ] 原有散落调用点迁移完毕
- [ ] 组装结果与重构前一致

### 2.2 非功能需求

#### NFR-1: 可复用性

- 创建向导的步骤系统应支持后续扩展新步骤
- 通知系统应支持通知其他模块的状态变化
- 散文质量检测器应支持插件式扩展新问题码

#### NFR-2: 兼容性

- 现有小说创建流程不被破坏
- 现有审校流程保持兼容
- 现有事件系统保持兼容

## 3. 技术约束

### 3.1 架构约束

- 前端向导基于 React 组件 + 自定义 Hook 实现
- 通知使用 Browser Notification API（需浏览器支持）
- 散文检测基于 regex，不依赖 LLM 调用
- 资源上下文重构需保持现有调用方接口不变

### 3.2 依赖约束

- 无前置依赖
- FR-5（冲突等级曲线）与 REQ-7063 共享上游参考代码

### 3.3 数据约束

- 创建向导中间状态存储在客户端 localStorage
- 待审自动提升状态存储在 NovelChapter 表
- 散文检测结果不持久化（按需计算）

## 5. 风险评估

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|----------|
| Browser Notification API 兼容性 | 通知功能在部分浏览器不可用 | 中 | 降级为页面内通知 |
| 创建向导步骤过多导致用户流失 | 创建完成率下降 | 低 | 支持跳步和保存进度 |
| 散文检测 regex 误报率高 | 用户体验下降 | 中 | 可调阈值 + 用户反馈优化 |
| 待审自动提升误触发 | 章节状态异常 | 低 | 双重确认机制 |
