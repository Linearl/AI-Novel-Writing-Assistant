# v0.2 架构知识库

> 本文档记录v0.2所有任务包的完整技术背景、关键决策和实现方案。
> 子代理创建任务包时请参考本文档，确保理解一致。

## 1. 版本背景

### 1.1 v0.1已完成

- 70+任务包完成（架构修复、易用性功能）
- 《玫瑰窃贼》验证中（30章，目标：用户能独立完成AI味不重的小说）
- v0.1定稿标准：160章小说，干预<20次

### 1.2 v0.2核心目标

**主题**：流程自动化 + 质量控制

**核心价值**：让用户能自动完成160章+的网文创作，系统自治处理大部分问题，用户只需启动任务和最终审核。

**验收标准**：
- 160章小说，干预<20次
- 创建新小说项目验证（非《玫瑰窃贼》）
- 核心：系统自动处理异常和恢复

### 1.3 调研结果

通过4个子代理调研现有代码，发现：

**已有功能**（70%-85%实现）：
- 三层任务队列体系（DirectorTaskQueue + TaskDispatcher）
- 完整的断点续传机制（DirectorRuntimeCheckpoint）
- 两层审计系统（light + full audit）
- 风格引擎（29个文件，含检测+重写）
- Toast通知、外部渠道通知（钉钉/企微）
- Health端点、启动门控

**缺失功能**（需要v0.2补齐）：
- LLM API传输层重试
- 通用模型切换
- 浏览器原生通知
- 环境变量启动校验
- 跨章节人物一致性检查
- AI味趋势追踪

## 2. 任务包详细信息

### M1: 异常自动处理（4天）

#### REQ-7040: API失败自动重试（1.5天）⭐️ P1

**背景**：
- 当前LLM API调用失败时直接抛出错误，用户需手动重试
- 长篇小说创作（160章+）会频繁阻塞

**技术方案**：
- **实现位置**：在`invokeStructuredLlmDetailed`层添加重试（方案A）
- **重试时机**：在策略降级之后重试（先尝试所有策略，最后再重试）
- **重试逻辑**：指数退避（1s → 2s → 4s），最多3次
- **429处理**：解析Retry-After头，等待指定时间（最多60秒）
- **错误分类**：仅重试可恢复错误（网络超时、502/503、429限流）
- **配置**：重试次数可配置，存储在Provider Secret表

**关键代码位置**：
- 新增：`server/src/llm/retryHandler.ts`
- 修改：`server/src/llm/structuredInvoke.ts`
- 复用：`StructuredOutputError`分类（transport_error可重试）

**依赖**：无

**验收标准**：
- [ ] 网络超时自动重试3次
- [ ] 429限流自动解析Retry-After并重试
- [ ] 重试次数可配置
- [ ] 正常调用延迟增加<100ms

#### REQ-7042: 错误分类器（1天）⭐️ P1

**背景**：
- 当前有StructuredOutputError分类（6种），但缺少通用错误分类
- 不同错误需要不同处理（可恢复→重试，需配置→通知用户，需人工→中断）

**技术方案**：
- **分类体系**：4类错误（可恢复、需配置、需人工、系统错误）
- **分类逻辑**：基于HTTP状态码和错误码自动分类
- **处理策略**：
  - 可恢复→触发REQ-7040的重试机制
  - 需配置→显示配置修复指引
  - 需人工→显示问题描述和建议
  - 系统错误→记录日志，显示友好错误消息
- **错误码映射**：建立错误码→分类的映射表

**关键代码位置**：
- 新增：`server/src/llm/errorClassifier.ts`
- 复用：`StructuredOutputError`分类体系

**依赖**：REQ-7040

**验收标准**：
- [ ] 错误自动分类准确率>90%
- [ ] 可恢复错误自动触发重试
- [ ] 需配置错误显示清晰的修复指引

#### REQ-7049: 自动重新生成（1天）⭐️ P1

**背景**：
- 当前已有风格重写机制（PostGenerationStyleReviewRunner）
- 但缺少：完全重新生成选项、重试策略

**技术方案**：
- **重试策略**：
  - 第1次：直接重新生成（相同参数）

- **重试限制**：最多3次，3次失败后通知用户
- **参数调整**：温度从0.7→0.5，top_p从0.9→0.95
- **复用**：基于现有的StyleRewriteService扩展

**关键代码位置**：

- 新增：`server/src/services/novel/quality/autoRegenerationService.ts`

**依赖**：REQ-7048

**验收标准**：
- [ ] 质量不合格自动重新生成
- [ ] 最多重试3次
- [ ] 每次重试调整参数
- [ ] 3次失败后通知用户

#### REQ-7056: 跨章节人物一致性（1.5天）⭐️ P1 新增

**背景**：
- 长篇小说需要维护人物状态时间线
- 当前缺少人物状态的结构化存储和跨章节检测

**技术方案**：
- **数据模型**：CharacterState表（每章每人物一个快照）
- **状态提取**：每章生成后用LLM提取人物特征
- **矛盾检测**：对比前后章节的人物状态，检测变化是否合理
- **存储方案**：快照机制（每章一个快照），不是链表结构
- **快照设计**：
  - 第N章开始前的状态 = 第N-1章结束后的状态（无需重复存储）
  - 每章结束后保存一个快照
  - 便于查询和UI展示

**关键代码位置**：
- 新增：`server/src/services/novel/character/characterStateService.ts`
- 新增：`server/src/services/novel/character/characterStateExtractor.ts`
- 修改：Prisma schema新增CharacterState表

**依赖**：无

**验收标准**：
- [ ] 每章生成后自动提取人物状态
- [ ] 保存到CharacterState表
- [ ] 检测相邻章节的人物矛盾
- [ ] 置信度>90%才报告为矛盾

**v0.3衔接**：
- v0.3将在v0.2基础上增加：时间线UI、状态对比、回溯编辑

```
├─ REQ-7052 (后台管理) ──→ REQ-7053 (通知)
```

## 4. 关键技术决策

### 4.1 REQ-7040: 重试机制

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 实现位置 | invokeStructuredLlmDetailed层 | 核心调用入口，易于管理 |
| 重试时机 | 策略降级之后 | 最大化成功率 |
| 429处理 | 解析Retry-After | 尊重服务端建议 |
| 重试次数 | 可配置（默认3次） | 灵活性 |
| 错误分类 | 网络错误+429 | 平衡覆盖率和安全性 |
| 配置存储 | Provider Secret表 | 与现有配置管理一致 |

### 4.2 REQ-7056: 人物状态存储

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 存储方案 | 快照机制（每章一个） | 查询简单、实现直观、便于UI展示 |
| 存储开销 | ~1.6MB/160章 | 完全可以接受 |
| 数据模型 | chapter_number + character_id | 唯一标识一个快照 |
| 状态提取 | LLM提取+置信度评分 | 灵活、准确 |
| 矛盾检测 | 置信度>90%才报告 | 避免误报 |

### 4.3 REQ-7051: 一致性检测

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 检测维度 | 时间>人物>逻辑>空间 | 按优先级排序 |
| 误报策略 | 宁可漏报，不要误报 | 用户体验优先 |
| 检测粒度 | 每章检测 | 早发现早修复 |
| 处理方式 | 高优先级矛盾触发重新生成 | 确保质量 |
