# REQ-3003 小说项目对话存档

## 基本信息

| 字段 | 内容 |
| --- | --- |
| 需求编号 | REQ-3003 |
| 优先级 | P2 |
| 版本 | v0.1 |
| 状态 | requirements_ready |
| 来源 | 用户需求——创作工作台缺少按项目维度的对话回顾能力 |

---

## 1. 背景与问题

### 现状

- 创作工作台（`NovelWorkspaceRail`）有 7 个创作流程 tab + 1 个工具 tab（版本历史），但**没有对话查看入口**
- `CreativeHub`（创作中枢）管理多线程对话，`CreativeHubThread` 可通过 `resourceBindingsJson` 松散关联 novelId
- 对话消息以 JSON 快照存储在 `CreativeHubCheckpoint.messagesJson` 中，无法按项目检索
- 旧的 `/api/chat` 没有服务端持久化，历史由前端 IndexedDB 维护

### 痛点

1. 用户在创作工作台内无法回顾该小说的历史对话——需切回 `/creative-hub` 筛选
2. 对话数据缺乏结构化归档，项目级对话不可检索、不可回溯
3. 没有按 novelId 的聚合查询 API

### 不改会怎样

- 用户创作时丢失对话上下文，反复重复相同的指导内容
- 项目交接/复盘时无法回溯全部对话决策过程
- 对话数据随 `CreativeHubThread` 删除而丢失，无独立归档

---

## 2. 目标与范围

### 2.1 目标

1. 在 `NovelWorkspaceRail` 新增"对话存档"tab（工具区），入口可见且可访问
2. 实现按 novelId 聚合查询所有关联对话线程的 API
3. 对话归档采用 JSONL 格式（与 Claude Code 一致），支持按时间段/线程查看
4. 对话存档页面提供基本的浏览和搜索能力

### 2.2 In Scope

**后端**：
- 新增 API：`GET /api/novels/:id/conversations` — 按 novelId 聚合对话列表
- 新增 API：`GET /api/novels/:id/conversations/export` — 导出 JSONL 归档
- `CreativeHubThread` 与 `Novel` 的关联查询逻辑
- 对话归档 JSONL 写入与读取

**前端**：
- `NovelWorkspaceRail` 新增"对话存档"tab
- 新建对话存档页面组件（列表视图 + 详情视图）
- JSONL 导出下载功能

**基础设施**：
- 无额外基础设施需求

### 2.3 Out of Scope

- 不改变 `CreativeHub` 现有数据模型和 API
- 不在全局 `Sidebar` 中添加对话 tab（仅创作工作台）
- 不改变旧的 `/api/chat` 路由
- 不做实时对话（那是 Creative Hub 的职责），本需求只做**已归档对话的只读查看**
- 不做全文搜索（v0.1 阶段仅按线程/时间浏览，全文搜索后续迭代）

---

## 3. 需求详情

### 3.1 导航入口

**WHEN** 用户进入创作工作台（`/novels/:id/edit`），**THE SYSTEM SHALL** 在 `NovelWorkspaceRail` 的工具区渲染"对话存档" tab，点击后路由到 `/novels/:id/conversations`。

### 3.2 对话列表

**WHEN** 用户访问对话存档页面，**THE SYSTEM SHALL** 展示该小说关联的全部对话线程列表，每条显示：
- 线程标题
- 创建时间
- 最近更新时间
- 消息数量
- 状态（活跃/已归档）

### 3.3 对话详情

**WHEN** 用户点击某条对话线程，**THE SYSTEM SHALL** 展示该线程的全部消息记录（只读），按时间顺序渲染。

### 3.4 JSONL 归档

**WHEN** 用户点击"导出归档"按钮，**THE SYSTEM SHALL**：
- 将该小说全部对话线程的消息序列化为 JSONL 格式
- 每行一个消息对象：`{"threadId", "runId", "messageIndex", "role", "content", "timestamp", "metadata"}`
- 触发浏览器下载，文件名为 `<novel-title>-conversations-<date>.jsonl`

### 3.5 对话归档存储策略

**WHEN** 对话产生（通过 Creative Hub 或 Chat API），**THE SYSTEM SHALL** 异步将消息追加写入项目级 JSONL 归档文件，确保对话数据独立于 CreativeHubThread 生命周期。

---

## 4. 验收标准

- [ ] `NovelWorkspaceRail` 渲染"对话存档"tab，点击可跳转至 `/novels/:id/conversations`
- [ ] 对话列表 API 正确返回按 novelId 聚合的线程列表
- [ ] 对话详情页正确渲染全部消息（只读），按时间序
- [ ] JSONL 导出格式正确：每行一个 JSON 消息对象，可被标准 JSONL parser 解析
- [ ] 类型检查通过：`pnpm typecheck`
- [ ] 单元测试通过：`pnpm test`
- [ ] 路由测试通过：`pnpm --filter @ai-novel/server test:routes`

---

## 5. 风险与约束

| 风险 | 缓解 |
| ---- | ---- |
| `CreativeHubThread.resourceBindingsJson` 是自由格式 JSON，novelId 可能缺失 | API 端做防御处理，缺失 novelId 绑定的线程不报错而是跳过 |
| 大量对话线程（>1000）时列表性能 | v0.1 只做分页，不做虚拟滚动 |
| JSONL 文件过大导致下载超时 | 按线程分文件下载，或流式生成 |
| CreativeHubCheckpoint.messagesJson 格式可能变更 | 归档时做 schema 验证，不合法消息记录 warning 跳过 |

---

## 6. 关联与边界

- 与 `CreativeHub`（`/creative-hub`）的边界：Creative Hub 负责**实时对话**，本需求负责**已归档对话的只读查看**
- 依赖 `CreativeHubThread` + `CreativeHubCheckpoint` 数据模型（`server/src/prisma/schema.prisma`）
- 与 `NovelWorkspaceRail` 的边界：新增 tab 不改变现有 7 个创作流程 tab 的行为
- 与 `Sidebar` 的边界：不在全局 Sidebar 中添加入口
