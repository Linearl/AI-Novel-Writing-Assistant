# REQ-2022 决策日志

## DEC-001: 扩展 REQ-2021 接口 vs 新建独立模块

| 项目 | 内容 |
| ---- | ---- |
| 决策日期 | 2026-06-28 |
| 决策者 | 自行决定 |
| 决策内容 | 扩展 REQ-2021 的 `DirectorDebugLogEntry` 接口和 `saveDirectorDebugLog` 函数，而非新建独立的日志模块 |
| 决策理由 | REQ-2021 已建立日志条目结构、磁盘写入管道和配置开关。在其基础上扩展字段是最小变更路径，避免重复的目录创建、文件写入、清理逻辑。新字段为可选（默认空数组），向后兼容 |
| 备选方案 | 方案 B：新建独立的 `directorEnhancedDebugLogger.ts`，独立的文件写入管道。拒绝原因：会导致两个并行的日志文件管道，增加维护成本，且断路器触发时需要写两个文件 |
| 影响 | `DirectorDebugLogEntry` 接口新增 5 个字段，`saveDirectorDebugLog` 函数签名不变（新增字段有默认值） |

## DEC-002: 内存缓冲区 vs 临时文件

| 项目 | 内容 |
| ---- | ---- |
| 决策日期 | 2026-06-28 |
| 决策者 | 自行决定 |
| 决策内容 | 使用内存环形缓冲区实时记录调试数据，断路器触发时 flush 到磁盘 |
| 决策理由 | 1) 内存访问延迟（~ns）远低于磁盘 I/O（~ms），不影响主流程性能；2) 50 条 LLM 调用记录的内存占用约 5-10MB，在可控范围内；3) fire-and-forget 场景下内存缓冲区最自然，无需管理临时文件生命周期 |
| 备选方案 | 方案 B：每次 LLM 调用后立即追加写入临时 JSON 文件，断路器触发时 rename。拒绝原因：频繁的磁盘 I/O 会影响主流程延迟，且临时文件管理（进程崩溃时清理）增加复杂度 |
| 影响 | 新增 `directorDebugBuffer.ts` 模块；进程异常退出时缓冲区数据丢失（可接受，因为仅用于调试） |

## DEC-003: 环形缓冲区大小

| 项目 | 内容 |
| ---- | ---- |
| 决策日期 | 2026-06-28 |
| 决策者 | 自行决定 |
| 决策内容 | LLM 调用历史缓冲区上限设为 50 条 |
| 决策理由 | 一次章节修复周期内通常涉及 3-10 次 LLM 调用。50 条覆盖了绝大多数场景（含重试），同时限制内存占用在 10MB 以内。内容快照和修复尝试无硬上限（数量级通常 < 20 条） |
| 备选方案 | 方案 B：无上限。拒绝原因：极端情况下内存占用不可控 |
| 影响 | 超过 50 次 LLM 调用时最早的记录会被丢弃，但实际场景中极少发生 |

## DEC-004: 详细级别设计

| 项目 | 内容 |
| ---- | ---- |
| 决策日期 | 2026-06-28 |
| 决策者 | 自行决定 |
| 决策内容 | 三级详细级别：minimal（仅 token/耗时）、standard（摘要）、verbose（完整） |
| 决策理由 | 不同场景对调试信息的需求不同：生产环境仅需 token 用量追踪（minimal），开发环境需要摘要信息定位问题（standard），深度调试需要完整 prompt 分析模型行为（verbose）。三级设计覆盖常见需求，且实现简单 |
| 备选方案 | 方案 B：两级（standard/verbose）。拒绝原因：缺少生产环境友好的最小化选项 |
| 影响 | 新增 2 个环境变量配置项；默认 "standard" 平衡调试能力和存储开销 |

## DEC-005: 临时性数据清理时机

| 项目 | 内容 |
| ---- | ---- |
| 决策日期 | 2026-06-28 |
| 决策者 | 自行决定 |
| 决策内容 | 修复成功（chapterStatus 变为 completed）时立即丢弃缓冲区数据；修复失败/断路器触发时保留数据直到 flush 到磁盘 |
| 决策理由 | 修复成功意味着调试数据不再有诊断价值，及时释放缓冲区空间。修复失败是需要调试的关键场景，必须保留完整数据链 |
| 备选方案 | 方案 B：统一在 flush 时清理，不区分成功/失败。拒绝原因：成功修复的调试数据无用但占用缓冲区，且断路器触发前的中间数据会丢失 |
| 影响 | 需要在章节状态变更处额外调用 `discardOnSuccess` |

## DEC-006: 保留时间 vs 文件数量上限

| 项目 | 内容 |
| ---- | ---- |
| 决策日期 | 2026-06-28 |
| 决策者 | 自行决定 |
| 决策内容 | 采用基于时间的保留策略（默认 7 天），替代 REQ-2021 的文件数量上限（100 个） |
| 决策理由 | 时间保留更符合调试场景的实际使用模式——开发者通常在几天内分析调试日志，超期日志价值递减。文件数量上限在高频触发场景下可能过早删除有价值的日志 |
| 备选方案 | 方案 B：保留文件数量上限策略。拒绝原因：高频触发时有价值的近期日志可能被删除 |
| 影响 | REQ-2021 的 `enforceMaxLogFiles` 保留（向后兼容），新增 `enforceRetention` 策略 |

## DEC-007: 原子写入

| 项目 | 内容 |
| ---- | ---- |
| 决策日期 | 2026-06-28 |
| 决策者 | 自行决定 |
| 决策内容 | 日志文件写入采用原子写入策略（先写临时文件，再 rename 替换目标文件） |
| 决策理由 | 并发场景下（多个断路器同时触发）可能导致 JSON 文件写入不完整。原子写入保证文件要么是旧的完整版本，要么是新的完整版本，不会出现半写状态 |
| 备选方案 | 方案 B：直接 writeFile。拒绝原因：写入过程中进程崩溃或并发写入可能产生损坏文件 |
| 影响 | 写入路径增加一步 rename 操作，对性能影响可忽略 |

## DEC-008: 日志文件分离策略

| 项目 | 内容 |
| ---- | ---- |
| 决策日期 | 2026-06-28 |
| 决策者 | 用户要求 |
| 决策内容 | 将调试日志分离为两个文件：简要日志（`*_brief.json`）和详细日志（`*_detail.json`），简要日志通过 `detailLogPath` 字段引用详细日志的相对路径 |
| 决策理由 | 1) 简要日志文件小（<50KB），便于快速浏览和筛选断路器触发场景；2) 详细日志文件可能较大（数 MB），仅在需要深入分析时读取，避免加载不必要的数据；3) 分离存储支持后续可能的日志聚合分析工具（只扫描简要日志索引）；4) 简要日志中的 `summary` 字段提供关键统计，无需打开详细日志即可判断问题严重程度 |
| 备选方案 | 方案 B：单文件存储所有信息。拒绝原因：单文件在详细级别为 verbose 时可能非常大（含完整 prompt/completion），加载和解析效率低；方案 C：数据库存储。拒绝原因：增加数据库依赖，且调试日志本质是临时性文件数据 |
| 影响 | 1) `saveDirectorDebugLog` 函数重构为 `saveDirectorDebugBrief` + `saveDirectorDebugDetail` 两个函数；2) 简要日志新增 `detailLogPath` 和 `summary` 字段；3) 断路器触发时需调用两个写入函数；4) 文件清理逻辑需同时处理两类文件 |
