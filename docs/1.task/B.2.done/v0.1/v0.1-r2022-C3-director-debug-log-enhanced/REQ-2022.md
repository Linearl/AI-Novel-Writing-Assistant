# REQ-2022 — 质量修复阶段调试日志增强

## 1. 目标

增强 auto-director 质量修复阶段的调试日志能力，记录模型的完整操作和输出（LLM 调用历史、章节内容演变、修复过程详情），使开发者能够事后完整重现修复过程、分析模型行为，快速定位修复失败的根因。

## 2. 范围

### 2.1 包含（Scope In）

1. **LLM 调用历史记录**: 记录每次 LLM 调用的 prompt（输入）、模型的完整输出（completion）、工具调用的详细信息（工具名称、参数、返回值）、token 使用量、耗时
2. **章节内容演变记录**: 在关键节点（draft、repair、acceptance）保存章节内容快照，记录每次变更的原因
3. **质量修复过程记录**: 审计结果、每次修复尝试的详细信息、修复成功/失败的原因
4. **临时性数据管理**: 修复成功（chapterStatus 变为 completed）时自动丢弃调试数据；仅在修复失败或异常时保留
5. **性能和存储优化**: 内存缓冲区实时记录，断路器触发时批量写入磁盘；限制缓冲区大小（最多 50 次 LLM 调用）；自动清理过期调试日志（7 天后自动删除）
6. **配置开关**: `DIRECTOR_DEBUG_LOG_ENABLED`（总开关）、`DIRECTOR_DEBUG_LOG_DETAIL_LEVEL`（minimal/standard/verbose）、`DIRECTOR_DEBUG_LOG_RETENTION_HOURS`（保留时间，默认 168 小时）
7. **日志文件分离**: 简要日志（`*_brief.json`）和详细日志（`*_detail.json`）分离存储，简要日志包含状态快照和详细日志的相对路径引用，便于快速浏览和深入分析

### 2.2 不包含（Scope Out）

1. 不实现实时日志流式推送（WebSocket/SSE）
2. 不实现日志的 Web UI 查看界面
3. 不修改现有的数据库事件日志（`DirectorAutomationLedgerEventService`）行为
4. 不修改断路器状态机逻辑或阈值
5. 不修改 REQ-2021 已实现的基础断路器快照功能，仅在其基础上扩展

## 3. 非目标（Non-Goals）

- 这不是通用的日志框架（logging framework），仅服务于 director debug 场景
- 这不是审计合规日志，不需要防篡改或签名
- 不替代已有的 ledger event 数据库持久化

## 4. EARS 验收条目

### E1. LLM 调用历史记录（Event-driven）

E1.1. **当** 质量修复流程中发生任何 LLM 调用时，系统应将该调用的完整输入（prompt）、输出（completion）、工具调用详情、token 使用量和耗时记录到内存缓冲区。
E1.2. **当** 内存缓冲区中的 LLM 调用记录超过 50 条时，系统应丢弃最早的记录（环形缓冲策略），保证内存使用有界。

### E2. 章节内容演变记录（State-change-driven）

E2.1. **当** 章节内容在 draft、repair、acceptance 任一关键节点发生变更时，系统应保存该节点的内容快照（含变更原因）到内存缓冲区。
E2.2. **当** 章节内容快照被保存时，应包含：`nodeType`（draft/repair/acceptance）、`content`（章节文本）、`reason`（变更原因）、`timestamp`、`chapterVersion`（如有）。

### E3. 质量修复过程记录（Event-driven）

E3.1. **当** 审计（audit）完成后，系统应将审计结果摘要（通过/不通过、发现的问题列表）记录到调试上下文。
E3.2. **当** 每次修复尝试开始和结束时，系统应记录：修复策略、输入内容摘要、输出内容摘要、成功/失败状态、失败原因（如有）。

### E4. 断路器触发时批量写入（Event-driven）

E4.1. **当** 断路器触发（`stopAutoExecutionForCircuitBreaker` 被调用）时，系统应将内存缓冲区中的所有调试数据分离为两个 JSON 文件写入磁盘：简要日志（`*_brief.json`）和详细日志（`*_detail.json`）。
E4.2. **当** 简要日志被写入时，应包含：基础信息（taskId, novelId, chapterId, timestamp）、autoExecution 快照、circuitBreaker 快照、`summary`（摘要统计）、`detailLogPath`（指向详细日志的相对路径，如 `./2026-06-28T10-30-00Z_task-uuid_detail.json`）。
E4.3. **当** 详细日志被写入时，应包含：`llmCallHistory`（LLM 调用历史数组）、`contentSnapshots`（章节内容演变数组）、`repairAttempts`（修复尝试数组）、`auditResults`（审计结果数组）。
E4.4. **当** 详细日志文件名生成时，应与简要日志使用相同的时间戳和 taskId 前缀，仅后缀不同（`_brief.json` vs `_detail.json`）。

### E5. 临时性数据管理（State-driven）

E5.1. **当** 修复成功（章节 chapterStatus 变为 `completed`）时，系统应自动丢弃该章节对应的内存调试数据，释放缓冲区空间。
E5.2. **当** 章节修复失败或异常（断路器触发、超时等）时，系统应保留该章节的调试数据直到写入磁盘。

### E6. 配置开关（Ubiquitous）

E6.1. **当** `DIRECTOR_DEBUG_LOG_ENABLED` 设为 `false` 时，系统应完全禁用调试日志记录功能，不产生任何额外内存开销或磁盘写入。
E6.2. **当** `DIRECTOR_DEBUG_LOG_DETAIL_LEVEL` 设为 `minimal` 时，系统仅记录 LLM 调用的 token 使用量和耗时，不记录 prompt/completion 内容。
E6.3. **当** `DIRECTOR_DEBUG_LOG_DETAIL_LEVEL` 设为 `standard`（默认）时，系统记录 LLM 调用的摘要信息（prompt 前 500 字符、completion 前 500 字符）+ 完整的工具调用信息 + token/耗时。
E6.4. **当** `DIRECTOR_DEBUG_LOG_DETAIL_LEVEL` 设为 `verbose` 时，系统记录 LLM 调用的完整 prompt 和 completion。
E6.5. **当** `DIRECTOR_DEBUG_LOG_RETENTION_HOURS` 被设置时（默认 168 = 7 天），系统应在每次写入新日志后检查并删除超过保留时间的旧日志文件。

### E7. 错误处理（Ubiquitous）

E7.1. **当** 日志写入失败时（磁盘满、权限不足等），系统应静默忽略错误，不中断断路器停止流程。
E7.2. **当** 日志目录不存在时，系统应在首次写入时自动创建。

## 5. 验收标准

- [ ] 质量修复流程中每次 LLM 调用的 prompt、completion、工具调用、token 用量、耗时均被记录
- [ ] 章节内容在 draft/repair/acceptance 节点的变更被记录为快照（含变更原因）
- [ ] 修复尝试的详细过程（策略、输入输出摘要、成败、原因）被记录
- [ ] 断路器触发时生成两个 JSON 文件：简要日志（`*_brief.json`）和详细日志（`*_detail.json`）
- [ ] 简要日志包含 `detailLogPath` 字段，指向详细日志的相对路径
- [ ] 简要日志包含 `summary` 字段，包含 LLM 调用次数、token 消耗、修复尝试次数等统计
- [ ] 详细日志包含 llmCallHistory、contentSnapshots、repairAttempts、auditResults 字段
- [ ] 章节修复成功后自动丢弃对应的内存调试数据
- [ ] `DIRECTOR_DEBUG_LOG_ENABLED=false` 时无额外内存开销和磁盘写入
- [ ] `DIRECTOR_DEBUG_LOG_DETAIL_LEVEL=minimal` 时仅记录 token/耗时
- [ ] `DIRECTOR_DEBUG_LOG_DETAIL_LEVEL=verbose` 时记录完整 prompt/completion
- [ ] 超过 50 次 LLM 调用时旧记录被丢弃（环形缓冲）
- [ ] 超过保留时间的旧日志文件被自动清理
- [ ] 日志写入失败不中断断路器停止流程
- [ ] 单元测试覆盖：缓冲区记录、级别控制、临时数据清理、磁盘写入、文件分离

## 6. 风险与约束

| 风险 | 缓解措施 |
|------|----------|
| 完整 prompt/completion 存储导致内存占用过高 | 缓冲区大小限制 50 条 + verbose 模式下 prompt/completion 仍受单条上限约束（如 100KB） |
| 并发写入导致 JSON 文件损坏 | 单文件写入 + 原子写入（先写临时文件再 rename） |
| 调试日志写入性能影响主流程 | 异步缓冲 + fire-and-forget 写入，不阻塞主流程 |
| 多个修复任务并发导致缓冲区数据交叉 | 缓冲区按 taskId 隔离，每个任务独立的缓冲区实例 |
| 日志文件包含敏感小说内容 | 明确为开发调试用途，仅本地存储，.gitignore 排除 |

## 7. 关联与边界

- 与 REQ-2021 的关系：REQ-2021 提供基础的断路器快照保存框架，本需求在其基础上扩展完整的调试数据采集能力。REQ-2021 的接口（`saveDirectorDebugLog`、`DirectorDebugLogEntry`）将被扩展而非替换
- 与 `DirectorAutomationLedgerEventService` 的边界：ledger 为数据库持久化事件，本需求为磁盘 JSON 调试快照，两者互补不冲突
- 与 LLM 客户端层的边界：本需求不修改 LLM 客户端，仅通过 wrapper 或事件监听采集调用信息
