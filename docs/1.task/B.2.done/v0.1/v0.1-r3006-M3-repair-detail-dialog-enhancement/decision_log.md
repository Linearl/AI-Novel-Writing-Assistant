# REQ-3006: 决策日志

## 决策 1: Token 统计数据源

**决策点**: Token 统计从哪里获取？

| 方案 | 优点 | 缺点 |
|------|------|------|
| A: 复用 task.tokenUsage | 无需新增 API | 可能不精确 |
| B: 新增专用 API | 精确 | 需要后端开发 |

**选择**: 方案 A（复用 task.tokenUsage）

**理由**:
1. task 表已有 tokenUsage 字段
2. 修复任务的 token 消耗会记录到该字段
3. 无需新增 API，降低实施成本

**日期**: 2026-06-28

## 决策 2: Diff 库选择

**决策点**: 使用哪个 diff 库？

| 库 | 体积 | 功能 | 维护状态 |
|----|------|------|----------|
| diff | ~5KB | 基础 diff 算法 | 活跃 |
| react-diff-viewer | ~50KB | React 组件，开箱即用 | 停止维护 |
| react-diff-viewer-continued | ~50KB | react-diff-viewer 维护版 | 活跃 |
| Monaco Editor diff | ~2MB | VS Code 体验 | 活跃 |

**选择**: `react-diff-viewer-continued`

**理由**:
1. 开箱即用的 React 组件
2. 支持 split view 和 unified view
3. 活跃维护，社区支持好
4. 体积可接受

**日期**: 2026-06-28

## 决策 3: Token 刷新策略

**决策点**: Token 统计如何刷新？

| 方案 | 实时性 | 性能影响 |
|------|--------|----------|
| A: 轮询（3s） | 中 | 低 |
| B: WebSocket/SSE | 高 | 中 |
| C: 手动刷新 | 低 | 无 |

**选择**: 方案 A（轮询 3s）

**理由**:
1. 实现简单，无需后端支持
2. 3s 间隔对用户体验影响小
3. 修复完成后停止轮询，避免资源浪费

**日期**: 2026-06-28

## 决策 4: Diff 按钮位置

**决策点**: "查看 Diff" 按钮放在哪里？

| 方案 | 位置 | 用户发现性 |
|------|------|------------|
| A: 步骤 6 头部 | 高 | 好 |
| B: 弹窗内 | 中 | 一般 |
| C: 侧边栏 | 低 | 差 |

**选择**: 方案 A（步骤 6 头部）

**理由**:
1. 与"保存"、"撤销"按钮纵向排列，位置显眼
2. 修复完成后用户自然会查看步骤 6
3. 符合用户操作习惯

**日期**: 2026-06-28

## 决策 5: 版本 Tab 命名

**决策点**: 版本 Tab 如何命名？

| 方案 | 示例 | 清晰度 |
|------|------|--------|
| A: 版本 N | 版本 1, 版本 2 | 中 |
| B: 时间戳 | 14:32:05 | 低 |
| C: 版本 N + 时间 | 版本 1 (14:32) | 高 |

**选择**: 方案 A（版本 N）

**理由**:
1. 简洁明了
2. 用户主要关心版本顺序，不是具体时间
3. 避免 UI 过于拥挤
