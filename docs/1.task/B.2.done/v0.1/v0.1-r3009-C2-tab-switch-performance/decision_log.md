# REQ-3009 决策日志

## D1: 采用 staleTime 缓存而非组件级缓存

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-07-03 |
| 决策人 | AI |
| 决策 | 通过 React Query 的 `staleTime` 配置实现条件查询缓存，而非引入组件级缓存或状态管理 |
| 原因 | `staleTime` 是 React Query 原生能力，零额外依赖，精确控制 query 级缓存时效。组件级缓存（如 Zustand 持久化）需要手动同步，维护成本高。 |
| 影响 | 所有条件查询受益，切换 tab 后 30 秒内不触发网络请求 |
| 备选方案 | Zustand 持久化 tab 状态 — 需要手动管理缓存失效，与 React Query 缓存层重复 |

## D2: invalidateNovelDetail 精细化而非全量失效

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-07-03 |
| 决策人 | AI |
| 决策 | 按 tab 类型只失效相关 query key，而非当前全量失效 15 个 key |
| 原因 | 全量失效导致切换 tab 后大量不相关 query 重新请求，浪费带宽且增加响应时间。按 tab 精细化失效只刷新必要数据。 |
| 影响 | 切换 tab 时网络请求数从 15+ 降至 2-3 |
| 备选方案 | 保持全量失效但添加请求去重 — 不能减少实际请求，只减少并发 |
