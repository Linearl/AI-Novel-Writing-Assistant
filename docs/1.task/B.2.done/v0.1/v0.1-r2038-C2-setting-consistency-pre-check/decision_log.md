# REQ-2038 决策日志

## 决策记录

### D1: 分阶段实施策略

**日期**: 2026-07-03
**决策**: 分阶段做，本次先做 LLM 内置校验
**备选方案**:
1. 一次性实现完整校验（LLM 校验 + 变更 diff + 影响分析 + 自动修复）
2. 分阶段做：先 LLM 内置校验，后续迭代加变更 diff + 影响分析
3. 纯规则引擎校验（不依赖 LLM）

**理由**:
- 方案 1 工作量过大，不适合单个任务包
- 方案 2 渐进式交付，快速获得价值
- 方案 3 规则引擎灵活性不足，无法处理语义层面的矛盾
- LLM 内置校验是核心能力，优先实现

**风险**: LLM 校验结果可能存在误报
**缓解**: 允许用户忽略已知/误报的矛盾项

---

### D2: 校验结果存储格式

**日期**: 2026-07-03
**决策**: 校验结果存储为结构化 JSON（矛盾项 + 严重级别 + 修复建议）
**备选方案**:
1. 结构化 JSON
2. Markdown 报告
3. 数据库存储

**理由**:
- JSON 便于程序解析和前端渲染
- 结构化数据支持过滤、排序、统计
- 文件系统存储与项目现有架构一致（feedback 系统也用文件系统）
- 数据库增加复杂度，当前数据量不需要

---

### D3: Auto-Director 集成策略

**日期**: 2026-07-03
**决策**: 校验不阻断 auto-director 主流程
**理由**:
- 与项目已有的 Auto-Director 质量门策略一致：校验结果为可见警告，非阻断错误
- 只有明确的 `stop_for_replan`、不可恢复的生成失败才能停止全局链
- 设定一致性问题是质量改进项，不是阻断项

---

### D4: 校验 Prompt 治理

**日期**: 2026-07-03
**决策**: 校验 prompt 在 `prompting/` 注册为 PromptAsset
**理由**:
- 遵循项目 Prompt Governance 规则
- prompt 集中管理，便于版本控制和复用
- 禁止在 service 文件中内联 prompt

---

### D5: 一键修复实现方式

**日期**: 2026-07-03
**决策**: 一键修复通过 LLM 根据建议修改对应设定字段，修复后重新校验
**备选方案**:
1. LLM 修复 + 重新校验
2. 纯前端字段替换（无 LLM）
3. 弹出编辑器让用户手动修改

**理由**:
- LLM 可以理解上下文，修复更自然
- 重新校验确保修复后不引入新矛盾
- 方案 2 无法处理复杂矛盾
- 方案 3 体验不如一键修复

**风险**: LLM 修复可能引入新矛盾
**缓解**: 修复后自动重新校验，展示完整报告
