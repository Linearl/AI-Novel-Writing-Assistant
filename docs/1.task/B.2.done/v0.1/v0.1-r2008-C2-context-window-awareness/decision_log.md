# REQ-2008 决策日志

## D1: 1M 上下文 Checkbox 默认勾选

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-06-27 |
| 决策人 | AI |
| 决策 | 设置 UI 使用 Checkbox（默认勾选 1M），取消勾选时为 256K |
| 原因 | 本项目以长篇小说写作为主，涉及大量章节内容、角色设定、世界观等上下文注入，256K 模型几乎不会被使用。1M 是绝大多数场景的首选，设为默认可减少用户配置步骤。 |
| 影响 | 新建路由默认 1M；用户可手动取消勾选降级到 256K |
| 备选方案 | 256K/1M Radio 双选 — 对本项目而言 256K 几乎无用，增加不必要的选择步骤 |

## D2: 行级截断而非 AI 摘要

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-06-27 |
| 决策人 | AI |
| 决策 | v1 使用行级截断（复用现有 summarizeContextBlock），不引入 AI 摘要 |
| 原因 | AI 摘要需要额外 LLM 调用，增加延迟和成本。行级截断零成本，且项目已有实现。v1 先验证压缩策略的正确性，后续再考虑 AI 摘要。 |
| 影响 | 截断后的上下文可能丢失重要信息（行级截断不理解语义） |
| 备选方案 | LLM 摘要 — 更智能但增加延迟和成本 |

## D3: 预留 30% 给输出

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-06-27 |
| 决策人 | AI |
| 决策 | `effectiveBudget = min(assetBudget, contextWindow * 0.7)` |
| 原因 | 模型的上下文窗口 = 输入 + 输出。需要为输出预留空间。30% 是安全余量，确保输出 token 不会因输入过大而被截断。 |
| 影响 | 实际可用输入上下文为窗口的 70% |

## D4: 四阶段渐进式压缩

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-06-27 |
| 决策人 | AI |
| 决策 | 压缩按优先级执行：丢弃低优先级块 → 截断消息 → 截断 optional 块 → 截断 required 块 |
| 原因 | 渐进式压缩保留最重要的信息。低优先级块最先丢弃，required 块最后处理。每步后检查是否达标，避免过度压缩。 |
| 影响 | 压缩过程可能需要多轮计算（但 token 估算是 O(n) 的，性能可接受） |

## D5: 压缩阈值 85% / 目标 40%

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-06-27 |
| 决策人 | AI |
| 决策 | 触发阈值 85%，压缩目标 40% |
| 原因 | 85% 给出足够的缓冲区，避免在边界处频繁触发。40% 目标确保压缩后有充足空间继续对话，不会很快再次触发。85% - 40% = 45% 的压缩幅度足够大。 |
| 影响 | 在 85%-100% 之间不会触发压缩（正常使用区间） |
| 备选方案 | 90%/50% — 阈值太高可能来不及压缩；70%/30% — 阈值太低会频繁触发 |
