# REQ-2023: 决策日志

## 决策 1: rejectedIntent 存储方案

**决策点**: 如何存储用户拒绝意图？

| 方案 | 优点 | 缺点 |
|------|------|------|
| A: 复用 validationNotesJson | 无需数据库迁移，复用现有字段 | 需要解析特殊格式 |
| B: 新增 rejectedIntent 字段 | 类型安全，查询方便 | 需要数据库迁移 |

**选择**: 方案 A

**理由**:
1. 无需数据库迁移，降低实施风险
2. validationNotesJson 已经是 JSON 数组，扩展性强
3. 解析逻辑简单，性能影响小

**日期**: 2026-06-28

## 决策 2: 意图输入是否必填

**决策点**: 用户拒绝时，意图输入是否必填？

| 方案 | 用户体验 | 数据完整性 |
|------|----------|------------|
| A: 必填 | 可能增加用户负担 | 高 |
| B: 可选 | 灵活，降低操作门槛 | 中 |

**选择**: 方案 B（可选）

**理由**:
1. 降低操作门槛，用户可快速拒绝
2. 意图是辅助信息，非核心数据
3. 用户可在后续修复时补充

**日期**: 2026-06-28

## 决策 3: 按钮显示策略

**决策点**: "不接受此风险"按钮何时显示？

| 方案 | 覆盖范围 | 用户认知负担 |
|------|----------|--------------|
| A: 仅高风险 | 精准 | 低 |
| B: 所有风险 | 全面 | 高 |

**选择**: 方案 A（仅高风险）

**理由**:
1. 高风险是最需要用户决策的场景
2. 低风险通常自动处理，无需用户干预
3. 减少 UI 复杂度

**日期**: 2026-06-28

## 决策 4: Prompt 注入位置

**决策点**: 用户意图注入到哪个 prompt？

| 方案 | 影响范围 | 实施难度 |
|------|----------|----------|
| A: 章节修复 prompt | 精准 | 低 |
| B: 所有章节生成 prompt | 广泛 | 高 |

**选择**: 方案 A（章节修复 prompt）

**理由**:
1. 用户意图主要是指导修复，不是指导新生成
2. 影响范围可控，不会影响新章节质量
3. 实施难度低，风险小

**日期**: 2026-06-28
