# REQ-2018 — 小说项目风险管理系统

## 1. 目标

为小说项目建立全生命周期风险管理系统，覆盖风险从发现、评估、处理、回溯到导出的完整链路。

## 2. 范围

### 2.1 包含（Scope In）

1. **风险数据模型**：统一的风险记录结构，覆盖章节风险、流水线风险、质量风险、资源风险等类型
2. **风险生命周期管理**：`open → ignored/accepted/resolved`，已处理风险保留不删除
3. **风险面板**：专用风险查看面板，显示风险列表、详情、影响评估和状态
4. **延期风险回溯**：允许之前忽略/接受的风险重新打开处理，评估对下游章节的影响
5. **系统性风险评估**：未处理风险对剧情的综合影响评估与警示
6. **风险导出**：支持 md/json 两种格式导出

### 2.2 不包含（Scope Out）

1. 不实现自动风险检测机制（依赖现有流水线中的风险识别）
2. 不实现跨小说项目的风险聚合
3. 不实现风险通知推送（邮件/Webhook）
4. 不修改现有自动导演流程的质量门行为

## 3. 非目标（Non-Goals）

- 这不是实时监控/告警系统
- 这不是通用的项目管理风险工具，仅服务于小说创作场景
- 不替代已有的 `DirectorQualityRepairRisk`，而是扩展其范围

## 4. EARS 验收条目

### E1. 风险记录（Ubiquitous）

E1.1. **当** 导演流水线在质量修复/章节生成/资源检查等阶段发现风险时，系统应自动创建 `NovelRisk` 记录。  
E1.2. **当** 风险被用户标记为"忽略"、"接受"或"已修复"时，该记录应迁移至对应状态并保留全部历史信息。  
E1.3. **当** 用户查询风险列表时，系统应支持按类型、状态、严重程度、关联章节过滤。

### E2. 风险面板（Event-driven）

E2.1. **当** 用户进入步骤 7（质量修复）时，"风险面板"应作为可展开区域显示在该步骤下方。  
E2.2. **当** 用户点击风险项时，应展示该风险的完整信息（标题、描述、关联章节、影响评估、时间线）。  
E2.3. **当** 用户展开风险面板时，应能看到所有风险（含已处理）的概览摘要。

### E3. 延期风险回溯（State-driven）

E3.1. **当** 用户选择之前忽略/接受的风险并决定重新处理时，系统应将该风险状态置为 `open`（或新状态 `reopened`）。  
E3.2. **当** 回溯的风险被重新打开后，系统应自动评估并提示该风险对下游章节的潜在影响范围。  
E3.3. **当** 回溯涉及修改历史章节时，系统应展示受影响的章节列表及预计修复代价。

### E4. 风险评估面板（Optional Feature）

E4.1. **当** 存在 3 个及以上未处理的风险时，系统应在风险面板顶部显示"剧情影响警示"横幅。  
E4.2. **当** 用户点击"风险评估"按钮时，系统应综合评估所有未处理风险对剧情的潜在影响，生成评估报告。  
E4.3. **当** 影响评估判断风险可能改变关键剧情走向（如角色死亡、角色关系断裂、世界设定冲突）时，应标记为"高影响"并给予醒目警示。

### E5. 导出（Ubiquitous）

E5.1. **当** 用户点击"导出风险"时，系统应提供 md 和 json 两种格式选择。  
E5.2. **当** 选择 md 导出时，应生成结构化的 Markdown 文档，包含所有风险（含已处理）的时间线和状态。  
E5.3. **当** 选择 json 导出时，应生成完整的 JSON 数据文件，包含所有字段。

## 5. 风险与未决项

| 风险 | 缓解措施 |
|------|----------|
| 与现有 `ResourceRiskPanel` 功能重叠 | 在设计阶段评估是否扩展现有组件或在旁边新增 |
| 前端面板可能造成步骤 7 UI 过于拥挤 | 采用可折叠设计，默认不展开 |
| 回溯修改对下游章节的影响评估复杂度高 | 首版给出受影响章节列表即可，精细评估后续迭代 |

## 6. 假设

- A1: 现有导演流水线中的风险点（`buildDirectorQualityRepairRisk`）会统一迁移到新风险模型中
- A2: 风险数据存储使用现有 Prisma + SQLite/PostgreSQL 架构
- A3: 风险评估面板的首版聚焦于人工可读的警示，不引入自动化评分算法
