# REQ-7002 决策日志

## D1: 拆分策略 — 按职责分层而非按行数机械切割

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-06-28 |
| 决策人 | AI |
| 决策 | 按职责将 NovelEdit.tsx 拆分为 hooks + 子组件 + 常量，而非按行范围机械切割 |
| 备选方案 | A: 按行数均分为 4-5 个文件；B: 按标签页拆分（每个 Tab 一个文件） |
| 原因 | 按职责拆分的模块边界清晰、内聚度高，后续维护时容易定位代码。按行数切割会导致同一职责分散到多个文件，按标签页拆分无法处理共享状态和工具栏逻辑。 |
| 影响 | 需要先分析文件的功能区块和依赖关系，再确定拆分边界 |

## D2: 状态管理保持 useState/useEffect 模式

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-06-28 |
| 决策人 | AI |
| 决策 | 拆分后保持现有的 useState/useEffect 模式，不引入 Zustand 或其他状态管理库 |
| 备选方案 | A: 引入 Zustand store 管理页面级状态；B: 使用 React Context |
| 原因 | 纯结构重构不应改变状态管理模式。引入新库增加依赖和学习成本，超出本次重构范围。现有 useState 模式通过自定义 hook 提取后即可获得良好的可维护性。 |
| 影响 | 状态通过 hook 参数/返回值在组件间传递，不引入新的全局状态 |

## D3: 原文件保留为 facade 重导出

| 字段 | 内容 |
| ---- | ---- |
| 日期 | 2026-06-28 |
| 决策人 | AI |
| 决策 | NovelEdit.tsx 拆分后保留为页面入口，通过 re-export 保持对外接口不变 |
| 备选方案 | A: 直接修改所有引用方的 import 路径 |
| 原因 | 保留 facade 可以零成本保持所有现有 import 路径不变，降低拆分风险。文件路由（vite-plugin-pages）依赖特定路径下的页面文件，不能移动。 |
| 影响 | NovelEdit.tsx 最终 ~400 行，包含页面编排逻辑 + re-export |
