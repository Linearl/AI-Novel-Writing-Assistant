# REQ-2027 决策日志

## D1：提取时机 — 保存前 + 保存后均可

- **决策点**：提取只能在保存前做，还是保存后也能做
- **选项**：A) 仅保存前  B) 仅保存后  C) 保存前 + 保存后均可
- **选择**：C — 保存前 + 保存后均可
- **理由**：用户可能编辑完想先提取再保存（保存前），也可能保存后才想起来要提取（保存后）。保存时在 `saveMutation.onMutate` 中缓存 `preEditContent`，确保保存后仍有编辑前的基准文本可比。
- **日期**：2026-06-30

## D2：提交给 AI 的输入格式

- **决策点**：提交给 LLM 的是 diff 片段还是完整章节内容
- **选项**：A) 仅 diff 片段（修改部分的上下文）  B) diff 视图（修改前后两个版本的完整章节内容）
- **选择**：B — diff 视图（修改前后完整章节内容）
- **理由**：完整上下文让 LLM 理解修改意图，而非仅看孤立的修改片段。例如用户把一段描写从华丽改为朴素，如果只看修改片段可能误判为"删减"，但结合完整章节能理解是"风格统一化"。
- **日期**：2026-06-30

## D3：preEditContent 缓存时机

- **决策点**：在哪个阶段缓存编辑前的原始内容
- **选项**：A) 章节加载时缓存（初始版本）  B) saveMutation.onMutate 时缓存（保存前一刻）  C) 两者结合
- **选择**：C — 两者结合
- **理由**：章节加载时初始化 `preEditContent` 为章节内容；`saveMutation.onMutate` 时更新缓存为当前编辑内容（此时 contentDraft 即将成为新的 savedContent）。切换章节时清除缓存。这样无论是保存前还是保存后，都有正确的"编辑前"基准。
- **日期**：2026-06-30

## D4：风格 fork 后自动切换绑定

- **决策点**：新风格画像创建后是否自动切换绑定
- **选项**：A) 自动切换  B) 仅创建不切换，用户手动切换
- **选择**：A — 自动切换
- **理由**：用户在章节编辑器中选择"提取风格画像"的意图很明确——要用新的画像。如果只创建不切换，用户还需要去写法引擎页面手动操作，体验断裂。
- **日期**：2026-06-30

## D5：反 AI 规则默认应用到当前项目

- **决策点**：提取的反 AI 规则是全局基线还是仅关联到当前风格画像
- **选项**：A) 关联到当前风格画像  B) 设为全局基线  C) 让用户选择
- **选择**：A — 关联到当前风格画像
- **理由**：章节编辑器的上下文是"当前小说"，用户的修改偏好可能只适用于当前项目。如果需要设为全局基线，用户可以去反 AI 规则页面手动调整。
- **日期**：2026-06-30

## D6：版本号策略

- **决策点**：新画像的版本号如何计算
- **选项**：A) 全局自增 ID  B) 按原画像名维度递增  C) 时间戳
- **选择**：B — 按原画像名维度递增
- **理由**：同一原画像多次 fork 时，版本号连续（v1, v2, v3）便于用户追溯。用时间戳不够直观，全局 ID 无法体现来源关系。
- **日期**：2026-06-30

## D7：按钮可用时机

- **决策点**：两个按钮什么时候可点击
- **选项**：A) 有未保存修改时  B) 保存成功后  C) 两者皆可
- **选择**：C — 两者皆可
- **理由**：保存前 `isDirty && contentDraft !== savedContent` 时可用；保存后 `preEditContent !== savedContent` 时可用。两种场景都保证有 diff 视图可提交。
- **日期**：2026-06-30

## D8：Prompt 共用 vs 分离

- **决策点**：反 AI 提取和风格 fork 用同一个 prompt 还是两个独立 prompt
- **选项**：A) 一个 prompt + mode 参数  B) 两个独立 prompt
- **选择**：A — 一个 prompt + mode 参数
- **理由**：两个场景的输入相同（diff 视图 + 现有规则上下文），只是输出结构不同。共用 prompt 减少维护成本，通过 mode 参数控制输出侧重。
- **日期**：2026-06-30
