# 任务包管理指南

> **新建或完成任何任务包之前，必须阅读本文档。**

---

## 1. 任务包构成（六件套）

每个任务包目录 `v11-r{编号}-{tag}-{简称}/` 必须包含以下文件：

| 文件 | 必需 | 说明 |
| --- | --- | --- |
| `README.md` | 是 | 任务包概述，从模板创建 |
| `REQ-{reqId}-{slug}.md` | 是 | 需求说明（工作副本，可修改），slug 与目录名描述段一致 |
| `REQ-{reqId}-{slug}-original.md` | 是 | 需求说明（冻结副本，从工作副本复制） |
| `tasks.md` | 是 | 开发任务清单，含验收标准 |
| `run_result.json` | 是 | 运行状态，控制自动归档 |
| `design.md` | 否 | 技术设计文档（复杂任务需要） |
| `decision_log.md` | 否 | 决策记录（复杂任务需要） |

### run_result.json 格式

```json
{
  "taskPackage": "v11-r{编号}-{tag}-{简称}",
  "status": "pending",
  "reqId": "REQ-XXXX",
  "createdAt": "2026-01-01T00:00:00+08:00"
}
```

`status` 字段取值：

| 值 | 含义 | 归档行为 |
| --- | --- | --- |
| `pending` | 未开始 | 不归档 |
| `analysis_done` | 分析完成，待开发 | 不归档 |
| `done` / `completed` | 已完成 | **自动归档** |

## 2. 自动归档机制

### 触发条件

提交时 pre-commit hook `req-sync.mjs archive` 自动检查：

1. 扫描 `B.todo/v11/` 下所有任务包
2. 读取 `run_result.json` 的 `status` 字段
3. 若 `status` 为 `done` 或 `completed`，且六件套文件齐全 → 迁移到 `B.2.done/v11/`

### 归档流程

```text
B.todo/v11/v11-rXXXX/  →  B.2.done/v11/v11-rXXXX/
     (工作区)                  (归档区)
```

### 归档失败常见原因

| 原因 | 解决方案 |
| --- | --- |
| 缺少 REQ-{reqId}-{slug}.md | 从 REQ-{reqId}-{slug}-original.md 复制或新建 |
| 缺少 REQ-{reqId}-{slug}-original.md | 从 REQ-{reqId}-{slug}.md 复制 |
| 缺少 run_result.json | 创建文件，status 设为 done |
| 缺少 tasks.md | 创建文件，标记已完成任务 |
| run_result.json status 不是 done | 改为 `"done"` 后重新提交 |

### 手动归档

如需手动触发归档（不提交代码）：

```bash
node scripts/req-sync.mjs archive --apply
```

## 3. 编号分类规则

> 完整规范见 `docs/1.task/requirements.md` §2.1

### 速查表

| 编号段 | 分类 | 典型内容 | 示例 |
| --- | --- | --- | --- |
| **1xxx** | 架构和基础设施 | 架构设计、基础设施搭建、工具链建设 | REQ-1001 四件套协作治理 |
| **2xxx** | 核心功能开发 | 新功能、业务逻辑实现、API 开发 | REQ-2031 上传自动用途识别 |
| **3xxx** | 用户界面和体验 | UI/UX 改进、交互优化、页面重构 | REQ-3056 反馈功能优化 |
| **4xxx** | 测试和质量保证 | 测试覆盖、测试修复、E2E、QA | REQ-4011 测试文件碎片化治理 |
| **5xxx** | 文档和规范 | 文档编写、规范制定、**文档修复** | REQ-5001 docs 目录重构 |
| **6xxx** | 运维和部署 | 部署脚本、运维流程、**环境搭建**、**部署修复** | REQ-6001 单入口反向代理 |
| **7xxx** | 技术债务和重构 | 代码质量、安全修复、重构、**代码去重** | REQ-7011 超长文件拆分治理 |
| **8xxx** | 研究和调研 | 技术调研、方案评估 | — |
| **9xxx** | 未分类和临时任务 | 临时任务、兜底分类 | — |

### 分配流程

```text
1. 判断任务类型（上面的速查表）
2. 确定编号段（如：文档修复 → 5xxx）
3. 查询下一个可用编号：
   node scripts/req-sync.mjs next --category <首位数字>
   例：文档任务 → node scripts/req-sync.mjs next --category 5
4. 创建任务包目录：B.todo/v11/v11-r{编号}-{tag}-{简称}/
5. 填写六件套文件
```

### 常见误分类速查

以下场景**容易被错放入 7xxx**，请注意：

| 实际内容 | 应归编号段 | 原因 |
| --- | --- | --- |
| 文档保鲜修复 | **5xxx** | 文档工作，非代码重构 |
| 文档审计修复 | **5xxx** | 文档工作，非代码重构 |
| 部署 backport / 部署修复 | **6xxx** | 运维部署，非技术债务 |
| E2E 测试清理合并 | **4xxx** | 测试工作，非技术债务 |
| 测试覆盖增强 | **4xxx** | 测试工作，非技术债务 |

## 4. 铁律

1. **编号一经分配不可变更**（`requirements.md` §1.1），分配前务必确认分类正确
2. **禁止将文档/部署/测试任务放入 7xxx**
3. **完成任务后必须将 status 改为 done**，否则不会触发自动归档
4. **归档后 B.todo 中可能残留空目录**，需手动清理（Windows 环境下命令行无法删除空目录）
5. **不确定时先查本文档和 `requirements.md` §2.1**，不要凭直觉分配
