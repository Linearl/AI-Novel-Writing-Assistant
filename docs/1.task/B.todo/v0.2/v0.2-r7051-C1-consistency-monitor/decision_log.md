# REQ-7051: 一致性实时监控 — 决策日志

## 决策1: 检测范围——全量 vs 增量

**决策日期**：2026-07-11

**背景**：一致性检查应该每次检查全部章节，还是仅检查新增章节与前几章的关联？

**选项**：
- **方案A**: 全量检查（每次检查所有已生成章节）
- **方案B**: 增量检查（仅检查当前章节与前N章的关联）
- **方案C**: 增量检查 + 定期全量校准

**决策**：方案B（增量检查）

**理由**：
1. 全量检查在160章+时耗时过长，不适合实时场景
2. 一致性问题通常出现在相邻或相近章节之间
3. 增量检查速度快（<10秒），可嵌入生成流程
4. 定期全量校准可作为后续扩展功能

**影响**：
- 回看章节数可配置（默认3章）
- 检查器仅加载当前章节和前N章内容
- 性能可控，适合异步执行

## 决策2: 矛盾检测方式——规则 vs LLM

**决策日期**：2026-07-11

**背景**：矛盾检测应该采用纯规则方式，还是调用LLM分析？

**选项**：
- **方案A**: 纯规则检测（模式匹配 + 逻辑比较）
- **方案B**: 调用LLM进行矛盾分析
- **方案C**: 规则检测为主，矛盾疑似时LLM确认

**决策**：方案A（纯规则检测）

**理由**：
1. 规则检测速度快，无额外LLM成本
2. 时间线和空间逻辑可通过模式匹配有效检测
3. 人物行为矛盾可通过特征比较捕捉
4. 预留LLM辅助接口，后续可升级

**影响**：
- 检测模块完全本地运行
- 检测精度依赖规则和模式质量
- 可通过配置调整检测敏感度

## 决策3: 矛盾严重程度分级

**决策日期**：2026-07-11

**背景**：检测到的矛盾如何分级？

**选项**：
- **方案A**: 二级（错误/警告）
- **方案B**: 三级（错误/警告/信息）
- **方案C**: 四级（致命/错误/警告/信息）

**决策**：方案B（三级）

**理由**：
1. 二级太粗，无法区分"必须修复"和"建议修复"
2. 四级太细，实际操作时区分度不高
3. 三级与现有质量检查系统的严重程度一致
4. 错误=必须修复，警告=建议修复，信息=仅供参考

**影响**：
- 错误级矛盾触发自动通知
- 警告级矛盾记录但不主动通知
- 信息级矛盾仅在报告中展示

## 决策4: 矛盾处理策略

**决策日期**：2026-07-11

**背景**：检测到矛盾后如何处理？

**选项**：
- **方案A**: 仅报告，用户自行决定是否修复
- **方案B**: 报告 + 自动阻断后续章节生成
- **方案C**: 报告 + 可配置是否阻断

**决策**：方案A（仅报告，不阻断）

**理由**：
1. 与项目"Auto-Director质量门"原则一致——质量门不自动阻断全局链
2. 矛盾检测可能有误报，阻断会影响创作流畅性
3. 用户应有最终决定权
4. 严重矛盾通过通知提醒用户关注

**影响**：
- 检测到矛盾后保存到数据库
- 严重矛盾发送通知
- 用户可在界面查看和处理矛盾列表
- 不影响章节生成流程

## 决策5: 数据缓存策略

**决策日期**：2026-07-11

**背景**：跨章节检查需要读取历史章节数据，如何高效缓存？

**选项**：
- **方案A**: 每次从数据库读取
- **方案B**: 内存缓存（最近N章的提取数据）
- **方案C**: Redis缓存

**决策**：方案B（内存缓存）

**理由**：
1. 项目目前无Redis依赖，引入成本高
2. 每次从数据库读取对160章+项目性能不佳
3. 内存缓存简单高效，仅缓存提取后的结构化数据
4. 缓存大小可控（仅缓存最近N章的摘要数据）

**影响**：
- 在 ConsistencyMonitor 中实现简单缓存
- 缓存内容为提取后的时间引用、人物行为、空间位置
- 缓存大小由 lookbackChapters 配置控制
- 服务重启后自动重建缓存

## 决策6: 矛盾反馈机制

**决策日期**：2026-07-11

**背景**：用户如何处理检测到的矛盾？

**选项**：
- **方案A**: 仅查看，无状态管理
- **方案B**: 支持标记为"已解决"或"忽略"
- **方案C**: 支持标记 + 反馈（纠正检测结果）

**决策**：方案B（标记状态管理）

**理由**：
1. 标记状态可避免重复提示
2. 已解决和忽略是常见的矛盾处理方式
3. 反馈机制可作为后续扩展功能
4. 保持系统简洁

**影响**：
- 在 ConsistencyViolation 表中新增 status 字段
- 提供 resolve 和 ignore API
- 已解决/忽略的矛盾不再显示在活跃列表中
- 保留历史记录供分析

## 决策7: 章节内容提取方式

**决策日期**：2026-07-11

**背景**：如何从章节内容中提取时间、人物、空间信息？

**选项**：
- **方案A**: 纯正则表达式匹配
- **方案B**: 中文NLP分词 + 模式匹配
- **方案C**: 预定义模式库 + 正则匹配

**决策**：方案C（预定义模式库）

**理由**：
1. 纯正则足够处理常见的时间/空间关键词
2. 中文NLP依赖引入成本高
3. 预定义模式库可动态更新
4. 平衡了准确性和实现复杂度

**影响**：
- 在检测器中内置常用模式库
- 模式库可通过数据库动态扩展
- 无法覆盖的复杂场景通过LLM辅助接口扩展

## 总结

| 决策 | 选择 | 理由 |
|------|------|------|
| 检测范围 | 增量检查 | 速度快，适合实时场景 |
| 检测方式 | 纯规则检测 | 无外部依赖，速度快 |
| 严重程度 | 三级 | 与现有系统一致 |
| 处理策略 | 仅报告不阻断 | 与质量门原则一致 |
| 数据缓存 | 内存缓存 | 简单高效，无额外依赖 |
| 反馈机制 | 标记状态管理 | 避免重复提示 |
| 内容提取 | 预定义模式库 | 平衡准确性和复杂度 |
