# REQ-7052: 后台任务管理 — 决策日志

## 决策1: 任务状态存储位置

**决策日期**：2026-07-11

**背景**：任务状态应该存储在内存还是数据库？

**选项**：
- **方案A**: 数据库存储（SQLite/PostgreSQL）
- **方案B**: 内存存储 + 定期快照

**决策**：✅ 方案A

**理由**：
1. 服务重启后任务状态不丢失
2. 支持多实例部署（如果有需要）
3. 查询方便，支持复杂筛选
4. 与现有的Prisma数据访问层一致

**影响**：
- 新增数据库迁移（background_tasks、task_checkpoints表）
- 任务状态读写增加少量DB开销

---

## 决策2: 暂停机制实现方式

**决策日期**：2026-07-11

**背景**：如何实现任务的暂停和恢复？

**选项**：
- **方案A**: 检查点机制（每步执行后保存检查点，暂停时保存当前检查点，恢复时从检查点继续）
- **方案B**: 中断信号（暂停时设置标志位，任务在循环间隙检查标志位）

**决策**：✅ 方案A

**理由**：
1. 检查点机制更可靠，恢复时不重复已完成步骤
2. 暂停后释放LLM连接等资源
3. 实现相对清晰，每个任务类型实现自己的步骤划分
4. 支持更复杂的恢复场景

**影响**：
- 每个任务类型需要定义自己的步骤划分
- 需要task_checkpoints表存储检查点
- 任务恢复时需要从检查点恢复上下文

---

## 决策3: 实时状态推送方式

**决策日期**：2026-07-11

**背景**：如何将任务状态实时推送到前端？

**选项**：
- **方案A**: SSE（Server-Sent Events）
- **方案B**: WebSocket
- **方案C**: 前端轮询

**决策**：✅ 方案A

**理由**：
1. SSE适合服务端到客户端的单向推送，符合任务状态推送场景
2. 实现简单，无需额外依赖
3. 自动重连支持
4. 浏览器原生支持，不需要Socket.IO

**影响**：
- 需要SSE连接管理（TaskSSEManager）
- Express需要配置SSE响应头
- 前端使用EventSource API

---

## 决策4: 并发任务限制

**决策日期**：2026-07-11

**背景**：如何控制同时执行的任务数量？

**选项**：
- **方案A**: 硬限制（最多5个，超过拒绝）
- **方案B**: 软限制（超过时排队等待）
- **方案C**: 无限制

**决策**：✅ 方案A

**理由**：
1. 硬限制保护系统资源（LLM API调用额度、内存）
2. 实现简单，不需要复杂的调度逻辑
3. 用户体验清晰（拒绝时立即告知）
4. 后续可以升级为软限制

**影响**：
- 提交任务时检查当前并发数
- 超过限制时返回429 Too Many Requests
- 前端需要处理任务被拒绝的情况

---

## 决策5: 任务参数存储格式

**决策日期**：2026-07-11

**背景**：任务参数如何存储在数据库中？

**选项**：
- **方案A**: JSON序列化存储
- **方案B**: 结构化字段存储
- **方案C**: 关联表存储

**决策**：✅ 方案A

**理由**：
1. 任务参数结构多样，JSON灵活
2. 读写简单，不需要复杂映射
3. 与现有的JSON字段使用模式一致
4. 查询时不需要解析（查询较少，主要是读取执行）

**影响**：
- params和result字段使用TEXT类型存储JSON
- 读取时需要JSON.parse
- SQLite对JSON支持有限，但够用

---

## 总结

| 决策 | 选择 | 理由 |
|------|------|------|
| 状态存储 | 方案A：数据库 | 服务重启不丢失，查询方便 |
| 暂停机制 | 方案A：检查点 | 可靠恢复，不重复已完成步骤 |
| 实时推送 | 方案A：SSE | 单向推送，实现简单 |
| 并发控制 | 方案A：硬限制5个 | 保护资源，实现简单 |
| 参数存储 | 方案A：JSON序列化 | 灵活，简单 |
