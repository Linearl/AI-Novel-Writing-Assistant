# REQ-7055: RAG 质量提升

## 1. 需求背景

### 1.1 问题描述

当前 RAG 系统仅使用基础的向量相似度检索，缺乏高级检索策略。在实际使用中，检索结果存在以下问题：
- 检索结果缺乏语义维度的精细匹配（如角色类型、目标读者等 facet 维度）
- 分块缺少上下文前缀，导致相似度计算偏差
- 无重排序机制，初始检索的排序不够精准
- 检索过程不透明，难以调试和优化
- 检索追踪数据无清理机制，持续膨胀

### 1.2 现状分析

**已有功能**：
- 基础向量检索（Qdrant）
- 混合检索（HybridRetrievalService）
- Embedding 服务
- RAG 索引服务（RagIndexService）
- 基础 RAG 配置

**缺失功能**：
- 分面检索（7 维 facet 过滤）
- 上下文分块（Anthropic 风格 contextual retrieval）
- 交叉编码重排（cross-encoder reranker）
- 检索追踪（retrieval trace observability）
- 追踪数据定时清理

### 1.3 目标用户

- 所有使用 RAG 检索功能的用户（自动导演、Creative Hub、知识库检索）
- 系统运维人员（需要检索可观测性来调试问题）

## 2. 需求定义

### 2.1 功能需求

#### FR-1: 分面检索（Chunk Facets）

**描述**：为每个 RAG chunk 附加 7 维 facet 元数据，支持 facet 感知的检索。

**规则**：
- 7 个 facet 维度：genreTags, sellingPointTags, targetReaders, strengths, weaknesses, characterRole, chapterAnchor
- 每个 facet 值为字符串数组（最多 12 个）
- 索引时自动提取 facet（可选，基于 LLM）
- 检索时支持 facet 过滤
- facet 值标准化处理

**验收标准**：
- [ ] 定义 7 维 facet 类型和常量
- [ ] facet 值归一化函数（normalizeRagFacets）
- [ ] 索引时可附加 facet 元数据
- [ ] 检索时支持 facet 过滤

#### FR-2: 上下文分块（Contextual Retrieval）

**描述**：参考 Anthropic 的 contextual retrieval 方法，为每个 chunk 生成上下文前缀。

**规则**：
- 基于文档全局上下文为 chunk 生成简短前缀（最多 260 字符）
- 前缀包含文档标题、所属章节、核心主题
- 前缀拼接到 chunk 文本前面用于 embedding
- 支持增量更新（仅重新生成变更的 chunk）
- 上下文来源哈希用于去重

**验收标准**：
- [ ] 实现 RagContextualChunkService
- [ ] 生成上下文前缀（CONTEXT_PREFIX_MAX_CHARS = 260）
- [ ] 前缀正确拼接到 chunk 文本
- [ ] 来源哈希去重机制

#### FR-3: 交叉编码重排（Reranker）

**描述**：使用 cross-encoder 模型对初始检索结果进行重排序。

**规则**：
- 支持外部 reranker API 调用
- 处理多种返回格式（relevance_score, relevanceScore, score, rank_score）
- 当 reranker 不可用时优雅降级
- 重排后保留 topK 结果
- 记录重排序结果的 used 状态

**验收标准**：
- [ ] 实现 RagRerankerService
- [ ] 支持多种 API 返回格式归一化
- [ ] reranker 不可用时降级为原始排序
- [ ] 重排序结果正确截断到 topK

#### FR-4: 检索追踪（Retrieval Tracer）

**描述**：记录完整的检索过程，用于调试和优化。

**规则**：
- 记录每个阶段的耗时：vector, keyword, fusion, reranker, decay
- 记录每个阶段的候选数量
- 记录查询文本（SHA-256 摘要）
- 记录命中结果详情
- 持久化到数据库（RagRetrievalTrace 表）
- 支持按查询、租户、时间范围查询追踪

**验收标准**：
- [ ] 实现 RagRetrievalTracer
- [ ] 记录 6 个阶段的时间快照
- [ ] 记录候选数量变化
- [ ] 查询摘要使用 SHA-256
- [ ] 持久化到 Prisma

#### FR-5: 追踪清理（Trace Retention）

**描述**：定期清理过期的检索追踪数据。

**规则**：
- 定时清理（默认每 6 小时）
- 可配置保留天数
- 使用 Node.js Timer + unref 避免阻止进程退出
- 清理结果日志记录
- 支持手动触发清理

**验收标准**：
- [ ] 实现 RagRetrievalTraceRetention
- [ ] 定时器使用 unref 不阻塞进程
- [ ] 按配置的保留天数清理
- [ ] 清理日志包含删除数量和截止时间

### 2.2 非功能需求

#### NFR-1: 性能

- 上下文前缀生成不应显著增加索引时间（<20% 开销）
- 重排序延迟 < 500ms（外部 API 除外）
- 检索追踪写入不应阻塞检索响应

#### NFR-2: 可靠性

- reranker 不可用时优雅降级，不影响基础检索
- 追踪写入失败不阻断检索
- 定时清理器崩溃后自动恢复

#### NFR-3: 可观测性

- 追踪数据可通过 API 查询
- 清理日志可通过服务日志查看

## 3. 技术约束

### 3.1 架构约束

- 所有新服务放入 `server/src/services/rag/`
- Prompt 注册在 `server/src/prompting/prompts/rag/`
- 遵循现有 RAG 配置模式（ragConfig）
- 使用现有 `runStructuredPrompt` 调用 LLM

### 3.2 依赖约束

- 依赖 Qdrant（已部署）
- 依赖 EmbeddingService（已有）
- 依赖 HybridRetrievalService（已有）
- 外部依赖：cross-encoder API（可选，支持降级）

### 3.3 数据约束

- 新增 RagRetrievalTrace Prisma 模型
- RagConfig 需要新增配置项（contextualRetrievalVersion, retrievalTraceRetentionDays）
