# REQ-7053: 关键节点通知 — 决策日志

## 决策1: 通知实现方式

**决策日期**：2026-07-11

**背景**：如何实现浏览器通知？

**选项**：
- **方案A**: Web Notifications API（浏览器原生）
- **方案B**: Toast通知（前端组件）
- **方案C**: 两者结合

**决策**：✅ 方案A

**理由**：
1. Web Notifications API支持系统级通知，即使用户不在当前标签页也能收到
2. Toast通知只在页面内可见，无法触达切换了应用的用户
3. 实现简单，不需要额外UI组件
4. 不支持的浏览器可静默降级

**影响**：
- 需要请求用户通知权限
- 未授权时静默降级
- 需要功能检测（`'Notification' in window`）

## 决策2: 通知配置存储位置

**决策日期**：2026-07-11

**背景**：通知配置存储在哪里？

**选项**：
- **方案A**: localStorage（前端存储）
- **方案B**: 用户偏好设置（数据库）
- **方案C**: 服务端配置

**决策**：✅ 方案A

**理由**：
1. 通知是前端功能，配置不需要同步到服务端
2. localStorage读写简单，无网络开销
3. 浏览器级别的配置，不需要用户账号系统
4. 后续可升级为数据库存储（如果需要多设备同步）

**影响**：
- 清除浏览器数据会丢失配置
- 不支持跨设备同步
- 实现简单

---

## 决策3: 免打扰模式实现

**决策日期**：2026-07-11

**背景**：如何实现免打扰模式？

**选项**：
- **方案A**: 固定时间段免打扰
- **方案B**: 基于用户行为智能判断
- **方案C**: 仅提供全局开关

**决策**：✅ 方案A

**理由**：
1. 固定时间段免打扰实现简单
2. 用户可自定义时间段
3. 行为判断需要复杂的分析，当前不需要
4. 全局开关太粗，不够灵活

**影响**：
- 默认免打扰时间：22:00-08:00
- 用户可自定义时间段
- 配置存储在localStorage

---

## 决策4: 通知频率控制

**决策日期**：2026-07-11

**背景**：如何防止通知过于频繁？

**选项**：
- **方案A**: 相同事件30秒内不重复发送
- **方案B**: 合并同类通知
- **方案C**: 每分钟最多3条通知

**决策**：✅ 方案A

**理由**：
1. 防止同一事件重复通知（如任务状态频繁变更）
2. 实现简单，基于时间戳判断
3. 不影响不同事件的正常通知

**影响**：
- 同一任务的状态变更通知30秒内只发一次
- 不同任务的通知独立控制
- 首次通知不受限制

---

## 决策5: 通知内容粒度

**决策日期**：2026-07-11

**背景**：通知内容应该包含多少信息？

**选项**：
- **方案A**: 仅标题（"任务完成"）
- **方案B**: 标题+简要内容（"第X章完成"）
- **方案C**: 标题+详细内容+操作按钮

**决策**：✅ 方案B

**理由**：
1. 标题+简要内容足以传达关键信息
2. 详细内容在通知中阅读不便
3. 操作按钮需要Service Worker，增加复杂度
4. 点击通知跳转到详情页更合适

**影响**：
- 通知标题表达事件类型
- 通知内容包含关键信息（章节号、错误原因等）
- 点击通知跳转到对应页面
