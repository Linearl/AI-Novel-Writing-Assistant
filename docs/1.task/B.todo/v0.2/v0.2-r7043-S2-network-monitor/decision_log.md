# REQ-7043: 网络状态监控 — 决策日志

## 决策1: 心跳探测实现方式

**决策日期**：2026-07-11

**背景**：心跳探测应该使用什么方式实现？

**选项**：
- **方案A**: 复用现有 `llmConnectivityService.testConnection`
- **方案B**: 新建独立的轻量级 HTTP 探测
- **方案C**: 使用 Node.js `dns.lookup` 检测 DNS 可达性

**决策**：✅ 方案A

**理由**：
1. `testConnection` 已经封装了完整的 LLM 连通性探测逻辑
2. 使用 plain probe 模式（16 maxTokens）足够轻量
3. 复用现有代码减少维护成本
4. 探测结果与实际 LLM 调用行为一致

**影响**：
- 心跳探测复用 `connectivity.ts` 的 `testPlainConnection`
- 探测延迟取决于 LLM Provider 响应时间
- 心跳配置需要知道当前默认 Provider

## 决策2: 断网判定阈值

**决策日期**：2026-07-11

**背景**：连续多少次失败才判定为断网？

**选项**：
- **方案A**: 1 次失败即判定（最敏感）
- **方案B**: 3 次连续失败才判定（平衡）
- **方案C**: 5 次连续失败才判定（最保守）

**决策**：✅ 方案B（3次连续失败）

**理由**：
1. 1 次失败太敏感，网络抖动会导致频繁状态切换
2. 5 次太保守，按 30 秒间隔需要 2.5 分钟才能检测到断网
3. 3 次（90 秒）是合理的平衡点
4. 恢复判定只需 1 次成功（快速恢复）

**影响**：
- 断网检测延迟：最多 90 秒（3 次 x 30 秒）
- 恢复检测延迟：最多 30 秒（1 次 x 30 秒）
- 配置参数 `failureThreshold = 3`，`recoveryThreshold = 1`

## 决策3: 心跳间隔

**决策日期**：2026-07-11

**背景**：心跳探测应该多长时间执行一次？

**选项**：
- **方案A**: 15 秒（最快检测，但 API 调用频率高）
- **方案B**: 30 秒（平衡检测速度和 API 开销）
- **方案C**: 60 秒（节省 API 开销，但检测延迟大）

**决策**：✅ 方案B（30秒）

**理由**：
1. 15 秒间隔可能触发 API 限流（特别是免费/低配额 Provider）
2. 60 秒间隔导致断网检测延迟过长（3 次 x 60 秒 = 3 分钟）
3. 30 秒间隔在检测速度和 API 开销之间平衡
4. 16 maxTokens 的轻量级请求，API 开销可忽略

**影响**：
- 心跳配置 `heartbeatIntervalMs = 30_000`
- 最短间隔限制 15 秒（防止配置过低导致限流）

## 决策4: 网络状态存储方式

**决策日期**：2026-07-11

**背景**：网络状态应该存储在哪里？

**选项**：
- **方案A**: 内存中维护（仅在进程生命周期内有效）
- **方案B**: 持久化到 SQLite（重启后保留历史）
- **方案C**: 持久化到文件（轻量级持久化）

**决策**：✅ 方案A

**理由**：
1. 网络状态是实时数据，历史状态价值有限
2. 服务重启后重新探测即可获取当前状态
3. 避免数据库写入开销
4. 简化实现复杂度

**影响**：
- 服务重启后状态重置（默认假设在线）
- 不保留历史断网记录（最近 10 次探测在内存中）
- 后续如需历史分析，可扩展为持久化

## 决策5: 事件发布机制

**决策日期**：2026-07-11

**背景**：网络状态变化事件应该如何发布？

**选项**：
- **方案A**: 使用现有 `novelEventBus` 发布
- **方案B**: 新建独立的 `NetworkEventBus`
- **方案C**: 使用 Node.js EventEmitter 直接发布

**决策**：✅ 方案A

**理由**：
1. 项目已有成熟的 EventBus 机制
2. Director 和 SideEffect 已经订阅 `novelEventBus`
3. 保持事件系统一致性
4. 不需要新建独立的事件总线

**影响**：
- 需要在 `events/types.ts` 中新增 `network:online` 和 `network:offline` 事件类型
- 需要在 `EventBus` 的 `summarizePayload` 中处理新事件类型
- Director 和 SideEffect 通过 `novelEventBus.on("network:...")` 订阅

## 决策6: 配置管理

**决策日期**：2026-07-11

**背景**：心跳配置应该在哪里管理？

**选项**：
- **方案A**: 环境变量（简单但不灵活）
- **方案B**: Provider Secret 表（与 REQ-7040 一致）
- **方案C**: 硬编码默认值 + 构造函数参数（最简单）

**决策**：✅ 方案C

**理由**：
1. S2 任务优先简单实现
2. 心跳配置变化频率低，不需要运行时动态调整
3. 构造函数参数支持测试时注入不同配置
4. 后续如需动态配置，可扩展为读取 Provider Secret

**影响**：
- `NetworkMonitor` 构造函数接受 `Partial<NetworkMonitorConfig>`
- 默认值在构造函数中设置
- 测试时可通过构造函数注入不同配置

## 总结

| 决策 | 选择 | 理由 |
|------|------|------|
| 心跳探测 | 方案A：复用 testConnection | 复用现有逻辑，减少维护成本 |
| 断网阈值 | 方案B：3次连续失败 | 平衡敏感度和误判率 |
| 心跳间隔 | 方案B：30秒 | 平衡检测速度和 API 开销 |
| 状态存储 | 方案A：内存维护 | 实时数据无需持久化 |
| 事件发布 | 方案A：novelEventBus | 保持事件系统一致性 |
| 配置管理 | 方案C：构造函数参数 | S2 任务优先简单实现 |
