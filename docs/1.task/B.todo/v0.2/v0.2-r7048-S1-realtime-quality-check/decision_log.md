# REQ-7048: 实时质量检查器 — 决策日志

## 决策1: 实现模式——独立服务 vs 嵌入生成流程

**决策日期**：2026-07-11

**背景**：质量检查应该作为独立服务模块，还是嵌入到现有的章节生成流程中？

**选项**：
- **方案A**: 独立服务模块，异步执行，不阻塞生成流程
- **方案B**: 嵌入生成流程，同步执行，生成后立即检查

**决策**：方案A

**理由**：
1. 质量检查耗时不确定，同步执行可能阻塞生成流程
2. 异步执行允许并发检查多个章节
3. 独立模块便于单独测试和维护
4. 与项目"模块化、高内聚低耦合"原则一致

**影响**：
- 在 `server/src/services/novel/quality/` 下新增模块
- 生成流程通过事件或回调异步触发检查
- 检查结果存储在独立的 `ChapterQualityReport` 表

## 决策2: 检查器架构——策略模式 vs 单一检查类

**决策日期**：2026-07-11

**背景**：多个检查维度（字数、结构、人物、情节、AI味）如何组织？

**选项**：
- **方案A**: 策略模式，每个维度独立实现，主类编排
- **方案B**: 单一检查类，所有维度逻辑集中在一个文件

**决策**：方案A

**理由**：
1. 每个维度逻辑差异大，独立实现更清晰
2. 符合SRP（单一职责）原则
3. 便于单独测试和后续扩展新维度
4. 与项目"单文件目标~600行"约束一致

**影响**：
- 每个检查器一个文件
- 主类仅负责编排和汇总
- 新增检查维度只需实现接口

## 决策3: 人物提取方式——正则模式 vs LLM提取

**决策日期**：2026-07-11

**背景**：如何从章节内容中提取人物名称？

**选项**：
- **方案A**: 基于中文命名模式的正则表达式提取
- **方案B**: 调用LLM进行人物识别
- **方案C**: 正则预提取 + 人物库交叉验证

**决策**：方案C

**理由**：
1. 纯正则提取召回率高但精度有限
2. LLM提取精度高但成本高、耗时长
3. 正则预提取 + 人物库交叉验证是最佳平衡点
4. 人物库提供语义验证，正则提供候选集合

**影响**：
- 需要访问人物设定数据库
- 人物提取逻辑与人物服务交互
- 检查结果可提供未定义人物和主要人物缺失两类报告

## 决策4: 情节连贯性检测方式

**决策日期**：2026-07-11

**背景**：如何检测章节之间的情节连贯性？

**选项**：
- **方案A**: 基于关键词/场景标签对比
- **方案B**: 调用LLM分析上下文连贯性
- **方案C**: 关键词检测 + LLM辅助验证

**决策**：方案C（初期以关键词检测为主，后续可扩展LLM）

**理由**：
1. 关键词检测速度快，成本低，适合实时检查
2. LLM分析精度高但耗时长，成本高
3. 初期以关键词检测满足基本需求
4. 预留LLM辅助验证接口，后续可扩展

**影响**：
- 实现基于场景关键词的时间线和场景跳跃检测
- 在QualityCheckConfig中预留LLM配置字段
- 可逐步升级为LLM辅助分析

## 决策5: 配置存储位置

**决策日期**：2026-07-11

**背景**：质量检查的配置参数应该存储在哪里？

**选项**：
- **方案A**: 环境变量
- **方案B**: 数据库系统配置表
- **方案C**: 配置文件

**决策**：方案B

**理由**：
1. 系统配置表已是配置的集中存储位置
2. 支持运行时动态调整，无需重启
3. 与项目现有的配置管理方式一致
4. 可通过设置页面调整（与模型配置一致）

**影响**：
- 在系统配置表中新增质量检查配置项
- 提供配置读取和更新API
- 前端设置页面可展示和调整配置

## 决策6: 异步触发方式

**决策日期**：2026-07-11

**背景**：章节生成完成后如何触发质量检查？

**选项**：
- **方案A**: 事件驱动（通过事件总线）
- **方案B**: 在生成回调中直接调用
- **方案C**: 轮询检查未检查的章节

**决策**：方案A

**理由**：
1. 与项目现有的事件总线架构一致
2. 松耦合，生成流程不需要知道质量检查的存在
3. 支持并发处理，不阻塞主流程
4. 可在事件处理器中处理失败重试

**影响**：
- 注册事件监听器监听章节生成完成事件
- 在事件处理器中异步执行质量检查
- 检查失败不影响章节保存

## 总结

| 决策 | 选择 | 理由 |
|------|------|------|
| 实现模式 | 独立服务模块 | 不阻塞生成流程，便于维护 |
| 检查器架构 | 策略模式 | SRP原则，便于扩展 |
| 人物提取 | 正则+人物库交叉验证 | 平衡精度和成本 |
| 情节连贯性 | 关键词检测为主 | 速度快，可扩展 |
| 配置存储 | 数据库系统配置表 | 支持动态调整 |
| 异步触发 | 事件驱动 | 松耦合，与现有架构一致 |
