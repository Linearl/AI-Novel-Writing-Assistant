# REQ-7040: API失败自动重试 — 决策日志

## 决策1: 实现位置选择

**决策日期**：2026-07-11

**背景**：重试逻辑应该在哪里实现？

**选项**：
- **方案A**: 在 `invokeStructuredLlmDetailed` 层添加
- **方案B**: 在 `factory.ts` 层添加重试包装

**决策**：✅ 方案A

**理由**：
1. `invokeStructuredLlmDetailed` 是绝大多数LLM调用的入口
2. 可以复用现有的 `StructuredOutputError` 分类
3. 不会影响策略降级逻辑（重试在策略降级之后）
4. 更容易管理和监控

**影响**：
- 修改 `server/src/llm/structuredInvoke.ts`
- 新增 `server/src/llm/retryHandler.ts`

---

## 决策2: 重试时机选择

**决策日期**：2026-07-11

**背景**：重试应该在策略降级之前还是之后？

**选项**：
- **方案A**: 在策略降级之后重试
- **方案B**: 在策略降级之前重试

**决策**：✅ 方案A

**理由**：
1. 策略降级是针对输出格式的兼容性问题
2. 重试是针对传输层错误（网络、限流）
3. 应该先尝试所有策略，最后再重试
4. 这样可以最大化成功率

**影响**：
- 重试逻辑在 `tryStructuredStrategies` 失败后触发
- 每次重试都会重新走一遍策略降级流程

---

## 决策3: 429限流处理策略

**决策日期**：2026-07-11

**背景**：收到429限流响应时，应该如何处理？

**选项**：
- **方案A**: 解析Retry-After头，等待指定时间
- **方案B**: 使用指数退避，忽略Retry-After

**决策**：✅ 方案A（优先使用Retry-After，fallback到指数退避）

**理由**：
1. Retry-After是服务端的建议，应该尊重
2. 服务端知道自己的限流策略
3. 如果没有Retry-After头，再使用指数退避
4. 设置最大等待时间（60秒）避免无限等待

**影响**：
- 需要解析Retry-After头（支持秒数和HTTP日期格式）
- 需要处理Retry-After头不存在的情况

---

## 决策4: 重试次数配置

**决策日期**：2026-07-11

**背景**：重试次数是否应该可配置？

**选项**：
- **方案A**: 硬编码默认3次，不支持配置
- **方案B**: 可配置，默认3次

**决策**：✅ 方案B

**理由**：
1. 不同的Provider可能有不同的限流策略
2. 用户可能想调整重试行为
3. 默认值3次是合理的（1+2+4=7秒总等待）

**影响**：
- 需要在Provider Secret表中存储配置
- 需要提供配置API
- 需要处理配置不存在的情况（使用默认值）

---

## 决策5: 错误分类策略

**决策日期**：2026-07-11

**背景**：哪些错误应该重试？

**选项**：
- **方案A**: 仅重试网络错误
- **方案B**: 重试网络错误 + 429限流
- **方案C**: 重试所有5xx错误

**决策**：✅ 方案B

**理由**：
1. 网络错误（ECONNRESET, ETIMEDOUT）是临时的，应该重试
2. 429限流是临时的，应该重试
3. 5xx错误可能是永久的，不应该默认重试
4. 4xx错误（401, 403, 400）不应该重试

**影响**：
- 定义明确的可重试错误列表
- 其他错误直接抛出，不重试

---

## 决策6: 日志策略

**决策日期**：2026-07-11

**背景**：重试事件应该如何记录日志？

**选项**：
- **方案A**: 每次重试都记录warn日志
- **方案B**: 仅记录最终失败的error日志
- **方案C**: 每次重试记录warn，最终失败记录error

**决策**：✅ 方案C

**理由**：
1. 每次重试都记录warn，便于追踪重试过程
2. 最终失败记录error，便于发现问题
3. 平衡了可观测性和日志量

**影响**：
- 重试时记录warn级别日志
- 最终失败时记录error级别日志
- 日志包含必要的调试信息（尝试次数、错误类型、等待时间）

---

## 决策7: 配置存储位置

**决策日期**：2026-07-11

**背景**：重试配置应该存储在哪里？

**选项**：
- **方案A**: 环境变量
- **方案B**: Provider Secret表
- **方案C**: 配置文件

**决策**：✅ 方案B

**理由**：
1. Provider Secret表已经是配置的集中存储位置
2. 可以按Provider维度配置不同的重试策略
3. 支持运行时动态更新
4. 与现有的Provider配置管理一致

**影响**：
- 在Provider Secret表中新增retryConfig字段
- 需要提供配置读取和更新API

| 重试时机 | 方案A：策略降级之后 | 最大化成功率 |
