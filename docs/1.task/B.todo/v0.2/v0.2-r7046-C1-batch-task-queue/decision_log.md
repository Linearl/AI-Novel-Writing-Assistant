# REQ-7046: 批量任务队列 — 决策日志

## 决策1: 队列存储方式

**决策日期**：2026-07-11

**背景**：批量任务队列应该存储在哪里？

**选项**：
- **方案A**: 内存队列（Redis/Memory）
- **方案B**: 数据库存储（Prisma）
- **方案C**: 文件存储（JSON文件）

**决策**：✅ 方案B（数据库存储）

**理由**：
1. 服务重启后队列状态不丢失
2. 与现有Prisma数据层一致
3. 可查询历史队列记录
4. 避免引入新的存储依赖

**影响**：
- 需要新增BatchQueue和Batch模型
- 队列状态持久化到SQLite/PostgreSQL

---

## 决策2: 批次分解策略

**决策日期**：2026-07-11

**背景**：如何将大任务分解为小批次？

**选项**：
- **方案A**: 固定大小（每批次N章）
- **方案B**: 按时间（每批次执行N分钟）
- **方案C**: 动态调整（根据执行速度调整批次大小）

**决策**：✅ 方案A（固定大小）

**理由**：
1. 实现简单，逻辑清晰
2. 用户容易理解和预期（"每批10章"）
3. 执行时间可估算（每章约X分钟）
4. 避免复杂的状态机逻辑

**影响**：
- 默认批次大小10章
- 可配置（5-50章）

---

## 决策3: 调度方式

**决策日期**：2026-07-11

**背景**：队列调度应该如何实现？

**选项**：
- **方案A**: 后台轮询（定时检查队列）
- **方案B**: 事件驱动（任务完成后触发下一个）
- **方案C**: 消息队列（Redis/RabbitMQ）

**决策**：✅ 方案A（后台轮询）

**理由**：
1. 实现简单，无需额外依赖
2. 1秒轮询间隔足够（调度开销<10ms）
3. 与现有架构兼容
4. 避免引入消息队列的复杂性

**影响**：
- 需要实现TaskScheduler后台调度器
- 轮询间隔可配置

---

## 决策4: 失败处理策略

**决策日期**：2026-07-11

**背景**：单个任务失败时如何处理？

**选项**：
- **方案A**: 立即停止队列
- **方案B**: 跳过失败任务，继续执行
- **方案C**: 重试失败任务，3次失败后跳过

**决策**：✅ 方案C（重试后跳过）

**理由**：
1. 暂时性错误（网络超时）应该重试
2. 永久性错误（章节内容非法）应该跳过
3. 3次重试是合理阈值
4. 不阻断整个队列的执行

**影响**：
- 失败任务加入重试队列
- 最多重试3次
- 3次失败后标记为永久失败

---

## 决策5: 批次大小配置

**决策日期**：2026-07-11

**背景**：批次大小是否应该可配置？

**选项**：
- **方案A**: 硬编码默认10章，不支持配置
- **方案B**: 可配置，默认10章

**决策**：✅ 方案B（可配置）

**理由**：
1. 不同用户的硬件环境不同
2. 不同章节的生成复杂度不同
3. 用户可能想调整批次大小以优化速度
4. 默认值10章是合理的

**影响**：
- 批次大小可配置（5-50章）
- 配置存储在BatchQueue.config字段

---

## 决策6: 队列暂停/恢复

**决策日期**：2026-07-11

**背景**：是否需要支持队列暂停/恢复？

**选项**：
- **方案A**: 不支持暂停，只能清空队列
- **方案B**: 支持暂停/恢复

**决策**：✅ 方案B（支持暂停/恢复）

**理由**：
1. 用户可能想暂停队列（如网络不好）
2. 暂停后可恢复，无需重新添加任务
3. 暂停期间不执行新批次
4. 恢复后从暂停点继续

**影响**：
- 队列状态新增PAUSED状态
- 调度器跳过PAUSED状态的队列

## 总结

| 决策 | 选择 | 理由 |
|------|------|------|
| 存储方式 | 方案B：数据库 | 持久化，与现有架构一致 |
| 批次分解 | 方案A：固定大小 | 实现简单，用户易理解 |
| 调度方式 | 方案A：后台轮询 | 实现简单，无额外依赖 |
| 失败处理 | 方案C：重试后跳过 | 平衡可靠性和效率 |
| 批次配置 | 方案B：可配置 | 灵活性 |
| 暂停恢复 | 方案B：支持 | 用户体验 |
