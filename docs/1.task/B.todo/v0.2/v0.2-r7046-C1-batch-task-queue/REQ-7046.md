# REQ-7046: 批量任务队列

## 1. 需求背景

### 1.1 问题描述

当前系统执行章节生成时，一次只能处理一个任务。对于需要生成300+章的长篇小说项目，用户必须等待一个章节完成后才能添加下一个，导致生产效率极低。缺乏批量任务排队和自动调度机制，用户需要反复手动操作，严重阻碍了大规模小说创作的流程自动化。

### 1.2 现状分析

**已有功能**：
- 单章节生成（逐章执行）
- 章节任务状态管理（pending/running/completed/failed）
- 检查点机制（可恢复执行）

**缺失功能**：
- ❌ 批量任务队列（支持300+章排队）
- ❌ 自动批次分解（将大任务拆分为小批次）
- ❌ 批次间自动衔接（前一批完成后自动启动下一批）
- ❌ 队列状态监控

### 1.3 目标用户

所有使用系统进行长篇小说创作的用户，特别是：
- 创作300+章超长篇网文的用户
- 需要批量生成章节的用户
- 希望无人值守批量执行的用户

## 2. 需求定义

### 2.1 功能需求

#### FR-1: 批量任务队列

**描述**：支持将大量章节生成任务加入队列，系统按顺序执行。

**规则**：
- 支持添加1-500个任务到队列
- 队列按FIFO顺序执行
- 支持暂停/恢复队列
- 支持清空队列

**验收标准**：
- [ ] 可一次性添加多个章节任务
- [ ] 任务按添加顺序执行
- [ ] 可暂停和恢复队列
- [ ] 可清空未执行的任务

#### FR-2: 自动批次分解

**描述**：系统自动将大任务分解为小批次，每批次执行固定数量的章节。

**规则**：
- 默认每批次10章
- 批次大小可配置（5-50章）
- 批次间自动衔接
- 批次失败可重试

**验收标准**：
- [ ] 300章任务自动分解为30个批次
- [ ] 每批次执行固定数量章节
- [ ] 批次完成后自动启动下一批
- [ ] 批次失败时支持重试

#### FR-3: 队列状态监控

**描述**：实时显示队列执行状态和进度。

**规则**：
- 显示当前批次进度
- 显示整体进度百分比
- 显示预计剩余时间
- 显示失败任务列表

**验收标准**：
- [ ] 实时更新队列状态
- [ ] 显示当前执行章节
- [ ] 显示整体进度
- [ ] 可查看失败任务详情

#### FR-4: 失败处理

**描述**：任务失败时的处理策略。

**规则**：
- 单个任务失败不阻断队列
- 失败任务加入重试队列
- 最多重试3次
- 3次失败后标记为永久失败

**验收标准**：
- [ ] 单个失败不阻断队列
- [ ] 失败任务自动重试
- [ ] 3次失败后标记永久失败
- [ ] 可查看永久失败任务

### 2.2 非功能需求

#### NFR-1: 性能

- 队列调度开销<10ms
- 支持同时管理500个任务

#### NFR-2: 可靠性

- 队列状态持久化
- 服务重启后恢复队列
- 不丢失任务

#### NFR-3: 可观测性

- 队列状态可通过API查询
- 执行日志完整可追踪

## 3. 技术约束

### 3.1 架构约束

- 必须复用现有的任务状态管理
- 队列调度在服务端后台运行
- 前端通过轮询或WebSocket获取状态

### 3.2 依赖约束

- 依赖：REQ-7044（检查点管理，清理配合）
- 被依赖：REQ-7047（进度可视化，消费队列状态）

### 3.3 数据约束

- 需要新增TaskQueue表存储队列状态
- Task表已有状态字段可复用

## 4. 验收标准

### 4.1 功能验收

- [ ] 可添加500个任务到队列
- [ ] 任务按顺序自动执行
- [ ] 批次间自动衔接
- [ ] 失败任务自动重试
- [ ] 队列状态实时更新

### 4.2 性能验收

- [ ] 调度开销<10ms
- [ ] 支持500任务并发管理

### 4.3 测试验收

- [ ] 单元测试覆盖率>80%
- [ ] 集成测试通过
- [ ] 压力测试（500任务）

## 5. 风险评估

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|----------|
| 队列死锁 | 任务停滞 | 中 | 超时机制 + 心跳检测 |
| 批次大小不当 | 效率低下 | 中 | 可配置批次大小 |
| 内存溢出 | 服务崩溃 | 低 | 限制队列大小 + 流式处理 |
| 任务丢失 | 用户数据丢失 | 低 | 持久化队列状态 |

## 6. 工作量评估

- **开发时间**：1.5天
- **测试时间**：0.5天
- **总计**：2天
