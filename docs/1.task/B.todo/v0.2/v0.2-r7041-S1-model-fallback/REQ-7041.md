# REQ-7041: 模型备用切换

## 1. 需求背景

### 1.1 问题描述

当前系统仅支持单一备用模型（`structuredFallbackSettings`），当主模型失败时统一切换到同一个备用模型。但在实际生产中，不同错误类型需要不同的应对策略：

- **429限流**：当前Provider过载，应切换到不同Provider
- **认证失败（401/403）**：API Key失效，应切换到不同Provider
- **模型不可用（503）**：特定模型下线，应切换到同Provider的不同模型
- **网络不可达**：Provider整体不可达，应切换到其他Provider

单一备用模型无法覆盖所有场景，导致部分错误场景下切换到的备用模型同样不可用。

### 1.2 现状分析

**已有功能**：
- `structuredFallbackSettings` — 单一备用模型配置（provider + model + temperature + maxTokens）
- `invokeStructuredLlmDetailed` 中的 try-catch — 主模型失败后尝试备用模型
- `StructuredOutputError.category` 错误分类（transport_error / schema_mismatch 等）
- `isRetryableError`（REQ-7040新增）— 网络/限流错误重试

**缺失功能**：
- ❌ 按错误类型选择不同备用模型
- ❌ 多级备用模型链（primary → fallback1 → fallback2）
- ❌ 备用模型切换日志（切换原因、目标模型）
- ❌ 备用模型健康状态追踪

### 1.3 目标用户

所有使用系统进行长篇小说创作的用户，特别是：
- 使用多Provider配置的用户
- 长时间连续生成（160章+）的用户
- 使用有限流API的用户

## 2. 需求定义

### 2.1 功能需求

#### FR-1: 错误类型驱动的模型切换

**描述**：主模型失败时，根据错误类型自动选择最合适的备用模型进行切换。

**规则**：
- `transport_error`（网络超时、502/503/504）→ 优先切换到不同Provider
- `429限流`（Rate Limit）→ 切换到不同Provider（限流通常针对Provider维度）
- 认证错误（401/403）→ 切换到不同Provider（API Key维度）
- 输出格式错误（schema_mismatch / malformed_json）→ 保持当前Provider，切换到不同模型（格式支持差异）
- 不可恢复错误（400 Bad Request）→ 不切换，直接报错

**验收标准**：
- [ ] 429限流错误触发切换到不同Provider
- [ ] 认证错误触发切换到不同Provider
- [ ] 503模型不可用触发切换到同Provider不同模型
- [ ] 输出格式错误触发切换到不同模型
- [ ] 400错误不触发切换

#### FR-2: 多级备用模型链

**描述**：支持配置多个备用模型，按优先级依次尝试。

**规则**：
- 最多支持3级备用模型（primary → fallback_1 → fallback_2 → fallback_3）
- 每级备用模型可独立配置 provider/model/temperature/maxTokens
- 当前级备用模型失败后，自动尝试下一级
- 所有备用模型都失败后，抛出最终错误

**验收标准**：
- [ ] 支持配置3级备用模型
- [ ] 第一级备用失败后自动尝试第二级
- [ ] 所有级都失败后正确报错
- [ ] 未配置的级别自动跳过

#### FR-3: 备用模型切换日志

**描述**：每次模型切换都记录详细日志，便于排查问题。

**日志内容**：
- 切换原因（错误类型、错误消息）
- 源模型（provider + model）
- 目标模型（provider + model）
- 切换结果（成功 / 失败）

**验收标准**：
- [ ] 每次切换都有日志记录
- [ ] 日志包含错误类型和目标模型信息
- [ ] 切换成功和失败都有记录

#### FR-4: 与REQ-7040重试机制协同

**描述**：模型切换与API重试机制协同工作，避免重复尝试。

**规则**：
- 先执行重试（REQ-7040），重试失败后再触发模型切换
- 重试期间不触发模型切换
- 模型切换后，新模型也支持重试（如果启用）

**验收标准**：
- [ ] 重试失败后才触发模型切换
- [ ] 重试期间不触发模型切换
- [ ] 切换后的模型也支持重试

### 2.2 非功能需求

#### NFR-1: 性能

- 模型切换决策延迟 < 50ms
- 切换过程不中断生成流程（透明切换）
- 备用模型连接预热不阻塞主调用

#### NFR-2: 可观测性

- 切换事件可通过日志追踪
- 切换次数可监控
- 切换成功率可统计

#### NFR-3: 兼容性

- 向后兼容现有 `structuredFallbackSettings` 配置
- 未配置多级备用时，行为与当前一致（单一备用）
- 不影响 `disableFallbackModel` 参数

## 3. 技术约束

### 3.1 架构约束

- 必须在 `invokeStructuredLlmDetailed` 层实现
- 在 REQ-7040 重试机制之后触发
- 复用现有的 `StructuredOutputError` 错误分类
- 复用现有的 `resolveAttemptTarget` 解析目标模型

### 3.2 依赖约束

- 依赖：REQ-7040（API失败自动重试）
- 被依赖：无

### 3.3 数据约束

- 备用模型配置存储在 `AppSetting` 表
- 切换日志存储在应用日志
- 不新增数据库表

## 4. 验收标准

### 4.1 功能验收

- [ ] 429限流自动切换到不同Provider
- [ ] 认证错误自动切换到不同Provider
- [ ] 503不可用自动切换到同Provider不同模型
- [ ] 多级备用模型链按优先级尝试
- [ ] 切换日志完整
- [ ] 与REQ-7040重试机制正确协同

### 4.2 性能验收

- [ ] 切换决策延迟 < 50ms
- [ ] 切换过程不中断生成
- [ ] 不影响正常调用延迟

### 4.3 测试验收

- [ ] 单元测试覆盖率 > 80%
- [ ] 集成测试通过
- [ ] 边界用例覆盖（无备用、1级、3级备用）

## 5. 风险评估

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|----------|
| 备用模型也不可用 | 所有模型失败 | 中 | 多级备用 + 明确报错 |
| 切换逻辑复杂 | 维护成本 | 低 | 保持与现有代码模式一致 |
| 配置错误 | 切换行为异常 | 低 | 配置校验 + 默认值 |
