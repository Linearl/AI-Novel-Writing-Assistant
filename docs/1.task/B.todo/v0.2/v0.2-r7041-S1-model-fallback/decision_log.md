# REQ-7041: 模型备用切换 — 决策日志

## 决策1: 实现位置选择

**决策日期**：2026-07-11

**背景**：模型切换逻辑应该在哪里实现？

**选项**：
- **方案A**: 在 `invokeStructuredLlmDetailed` 层扩展现有逻辑
- **方案B**: 在 `factory.ts` 层添加切换包装
- **方案C**: 新建独立的 `modelFallbackManager.ts` 作为中间层

**决策**：✅ 方案A

**理由**：
1. `invokeStructuredLlmDetailed` 已经有备用模型切换逻辑，扩展最自然
2. 可以复用现有的 `resolveAttemptTarget` 和 `tryStructuredStrategies`
3. 不需要新增额外的调用层次
4. 与REQ-7040重试机制在同一个函数中协同

**影响**：
- 修改 `server/src/llm/structuredInvoke.ts`
- 新增 `server/src/llm/fallbackChain.ts`

## 决策2: 备用模型链长度

**决策日期**：2026-07-11

**背景**：最多支持几级备用模型？

**选项**：
- **方案A**: 最多1级（与当前一致，仅增强错误类型感知）
- **方案B**: 最多2级
- **方案C**: 最多3级

**决策**：✅ 方案C

**理由**：
1. 3级备用可覆盖：primary → 同Provider不同模型 → 不同Provider A → 不同Provider B
2. 实际场景中3级足够应对绝大多数故障
3. 超过3级的配置复杂度收益递减
4. 代码实现上用数组遍历，3级和2级成本相同

**影响**：
- 配置存储需要支持最多3组 fallbackChain.N.* 键值
- 遍历逻辑需要处理数组长度

## 决策3: 错误类型映射策略

**决策日期**：2026-07-11

**背景**：不同错误类型应该映射到什么样的备用模型选择策略？

**选项**：
- **方案A**: 所有错误统一切换到固定备用模型
- **方案B**: 按错误类型选择不同切换策略（不同Provider / 同Provider不同模型）
- **方案C**: 按错误类型 + Provider亲和度综合决策

**决策**：✅ 方案B

**理由**：
1. 429限流通常是Provider维度的，切换到不同Provider更有效
2. 503模型不可用通常是模型维度的，切换到同Provider不同模型更合理
3. 认证错误（401/403）是API Key维度的，必须切换到不同Provider
4. 输出格式错误是模型能力差异，切换到不同模型即可
5. 方案C过于复杂，实际场景不需要Provider亲和度

**影响**：
- 需要 `classifyFallbackTrigger` 函数将错误映射到触发原因
- 需要 `selectFallbackModel` 函数根据触发原因选择备用模型

## 决策4: 配置存储位置

**决策日期**：2026-07-11

**背景**：多级备用链配置应该存储在哪里？

**选项**：
- **方案A**: 环境变量
- **方案B**: AppSetting表（与现有 structuredFallbackSettings 一致）
- **方案C**: 新增数据库表

**决策**：✅ 方案B

**理由**：
1. 与现有的 `structuredFallbackSettings` 存储方式一致
2. 支持运行时动态更新
3. 不需要数据库迁移
4. 可以通过 settings 页面管理

**影响**：
- 使用 `AppSetting` 表的 `fallbackChain.*` 键值
- 读写逻辑复用 `prisma.appSetting` 操作模式

## 决策5: 向后兼容策略

**决策日期**：2026-07-11

**背景**：如何处理现有的单一备用模型配置？

**选项**：
- **方案A**: 替换现有逻辑，强制使用新配置格式
- **方案B**: 新配置优先，无新配置时回退到原逻辑
- **方案C**: 启动时自动迁移旧配置到新格式

**决策**：✅ 方案B

**理由**：
1. 不破坏现有用户的配置
2. 无需用户手动迁移
3. 新功能渐进式启用
4. 实现简单，无需迁移脚本

**影响**：
- `invokeStructuredLlmDetailed` 中保留原有 fallbackSettings 逻辑作为 fallback
- 仅在 `fallbackChain.enabled = true` 时使用新逻辑

## 决策6: 与REQ-7040重试机制的协同方式

**决策日期**：2026-07-11

**背景**：模型切换与API重试应该如何协同？

**选项**：
- **方案A**: 先重试，重试失败后切换模型
- **方案B**: 先切换模型，切换后的新模型再重试
- **方案C**: 重试和切换并行（同时尝试多个模型）

**决策**：✅ 方案A

**理由**：
1. 重试成本低（无网络切换），应该先尝试
2. 切换模型成本高（可能需要不同的API Key/配置）
3. 顺序执行逻辑清晰，易于调试
4. 方案C实现复杂，且可能加剧限流

**影响**：
- 执行顺序：策略降级 → 重试 → 备用模型切换
- 切换后的新模型也支持重试（如果启用）

## 总结

| 决策 | 选择 | 理由 |
|------|------|------|
| 实现位置 | 方案A：invokeStructuredLlmDetailed层 | 扩展现有逻辑最自然 |
| 链长度 | 方案C：最多3级 | 足够覆盖实际场景 |
| 错误映射 | 方案B：按错误类型选择策略 | 不同错误需要不同应对 |
| 配置存储 | 方案B：AppSetting表 | 与现有配置管理一致 |
| 向后兼容 | 方案B：新配置优先，回退原逻辑 | 不破坏现有配置 |
| 与重试协同 | 方案A：先重试后切换 | 成本低的先尝试 |
