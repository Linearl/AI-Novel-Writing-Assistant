# 审计任务包模板说明

## 1. 为什么用审计包而非多条需求

代码审计（ll-audit）产生的问题通常具有以下特点：

| 特点 | 对应管理挑战 |
| --- | --- |
| 问题数量多且分散（可数十条） | 拆分为多条 REQ 需求会过于碎片化 |
| 优先级由严重性决定（P1-P4） | 与常规需求优先级维度不同 |
| 常含"可忽略"项（私有仓库） | 需灵活处理，不宜一刀切做需求 |
| 问题间可能存在依赖 | 部分问题需整体处理 |
| 观察项（P4）不要求修复 | 不应作为正式需求管理 |

**因此，将审计整包作为一个任务包管理**，问题作为子项按优先级归档，是更高效的方式。

## 2. 审计包目录结构

```text
{YYYYMMDD}_audit-fix/           # 审计任务包根目录
│
├── README.md                   # 主报告：评分、问题汇总、修复路线图
├── decision_log.md             # 决策留痕日志
├── effective_config.yaml       # ll-audit 生效配置快照
│
├── P1/                         # P1 - 必须修复（每项一个子目录）
│   ├── P1-001-xxx/
│   │   ├── 问题描述.md
│   │   ├── 解决方案.md
│   │   └── 实施总结.md         # 仅已修复项需要
│   ├── P1-002-xxx/
│   │   └── ...
│   └── ...
│
├── P2/                         # P2 - 应该修复
│   ├── P2-001-xxx/
│   │   ├── 问题描述.md
│   │   └── 解决方案.md        # P2 项不一定需要实施总结
│   └── ...
│
├── P3/                         # P3 - 建议修复
│   └── P3-001-xxx/
│       └── 问题描述.md         # P3 通常只记录问题，不一定修复
│
├── P4/                         # P4 - 观察项
│   └── P4-001-xxx/
│       └── 问题描述.md         # 仅标记状态（待修复/预期设计/无需修改）
│
└── skipped/                     # B2 阶段跳过的项
    ├── skip_log.md             # 跳过问题日志
    └── debt_backlog.md         # 技术债台账（汇总所有跳过项）
```

## 3. 文件说明

### 3.1 README.md（主报告）

审计包的入口文件，包含：

- **基本信息**：审计日期、范围、工具、分支策略
- **总体评分**：6 维度评分（安全性、稳定性、架构设计、代码质量、性能效率、可维护性）
- **问题汇总**：P1/P2/P3/P4 各有多少条
- **各优先级问题详情**：每条问题的 ID、文件、简述
- **优先修复路线图**：分阶段建议修复顺序
- **复核记录**：交叉验证结果
- **分支与合并记录**：审计分支状态

### 3.2 decision_log.md（决策留痕）

记录审计过程中的关键决策点：

| 决策阶段 | 典型决策点 |
| --- | --- |
| A0 | 审计范围（全量/部分） |
| A1 | 并行子 Agent 数量 |
| A2 | 任务目录命名、问题合并策略 |
| B0 | 是否启用审计分支 |
| B1 | 跳过项处理方式 |
| B2 | 优先级处理范围 |

### 3.3 effective_config.yaml（配置快照）

ll-audit 执行时的生效配置，用于复现和审计。

### 3.4 P{X}/ 问题子目录

每个子目录对应一条审计问题，命名格式：`P{优先级}-{3 位序号}-{slug}`

**P1 子项包含**：
- `问题描述.md` — 问题详情、风险分析、复核记录
- `解决方案.md` — 修复策略、修改计划、验证计划、回退方案
- `实施总结.md` — 仅已修复项需要，记录实际修复内容

**P2 子项**通常包含问题描述和解决方案，可选实施总结。

**P3/P4 子项**通常只需问题描述（记录现状和状态）。

### 3.5 skipped/（跳过项）

| 文件 | 作用 |
| --- | --- |
| `skip_log.md` | 记录在 B2 阶段选择跳过的问题及原因 |
| `debt_backlog.md` | 将所有跳过项汇总为技术债台账，供后续迭代跟踪 |

## 4. 命名规范

### 4.1 审计包目录

```text
{YYYYMMDD}_audit-fix
```

示例：`20260401_audit-fix`、`20260418_audit-fix`

### 4.2 问题子目录

```text
P{优先级}-{3位序号}-{slug}
```

示例：`P1-001-backend-env-credentials`、`P2-002-auth-session-expire-race`

### 4.3 提交信息模板

```text
fix(P{优先级}-{序号}): {简短描述}
```

示例：`fix(P1-001): remove backend/.env from git, add .env.example`

---

## 5. 与常规任务包（六件套）的区别

| 维度 | 常规任务包 | 审计任务包 |
| --- | --- | --- |
| 入口文件 | `README.md` | `README.md`（主报告格式不同） |
| 需求文档 | `REQ-XXXX.md` | 不需要（问题即需求） |
| 任务拆解 | `tasks.md` | 不需要（问题清单即任务） |
| 方案设计 | `design.md` | 合并到 `解决方案.md` |
| 决策留痕 | `decision_log.md` | `decision_log.md`（结构相同） |
| 状态管理 | 需求状态（P1/P2/P3） | 问题状态（P1/P2/P3/P4 + skipped） |
| 优先级维度 | 业务优先级 | 严重性优先级 |

## 6. 使用场景

### 6.1 创建新审计包

1. 复制 `template/audit-package` 整个目录到 `B.todo/{version}/` 或 `B.2.done/{version}/`
2. 重命名为 `{YYYYMMDD}_audit-fix`
3. 填充各文件内容
4. 问题子目录复制 `P1-001-example-issue` 并重命名

### 6.2 审计包归档

审计任务完成后，从 `B.todo/{version}/` 移动到 `B.2.done/{version}/`：

```bash
git add B.todo/{version}/{YYYYMMDD}_audit-fix/
git mv B.todo/{version}/{YYYYMMDD}_audit-fix B.2.done/{version}/
```

### 6.3 问题修复跟踪

在 `skipped/skip_log.md` 中记录跳过的项，在 `skipped/debt_backlog.md` 中汇总为技术债。

每个子项的 `问题描述.md` 中标记修复状态：
- [ ] 待修复
- [x] 已修复（在 `实施总结.md` 中记录）

## 7. ll-audit 技能集成

审计包与 ll-audit 技能配合使用：

1. ll-audit 生成审计结果
2. 结果写入审计包的 `README.md` 和各 `P{X}/` 子目录
3. decision_log.md 由技能自动维护
4. effective_config.yaml 记录技能执行配置

ll-audit 技能配置参考：`template/audit-package/effective_config.yaml`
