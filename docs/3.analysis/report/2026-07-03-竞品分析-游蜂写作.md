# 游蜂写作竞品分析报告

## 来源

- 产品官网：https://www.youfengxiezuo.com/xiezuo.html
- 更新日志：https://www.youfengxiezuo.com/changelog.html
- 分析日期：2026-07-03

## 产品概况

游蜂写作是一款**桌面级、全链路、数据本地**的长篇网络小说工程化创作系统。定位明确：面向长篇连载（数百章），核心卖点是"不失忆、不矛盾、不烂尾"。

**定价模式**：软件授权费（日卡/月卡/年卡），不含 API 额度，用户自备 API Key。

**版本节奏**：v1.0.5（2026-03-10）→ v1.1.8.1（2026-06-28），约 4 个月 23 个版本，迭代极快。

---

## 八阶段流水线（核心架构）

游蜂的创作流分为八个阶段，每阶段有独立校验：

```
灵感 → 剧情树 → 设定 → 关系网 → 大纲 → 正文 → 指令库 → 成品投稿
```

对比 ai-novel 的 auto-director 流水线，游蜂的阶段划分更细、用户控制粒度更高。

---

## 值得借鉴的设计

### 1. 剧情树（三层骨架）

游蜂用**三层剧情树**规划长篇结构：
- **第一层**：全书开端、转折与归宿（锚点）
- **第二层**：卷级弧线（按批次生成，可逐卷确认后再继续）
- **第三层**：章节目标（细化到每章）

**我们的差距**：ai-novel 的 volume strategy + beat sheet 机制类似，但缺少"逐层确认后再继续"的用户控制点。游蜂支持**逐级清除和重新生成**（只重做某一层，不影响其他层），降低时间和成本。

**建议借鉴**：
- 在 auto-director 的 volume planning 阶段增加"层间确认门控"
- 支持单层重做而非全量重跑

### 2. 关系网全程追踪

游蜂的关系网功能包括：
- **势力阵营**梳理
- **角色档案**逐个建立
- **生死登退场**全程对齐（角色退场后绝不穿帮复活）
- **关系自洽**校验（人物关系随剧情更新且彼此呼应）
- 支持**导出 TXT / 导入 TXT**

**我们的差距**：ai-novel 有 character + relationship 模型，但缺少：
- 退场角色自动"冻结"（防止穿帮复活）
- 关系网的独立校验层（我们的校验在 auto-director quality loop 中，不是独立阶段）
- TXT 导入导出（方便用户在不同工具间迁移）

**建议借鉴**：
- 增加角色退场状态机（active → exited → frozen），退出角色不再参与正文生成
- 关系网增加独立的一致性校验报告

### 3. 设定一致性自动校验

游蜂的设定模块：
- 数十个字段**分章节渐进生成**（不一次性糊一大坨）
- **自动校验前后一致**（"上一卷能飞下一卷不能"这种硬伤在生成时就被拦下）
- 设定**持续可扩**（随剧情推进补充与修订）

**我们的差距**：ai-novel 的 world/setting 模型存在，但一致性校验主要依赖 auto-director quality loop 的事后审查，没有前置的设定内矛盾检测。

**建议借鉴**：
- 在设定生成后增加"设定内一致性校验"步骤（检测互相矛盾的字段）
- 设定变更有 diff 审查（改动了什么、影响哪些已有章节）

### 4. 大纲"终稿"机制

游蜂允许用户对大纲章节设置**终稿标记**：
- 终稿章节不会因后续审查/补充关系网/补充时间线而被修改
- 支持按章节区间审查（终稿章节跳过审查）
- 审查后支持**一键全部替换**或按问题类型筛选替换

**我们的差距**：ai-novel 的 outline 没有"锁定"机制，auto-director 的 replan 可能改动用户已确认的章节。

**建议借鉴**：
- 增加章节/卷的"锁定"状态，被锁定的内容不参与自动修改
- 审查结果支持批量应用（而非逐条确认）

### 5. 千层饼分层大纲

游蜂的大纲分层：
- 从粗到细：全书走向 → 卷级弧线 → 章节大纲 → 每章张弛分布
- **按卷生成**：不用等全部大纲跑完才能写作，支持中途暂停或断点继续
- **伏笔埋收追踪**：前面埋的伏笔后面记得回收

**我们的差距**：ai-novel 的 volume/chapter planning 是按卷推进的，但伏笔追踪（payoff ledger）还在早期阶段。

**建议借鉴**：
- 伏笔埋收的可视化追踪（当前埋了哪些、哪些已收、哪些过期未收）
- 大纲按卷生成后立即解锁该卷正文写作（不等全部大纲完成）

### 6. 字数铁律

游蜂严格执行：**单章 1000-4000 字**，面向连载平台口径。

**我们的差距**：ai-novel 的 chapter word count 是配置项，但没有作为"铁律"执行。生成时经常超出或不足。

**建议借鉴**：
- 将字数范围作为 generation 的 hard constraint（而非 soft hint）
- 生成后自动检测并触发 rewrite 如果字数不达标

### 7. 指令库（写作风格模板库）

游蜂的指令库：
- 分类：核心灵感风格模板 / 剧情树风格模板 / 大纲摘要风格模板 / 正文润色模板 / 人类范文库
- **一键切换**：同一类指令存多份（热血版/细腻版/爽文快节奏版）
- **范文带笔触**：丢入范文，AI 照着那股味道写
- 支持**指令模板导入文章拆解为新模板**

**我们的差距**：ai-novel 有 style engine，但缺少：
- 用户可自定义的指令模板库
- 范文驱动的风格学习
- 模板分类管理

**建议借鉴**：
- 增加用户自定义 prompt 模板的 CRUD
- 支持范文导入 → 风格提取 → 生成模板

### 8. 正文润色功能

游蜂支持：
- 写完正文后**润色**（与原文对比，可自定义改进）
- 润色模板可在指令库添加和修改
- 复制当前章/全部正文

**我们的差距**：ai-novel 有 rewrite/expand/compress，但缺少专门的"润色"模式（保持内容不变，只优化文笔）。

**建议借鉴**：
- 增加 `polish` generation mode（输入正文 → 输出润色版，附 diff 对比）

### 9. 数据导入导出

游蜂在多个模块支持 TXT 导入导出：
- 设定导入/导出 TXT
- 关系网导入/导出 TXT
- 大纲导入/导出 TXT
- 剧情树导入 TXT（格式：`章节名称-----章节目标`）
- 世界书导出（可导入到游蜂·角色扮演软件）

**我们的差距**：ai-novel 有 novel export（markdown/json），但缺少：
- 各模块独立的 TXT 导入导出
- 跨产品数据互通（如导出到角色扮演工具）

**建议借鉴**：
- 关键资产（设定/大纲/关系网）增加独立的 TXT 导入导出
- 便于用户在不同工具间迁移和备份

### 10. 断点续传

游蜂在多个层级支持断点续传：
- 剧情树生成支持**停止 + 断点续传**
- 大纲按卷生成，支持**中途暂停或断点继续**
- 可先生成已有大纲的正文，不等全部大纲完成

**我们的差距**：ai-novel 的 auto-director 有 checkpoint/resume 机制，但在 volume planning 和 chapter list generation 层面缺少细粒度的暂停/继续。

## 核心结论

1. **游蜂的用户控制粒度远超我们**：逐层确认、终稿锁定、逐级重做、断点续传——这些设计让用户始终掌握主导权。ai-novel 的 auto-director 倾向全自动化，用户介入点少。

2. **游蜂的"一致性"是系统级设计**：从设定校验到关系网追踪到退场状态机，一致性不是事后审查，而是在每个阶段内建。ai-novel 的一致性主要依赖 auto-director quality loop 的事后补救。

3. **游蜂的数据可移植性更强**：多模块支持 TXT 导入导出，便于用户在不同工具间迁移。ai-novel 的数据导出集中在最终产物。

4. **ai-novel 的自动化程度更高**：LangGraph auto-director + 多 Agent 协作 + 质量循环是游蜂不具备的。这是我们的核心差异化优势。

5. **迭代速度惊人**：4 个月 23 个版本，说明团队执行力强，值得关注后续动态。
